## 1. Data

**Q: 現在のデータセット規模はどれくらいですか？（総エピソード数、サイズ、データ収集速度、シェーバー梱包データの構成、成功/失敗の分布）**

### 回答案

**iREX 2025 シェーバー梱包デモの実績:**
- データ収集: 約200エピソード / 約90分（4スキル連続実行による梱包デモ）
- 学習時間: 約1日 / スキル

**データ構成:**
- 入力: カメラ画像、ロボット状態（関節角・トルク等）、言語指示（タスクID）
- 出力: 連続的なロボット動作（Action）
- フォーマット: RLDS / LeRobot形式への変換パイプラインを整備中

**収集速度:**
- テレオペレーションによる収集が主。1スキルあたり約200エピソード / 90分が目安
- [要確認] 現時点の総データセットサイズ（GB）、成功/失敗の分布比率

**FY26 データ戦略:**
- Robo Syncをデプロイ＋ログ収集の統合インターフェースとして活用
- テレオペ（高品質・少量）、シミュレーション（大量・バリエーション拡張）、人間動画（事前学習用）の3層構造でスケール
- CECクラウド基盤と連携し、データレイク / MLOps パイプラインを整備予定

## 2. Hardware (1)

**Q: iREXのシェーバーデモにおける主要プラットフォームはどれですか？また、その他のプラットフォームの優先順位は？**

### 回答案

**iREX 2025 デモ:**
- [要確認] 使用ロボット（UR5？その他？）

**FY26 ハードウェア戦略（2軸）:**

| 軸 | ハードウェア | 用途 |
|----|------------|------|
| **産業用ロボット** | UR5等 | 現場PoC・RoboSync統合。現場耐久性・QMS対応が容易 |
| **研究・実験用ロボット** | Aloha Stationary, Doraemon AI, SO1等 | ラボ検証、SDR（Software Defined Robot）文脈の推進 |

**優先順位:**
1. **UR5（産業用）**: RoboSync既存統合、FY26の現場PoC主軸
2. **研究用ハード**: 双腕タスク検証、新しいハードウェア形態の探索
3. **ヒューマノイド / 移動ロボット**: 長期軸（FY27以降）、テレオペ前提のユースケース

Robo Syncのハードウェア抽象化レイヤにより、上位のVLAスキルは特定ハードウェアに依存しない設計となっている。

## 3. Hardware (2) — 推論要件

**Q: 推論（インファレンス）要件は何ですか？（実行頻度、どこで実行するか、レイテンシ要件）**

### 回答案

**推論アーキテクチャ:**

| レイヤ | 推論場所 | レイテンシ要件 | 実行頻度 |
|--------|---------|-------------|---------|
| **Task Planning Hub（LLM Agent）** | クラウド / エッジ | 秒〜数秒オーダー | タスク切替時 |
| **Symbol Hub（VLM）** | クラウド / エッジ | 秒オーダー | 状態認識・成功/失敗判定時 |
| **Sensorimotor Hub（VLA Policy）** | エッジ（ロボット近傍） | **ミリ秒〜数十ミリ秒** | **リアルタイム連続推論**（制御ループ内） |

**VLA Policy（π0）の推論:**
- VLAはリアルタイムのロボット制御ループ内で動作するため、エッジ側での推論が必須
- [要確認] 現在の推論周波数（Hz）、使用GPU/アクセラレータ、レイテンシ実測値

**FY26 方針:**
- 小規模実験: ローカル / オンプレ
- 大規模学習: CECクラウド基盤
- 推論（VLA Policy）: エッジデバイス（ロボット近傍GPU）

## 5. Training (2) — 評価指標

**Q: どの評価指標を使用していますか？（成功率、タスク完了時間、軌跡誤差 など）**

| カテゴリ | 指標 | 説明 |
|---------|------|------|

## 6. Safety

**Q: 学習ポリシー実行時の安全フレームワークはどうなっていますか？**

### 回答案

**Robo Syncの安全制御基盤:**

Robo Syncは、VLA由来の学習ベース動作と既存のブロックベース制御を安全に共存させる設計。

| 安全機能 | 内容 |
|---------|------|
| **安全制約（速度・加速度・ソフトリミット）** | VLAから受け取った目標姿勢・軌道を、事前定義された安全制約の範囲内で実行 |
| **力制限** | トルクプロファイルに基づく異常検知（締め過ぎ・押し込み過多の検知） |
| **非常停止（e-stop）** | ハードウェアレベルの非常停止 |
| **失敗判定・停止条件** | VLAスキルの入出力IF・失敗判定・停止条件を文書化（Q1で合意予定） |
| **人介在リカバリー** | α版: 異常時は人介在でリカバリー。β版: 異常時自動停止 |

**FY26 QMS段階計画:**

| 段階 | 安全性レベル |
|------|------------|
| **α版（9月）** | Robo Sync標準の安全制約に準拠。人介在でのリカバリー体制あり |
| **β版（3月）** | 現場固有の安全パターン対応。異常時の自動停止 |
| **リリース（FY27〜）** | 安全要件の文書化・監査対応 |

**Agent自律適応における安全設計:**
- 代表的失敗パターン3種（把持失敗、配置ズレ、落下）の振る舞いを定義（Q1）
- 失敗時のリトライ自動実行（Q2 α版で検証）
- [要確認] OOD（Out-of-Distribution）検知の具体的アプローチ、フェールバック動作の詳細、ロールバック機構の有無

## 7. Responsible AI (1) — データ品質

**Q: 学習データの品質と代表性（多様性）をどのように確保していますか？**

### 回答案

**データ品質の確保:**

| 観点 | アプローチ |
|------|----------|
| **オペレーターの多様性** | [要確認] 現在のテレオペ実施者数・多様性。FY26では複数オペレーターによるデータ収集を計画 |
| **エッジケースのカバレッジ** | シミュレーションによるバリエーション拡張（物体配置・照明・ノイズ）。RLや自動探索による失敗パターンの網羅 |
| **データトレーサビリティ** | 「どのデータが、どのモデル更新に寄与したか」を追跡可能にする設計 |
| **成功/失敗ラベル** | Robo Syncでの実行ログに成功/失敗ラベルを付与。VLMによる自動判定の導入を予定 |

**3層データ戦略による多様性確保:**
1. **テレオペ（Human Teleoperation）**: 最も高品質。熟練者の暗黙知を含む。量より質を重視
2. **シミュレーション（Synthetic Data）**: 大量にスケール可能。バリエーション拡張・失敗パターン網羅
3. **人間動画（Human Videos）**: エゴセントリック動画等から物体・手のキーポイントを抽出。事前学習用

## 11. Deployment (3) — デプロイパイプライン

**Q: 学習済みモデルはどのようにロボットへデプロイしていますか？**

### 回答案

**データライフサイクル（5ステップ）:**

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ Model Deploy │────→│   Logging    │────→│ Data Export   │
│ (Robo Sync   │     │ (現場実行中  │     │ (CECクラウド  │
│  経由で現場へ)│     │  ログ収集)   │     │  RLDS変換)    │
└──────────────┘     └──────────────┘     └──────────────┘
       ↑                                         │
       │                                         ▼
┌──────────────┐                          ┌──────────────┐
│  Re-Deploy   │←────────────────────────│Train/Fine-tune│
│ (改善モデル  │                          │(現場ログで    │
│  を再投入)   │                          │ モデル更新)   │
└──────────────┘                          └──────────────┘
```

| ステップ | 内容 |
|---------|------|
| **Model Deploy** | 学習済みポリシーをRobo Sync経由でデプロイ。Robo SyncのGUI上でスキルブロックとして配置 |
| **Logging** | 実行中のカメラ画像、ロボット状態、タスクシナリオ、成功/失敗ラベルをRobo Sync側で一元収集 |
| **Data Export** | CECクラウド基盤へ転送、RLDS形式に変換 |
| **Train / Fine-tune** | テレオペ / シミュレーション / 現場ログを統合しモデル更新 |
| **Re-Deploy** | 改善モデルを再デプロイ、効果を評価 |

**バージョニング・モニタリング:**
- データトレーサビリティ: どのデータがどのモデル更新に寄与したか追跡可能
- 成功率・サイクルタイム・人介在頻度で効果を継続モニタリング
- CECのCI/CDパイプライン: モデル更新→検証→デプロイの自動化を予定

[要確認] 現時点のバージョニング運用（モデルレジストリ等）の具体的な仕組み
