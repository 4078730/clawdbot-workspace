## 1. 対象業務

### 1.1 現場概要

| 項目 | 内容 |
|------|------|
| **拠点** | 彩都パーツセンター |
| **業務** | オートストア出庫作業 |
| **規模** | 12万品番、月間3万点出荷 |
| **設備** | オートストア（2万ビン、AGV14台） |
| **付加価値作業比率** | 30%（業界トップ水準） |

### 1.2 出庫作業フロー（動画分析 2026-02-16）

```
[1] 準備・登録
    納品書の束を作業台に持ってくる → モニター操作で作業者登録

[2] スキャン・チェック
    バーコードリーダーで納品書を1枚ずつスキャン（〜10枚/束）
    ペンでチェック

[3] ビン到着・ピック                     ← α版の対象範囲
    オートストアからビンが作業者の右側に到着
    ビン内構造: 段ボール → 緩衝材（プチプチ等）→ パーツ
    納品書に対応するパーツを確認・ピック → 作業台に配置

[4] 個装（物による）                     ← β版の対象範囲
    パーツに合った袋に入れ、納品書を同封
    袋を閉じて左側のワゴンに配置

[5] 次のオーダーへ
    1オーダー: 約1〜3分
```

### 1.3 α版が扱う複雑さ

α版では「オペレーションの標準化が不可能」な部分はスコープ外とし、
**商品の形状・包装状態の複雑さに対するVLA制御の有効性**を実証する。

#### α版が扱う課題

| 課題 | 具体例 |
|------|--------|
| **形状の多様性** | 天板（平面パーツ：端の把持点が一意に定まらない）、不定形パーツ、小型部品 |
| **包装状態の多様性** | 段ボール内のプチプチ包装、袋入り、仕切り付きビン |
| **取り出しにくさ** | 立てかかってつかみにくい、一度で掴めない、下に埋まっている |

> → VLAによる操作制御ポリシーで、形状・状態に応じた把持・取り出し動作を生成する

#### α版のスコープ外（β版以降）

- 個装（袋入れ・納品書同封）
- 梱包（段ボール開封含む）
- 移動（台車配置）
- 複数スキル切替のAgenticな制御

## 2. フェーズ定義

### 2.1 全体構造 — 複雑さの3軸で段階的に拡張

**設計原則:** VLAはルールベースの「置き換え」ではなく「段階的補完」。
ルールベースで対応可能なものはルールベースで処理し、VLAが困難なタスクを補完。
複雑さを3軸に分け、時間的に分離して取り組む。

| 軸 | α版（CA） | β版 | GA（参考） |
|----|----------|-----|----------|
| **手** | 吸着/平行グリッパ（手動交換） | 同左 + 自動チェンジャー検討 | 指の制御（研究成果統合、GA オプション） |
| **腕** | 単腕・VLA制御 | 双腕・Agentic制御・個装/段ボール開封 | 双腕高度化（GA オプション） |
| **移動** | なし（固定ステーション） | TBD（台車配置まで） | 移動ロボット（GA オプション） |

### 2.2 α版（CA: Controlled Availability）— Q2末/9月

**ゴール:** RoboSync上でVLA制御ポリシーが動作し、形状・包装状態が複雑なパーツの
ビン内ピック → 作業台配置を実現する

#### スコープ

| 項目 | 内容 |
|------|------|
| **IN** | ビン内パーツのピック → 作業台に配置 |
| **OUT** | 個装、梱包、台車への移動、納品書処理、Agentic制御 |
| **対象パーツ** | 形状・包装状態が複雑なパーツ（具体ケースはRoboSyncチームと選定） |
| **人の役割** | 個装・梱包・納品書処理は人が担当。リカバリー作業も人が実施（人×ロボット協働） |

> **人の介在について:** ロボットのピック失敗・異常時には人が手動でリカバリーを行う。
> β版以降ではテレオペ介入や操作ログ記録によるリカバリーデータ蓄積を目指すが、
> α版では開発工数を考慮し手動リカバリーとする。

#### 成功基準（仮）
*期初時点の目安。現場・実際のタスク・技術レベルに合わせて調整。*

| 指標 | 目標 | 備考 |
|------|------|------|
| **ピック成功率** | **90%** | 対象パーツを絞った上での成功率 |
| **リトライ込み成功率** | **95%** | 1回失敗後のリトライを含む |
| **ログ蓄積** | 稼働 | 実行ログが記録される状態 |
| **安全性** | **インシデント0** | 物理的な怪我に繋がる事故は絶対にゼロ |

#### 技術要件

| # | 要件 | 詳細 |
|---|------|------|
| T-1 | **VLA制御ポリシー** | 形状・包装状態が複雑なパーツに対して、操作軌道を生成・実行 |
| T-2 | **RoboSync統合** | VLAスキルブロックとしてRoboSync上で実装・実行・管理 |
| T-3 | **ログ収集** | 実行ログ（画像・ロボット状態・成功/失敗）の記録 |
| T-4 | **安全制御** | RoboSync標準安全制約に準拠。物理的な怪我につながる危険箇所は絶対に対処 |
| T-5 | **ルールベース連携** | ルールベースで対応可能なケースはルールベースで処理。VLAは困難なケースを補完 |

*※ Agentic要素（複数スキル切替、自律的再計画等）はα版には含まない*

#### 事業的価値（内部レビュー向け）

彩都パーツセンターの出庫作業では、ビン内のパーツが多様な形状・包装状態で存在する。
例えば、平面パーツ（天板等）は端の把持点が一意に定まらず、
不定形パーツやプチプチに包まれた状態のパーツは吸着のみでは対応不可能。
α版では、こうした形状・状態の複雑さに対してVLA制御ポリシーが有効であることを、
「ビンからパーツをピックして作業台に置く」という限定されたスコープで実証する。
個装は人が担当し、人×ロボット協働でオペレーションを成立させる。

### 4.2 α版の制御フロー

α版では **Sensorimotor Hub（VLA Policy）+ Robo Sync** のみ。Task Planning Hub、Agentic制御は不使用。

```
入力                         処理                          出力
─────                       ────                          ────
カメラ画像（RGB）    ──→  VLA Policy  ──→  関節角度制御
  640×480 / 30fps            （推論）         （q1〜q6）
  + ロボット関節角度                        + グリッパ開閉指令
  + パーツ情報（PMS）
    パーツ名・品番・
    ビン内ピック位置
        │
        ▼
    Robo Sync
    ・安全チェック（速度・力・範囲制限）
    ・UR5eへの指令送信
    ・実行ログ記録
```

#### VLAモデル（確定）

**α版採用モデル: π0 / π0.5 (Physical Intelligence)**

| モデル | 特徴 | α版での位置づけ |
|--------|------|---------|
| **π0** | Physical Intelligence社。Flow Matching + VLA。汎化性能が高い | **採用確定**。Fine-tuningベースライン |
| **π0.5** | π0の後継。マルチタスク・スケーラビリティ向上版 | **採用確定**。π0と並行評価 |

#### VLA Policy の入出力仕様（α版）

| 項目 | 仕様 |
|------|------|
| **入力: 画像** | RGB 画像（Hand-Eye カメラ）/ **640×480 / 30fps** / 深度は使用しない |
| **入力: ロボット状態** | 関節角度 6次元（q1〜q6）+ グリッパ状態 1次元 = 7次元 |
| **入力: パーツ情報** | PMSから取得するパーツ仕様（パーツ名・品番・ビン内ピック位置 ※仕切りがある場合）。言語指示ではなくパーツ仕様書として読み込む |
| **出力: アクション** | **関節角度制御** — 各関節の目標角度 6次元（q1〜q6）+ グリッパ指令 1次元 = 7次元 |
| **制御周期** | **10Hz**（確定） |

### 4.3 ルールベースとVLAの切替条件

```
                パーツ認識
                    │
            ┌───────┴───────┐
            │               │
     ルールベースで       ルールベースでは
     対応可能か？         対応不可
            │               │
            ▼               ▼
     ┌────────────┐   ┌────────────┐
     │ルールベース │   │ VLA Policy │
     │ピック       │   │ ピック      │
     │（既存）     │   │（新規）     │
     └────────────┘   └────────────┘
```

| 条件 | 制御方式 | 判定方法 |
|------|---------|---------|
| 箱物・整列済み・吸着面が明確 | ルールベース | 画像認識で物体カテゴリ判定 |
| 不定形・包装状態が複雑・把持点不明確 | VLA Policy | 上記以外（α版初期は全VLAも可） |
| VLA推論が異常値を返した場合 | **停止→人介在** | アクション値の範囲チェック |
| VLAが一定時間内にタスク完了しない場合 | **タイムアウト→人介在** | タイマー（閾値TBD） |

> **α版初期の方針:** 最初はVLAのみで動かし性能データを蓄積する。
> ルールベースとの切替は、VLAの得意/不得意が明確になった段階で導入。

### 4.4 β版の制御フロー

β版では **Task Planning Hub（LLM Agent）** が加わり、8工程のオーケストレーションを行う。

```
LLM Agent（Task Planning Hub）
│
├─ [1] バーコードスキャン → スキャンスキル呼出
├─ [2] モニター操作      → UI操作スキル呼出
├─ [3] 段ボール開封       → 開封スキル呼出（双腕）
├─ [4] パーツピック       → ピックスキル呼出
├─ [5] ビンに戻す        → 配置スキル呼出
├─ [6] 個装              → 個装スキル呼出（双腕）
├─ [7] 台車配置          → 配置スキル呼出
└─ [8] 次のオーダー      → サイクル管理

各スキルは Robo Sync 上のスキルブロックとして実装
スキル間の遷移は BT（ビヘイビアツリー）で定義
```

### 4.5 RoboSyncとの統合

| レイヤ | 責任 | 実装場所 |
|--------|------|---------|
| **タスク管理** | スキル選択・切替・再計画 | Task Planning Hub（β版） |
| **スキル実行** | VLA推論 → アクション生成 | VLAスキルブロック（RoboSync上） |
| **安全制御** | 速度制限・力制限・範囲制限・緊急停止 | RoboSync標準安全レイヤ |
| **ハード制御** | UR5eドライバ・EE制御・カメラ取得 | RoboSyncハード抽象化レイヤ |

### 5.2 β版：8工程の詳細プロセス

#### 工程[1] バーコードスキャン

```
入力:  俯瞰カメラ画像（納品書の位置検出）
処理:  ロボットがコードリーダーを把持 → 納品書上のバーコードに位置合わせ → スキャン実行
出力:  オーダー情報（品番・数量・個装指示）
技術:  ツール操作スキル（コードリーダーの把持と操作）、バーコード位置検出（画像認識）
課題:  バーコードリーダーの把持安定性、読み取り角度の調整
```

**伝票の実物仕様（現場確認済み）:**

| 項目 | 内容 |
|---|---|
| 伝票セット構成 | 3種類1セット：①売上計算票（経理用）②納品通知票・出荷明細票（仕入先元帳用）③入庫指図票・出庫票（商品元帳用） |
| バーコード種別 | 1次元バーコード（Code 128系）、各伝票右下に印刷 |
| スキャン対象 | 3枚のうち最上面の1枚のバーコードをスキャン（伝票番号・オーダー番号に対応） |
| スキャンで取得する情報 | 品番（例: ET-LAV300）、棚番号（例: B02A-C000）、数量、個装形態コード（例: NW1） |
| 伝票の置き方 | 作業台上にまとめて積まれた状態で到着。上面が読み取り対象 |

#### 工程[2] モニター操作（ビン呼び出し）

```
入力:  オーダー情報（工程[1]の出力）、モニター画面画像
処理:  ロボット or ソフトウェア連携でオートストアにビン呼び出し指令を送信
出力:  ビン呼び出し完了通知
技術:  API連携（オートストアシステムとの直接通信）or タッチパネル操作（VLA）
課題:  オートストアとのシステムIF定義、タッチ操作の精度
備考:  API連携が可能ならロボットの物理操作は不要。要現場確認
```

#### 工程[3] ビン内段ボール開封（必要時）

```
入力:  ビン内の画像（段ボール有無の判定）
処理:  片腕で段ボールを押さえ、もう片腕でフラップを開ける
出力:  段ボール開封完了、パーツが見える状態
技術:  双腕協調制御、力制御（段ボールを破壊しない程度の力）
課題:  段ボール形状の多様性（テープ有無、フラップの向き）、双腕VLAの難易度
前提:  双腕ロボットが必要
```

#### 工程[4] パーツのピック（α版拡張）

```
入力:  ビン内画像、オーダー情報（品番・数量）
処理:  α版のピック動作を拡張
       ・指定品番のパーツを識別（12万品番の一部）
       ・指定数量をピック（「3個取り出す」→ 3回ピックサイクル）
       ・緩衝材からの取り出し（プチプチを剥がす等）
出力:  指定パーツが作業台上に配置された状態
技術:  VLA Policy（α版ベース）+ 品番識別（VLM）+ カウント
課題:  品番識別の精度、数量管理、緩衝材除去の制御
```

#### 工程[5] 残りをビンに戻す

```
入力:  ビン内・作業台の画像
処理:  取り出した後のビン内を整理。緩衝材や不要な梱包材をビンに戻す
出力:  ビンが元の状態に近い形で整理された
技術:  VLA Policy（整理・配置スキル）
課題:  「元の状態」の定義、壊れ物の取り扱い
```

#### 工程[6] 個装（袋入れ + 納品書同封 + 袋閉じ）

```
入力:  作業台上のパーツ、納品書（3枚セット）、個装用袋
処理:
  6a. 個装形態コード（伝票から取得）に基づいて適切なサイズの袋を選択
  6b. 片腕で袋を開き、もう片腕でパーツを入れる
  6c. 納品書（3枚セットのうち指定の1枚）を袋に同封
  6d. 袋を閉じる（輪ゴム or シーラー — 個装形態コードで指定）
出力:  個装済みパーツ（袋入り・納品書同封）
技術:  双腕協調制御、柔軟物操作（袋）、VLA Policy
課題:  袋を開く動作（柔軟物操作の高難度タスク）、輪ゴム留め、納品書の枚数管理
備考:  輪ゴム留めは特に高難度。β版で全工程を完遂できるかは検証次第
```

**個装形態コードの実物仕様（現場確認済み）:**

| 項目 | 内容 |
|---|---|
| コード例 | NW1（伝票上の「個装形態」欄に印字）|
| 意味 | 品番ごとに袋のサイズ・包装方法・同封書類が異なる（コード体系は要詳細確認） |
| 同封する納品書 | 3枚セットのうち指定の1枚（どれかは要確認） |
| 袋サイズのバリエーション | 作業台上に各種サイズの小袋が常設されており、コードに応じて選択 |
| ⚠️ 要確認 | 個装形態コードの全種類と対応する袋サイズ・包装手順の一覧（中田さん/一力さんに確認）|

#### 工程[7] 台車/ワゴンに配置

```
入力:  個装済みパーツの画像、台車の空き位置
処理:  個装済みパーツを台車/ワゴンの適切な位置に配置
出力:  パーツが台車上に配置された
技術:  配置スキル（VLA or ルールベース）、台車位置認識
課題:  リーチ制限（台車がロボット作業範囲内にあるか）、配置位置の最適化
```

#### 工程[8] 次のオーダーへ → [1]に戻る

```
入力:  現在のオーダー完了通知
処理:  次のオーダーが存在するか確認 → [1]に戻る
出力:  サイクル継続 or 全オーダー完了
技術:  Agentic制御（LLM Agent）、サイクル管理
課題:  異常時の判断（スキップ? リトライ? 人に通知?）
```

### 8.1 α版の検証

#### 成功/失敗の定義

| 判定 | 定義 |
|------|------|
| **ピック成功** | 対象パーツをビンからピックし、作業台上の指定位置に配置できた |
| **ピック部分成功** | ピックは完了したが、配置位置・向きが不適切。VLM判定でフラグ立て |
| **ピック失敗** | パーツを把持できなかった / 落下した / タイムアウト |

#### 評価指標

| 指標 | 定義 | α版目標 |
|------|------|---------|
| **ピック成功率** | 成功 / 総トライアル数 | **90%**（対象パーツを絞った上で） |
| **リトライ込み成功率** | 1回リトライを含む成功率 | **95%** |
| **ログ蓄積** | 実行ログが記録される状態 | 稼働 |
| **安全性** | インシデント件数 | **0件**（物理的な怪我につながる事故は絶対にゼロ） |

#### 検証ステージ

| ステージ | 場所 | 目的 |
|---------|------|------|
| **ラボ模擬環境** | 社内ラボ | ハードウェアセットアップ、VLAスキル基本動作確認 |
| **拡張ラボ** | 社内ラボ（拡大） | 複数パーツカテゴリ、VLA性能評価 |
| **現場検証** | 彩都パーツセンター実環境 | 実環境での動作確認、環境差異の検出 |

### 8.2 β版の検証

#### 成功/失敗の定義

| 判定 | 定義 |
|------|------|
| **サイクル成功** | 工程[1]〜[8]を完遂し、個装済みパーツが台車に配置された |
| **サイクル部分成功** | 一部工程で人介在が発生したが、最終的にサイクルが完了した |
| **サイクル失敗** | サイクルを完了できなかった（ロボット停止、パーツ破損等） |

#### 評価指標

| 指標 | 定義 | β版目標 |
|------|------|---------|
| **タスク成功率** | サイクル成功 / 総サイクル数 | **TBD** |
| **安定稼働** | 継続運用可能な状態 | 「ちゃんと動くこと」が最優先 |
| **安全性** | インシデント0、異常時自動停止の動作確認 | **0件** |
