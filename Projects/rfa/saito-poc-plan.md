| 項目 | 内容 |

| **拠点** | 彩都パーツセンター |

| **付加価値作業比率** | 30%（業界トップ） |

### 1.2 出庫作業フロー（動画分析 2026-02-16）

```
[1] 準備・登録
    納品書の束を作業台に持ってくる → モニター操作で作業者登録

[2] スキャン・チェック
    バーコードリーダーで納品書を1枚ずつスキャン（〜10枚/束）
    ペンでチェック

[3] ビン到着・ピック  ← α版の対象範囲
    オートストアからビンが作業者の右側に到着
    ビン内構造: 段ボール → 緩衝材（プチプチ等）→ パーツ
    納品書に対応するパーツを確認・ピック
    → 作業台に配置

[4] 個装（物による）  ← β版の対象範囲
    パーツに合った袋に入れ、納品書を同封
    袋を閉じて左側のワゴンに配置

[5] 次のオーダーへ
    1オーダー: 約1〜3分
```

### 1.3 α版で扱う複雑さ

α版では「オペレーションの標準化が不可能」な部分はスコープ外とし、
**商品の形状・包装状態の複雑さ**に対するVLA制御の有効性を実証する。

**具体的にα版が扱う課題：**

| 課題 | 具体例 |
|------|--------|
| **形状の多様性** | 天板（平面パーツ：端の把持点が一意に定まらない）、不定形パーツ、小型部品 |
| **包装状態の多様性** | 段ボール内のプチプチ包装、袋入り、仕切り付きビン |
| **把持方法の選択** | 吸着で掴めるもの / 平行グリッパが必要なもの → エンドエフェクタ選択を含む |

→ **VLAによる操作制御ポリシー**で、形状・状態に応じた把持・取り出し動作を生成

**α版のスコープ外（β版以降）：**
- 個装（袋入れ・納品書同封）
- 梱包（段ボール開封含む）
- 移動（台車配置）
- 複数スキル切替のAgenticな制御

## 2. フェーズ定義

### 設計原則

> **MVP原則:** 各フェーズはPoCではなく、スコープ限定のMVP。
> Functional → Reliable → Usable を薄くでも全層通す。
> α版→β版→GAが連続的な改善サイクルとなり、投資が累積的に価値を生む構造。

> **ルールベースとVLAの役割分担:**
> VLAはルールベースの「置き換え」ではなく「段階的補完」。
> ルールベースで対応可能なものはルールベースで処理し、VLAが困難なタスクを補完。
> 段階的にVLAの対応範囲を拡大し、最終的にVLA主力へ移行。

> **複雑さの3軸（時間的に分離して取り組む）:**
> 1. **手の複雑さ** — エンドエフェクタの種類・指の制御
> 2. **腕の複雑さ** — 単腕/双腕、操作制御の汎用性
> 3. **移動の複雑さ** — 固定ステーション / 移動を伴う作業

### 2.1 α版（CA: Controlled Availability）— Q2末/9月

**ゴール:** RoboSync上でVLA制御ポリシーが動作し、形状・包装状態が複雑なパーツのピック → 作業台配置を実現する

#### スコープ

| 項目 | 内容 |
|------|------|
| **IN** | ビン内パーツのピック → 作業台に配置 |
| **OUT** | 個装、梱包、台車への移動、納品書処理、Agentic制御 |
| **対象パーツ** | 形状・包装状態が複雑なパーツ（具体ケースはRoboSyncチームと選定） |
| **人の役割** | 個装・梱包・納品書処理は人が担当。リカバリー作業も人が実施（人×ロボット協働） |

**人の介在について:**
ロボットのピック失敗・異常時には人が手動でリカバリーを行う。
β版以降では、テレオペレーション（遠隔操作）での介入や、UMI等での操作ログ記録による
リカバリーデータ蓄積を目指すが、α版では開発工数を考慮し手動リカバリーとする。

#### ハードウェア

| 項目 | 仕様 |
|------|------|
| **ロボットアーム** | UR5（単腕） |
| **エンドエフェクタ** | 吸着 / 平行グリッパ（タスクに応じて選択。要：奥村さんと検討） |
| **指** | 使用しない（福岡拠点で研究進行中。指の制御自体が研究対象） |
| **カメラ** | Intel RealSense D435i（チーム共通） |
| **移動機構** | なし（固定ステーション） |

#### 技術要件

| # | 要件 | 詳細 |
|---|------|------|
| T-1 | VLA制御ポリシー | 形状・包装状態が複雑なパーツに対して、操作軌道を生成・実行 |
| T-2 | RoboSync統合 | VLAスキルブロックとしてRoboSync上で実装・実行・管理 |
| T-3 | ログ収集 | 実行ログ（画像・ロボット状態・成功/失敗）の記録。学習に直接使えるかは未定だが、少なくともログとして残す |
| T-4 | 安全制御 | RoboSync標準安全制約に準拠。物理的な怪我につながる危険箇所は絶対に対処。どのレイヤーまで安全性を担保するかは要定義 |
| T-5 | ルールベース連携 | ルールベースで対応可能なケースはルールベースで処理。VLAは困難なケースを補完 |

※ Agentic要素（複数スキル切替、自律的再計画等）はα版には含まない

#### 成功基準

| 指標 | 目標 | 備考 |
|------|------|------|
| ピック成功率 | 90% | 対象パーツを絞った上での成功率 |
| リトライ込み成功率 | 95% | 1回失敗後のリトライを含む |
| ログ蓄積 | 稼働 | 実行ログが記録される状態 |
| 安全性 | インシデント0 | 物理的な怪我に繋がる事故は絶対にゼロ |

#### 事業的価値（内部レビュー向け説明）

> 彩都パーツセンターの出庫作業では、ビン内のパーツが多様な形状・包装状態で存在する。
> 例えば、平面パーツ（天板等）は端の把持点が一意に定まらず、
> 不定形パーツやプチプチに包まれた状態のパーツは吸着のみでは対応不可能。
>
> α版では、こうした**形状・状態の複雑さ**に対してVLA制御ポリシーが有効であることを、
> 「ビンからパーツをピックして作業台に置く」という限定されたスコープで実証する。
> 個装は人が担当し、人×ロボット協働でオペレーションを成立させる。
>

### 2.2 β版 — Q4末/3月

**ゴール:** 個装・梱包を含む出庫作業サイクル全体への拡張。「ちゃんと動くこと」が最優先。

#### スコープ

| 項目 | 内容 |
|------|------|
| **IN** | ピック → 梱包材からの取り出し → 個装 → 作業台配置 |
| **個装** | ✅ 含む（挑戦的目標） |
| **梱包** | ✅ 含む |
| **移動** | △ 要検討（台車への配置まで入れるか） |
| **対象パーツ** | α版で扱った範囲を多品種に拡張 |

#### ハードウェア

| 項目 | 仕様 |
|------|------|
| **ロボットアーム** | UR5（単腕）→ 一部双腕（ラボ検証並行） |
| **エンドエフェクタ** | 吸着 / 平行グリッパ |
| **指** | 使用しない（研究は並行で進行中 — 福岡拠点） |

#### 技術要件

| # | 要件 | 詳細 |
|---|------|------|
| T-6 | 梱包材取り出し制御 | プチプチ・段ボールからのパーツ取り出し。操作軌道の汎用化 |
| T-7 | 個装スキル | パーツに合った袋への挿入、納品書同封、袋閉じ |
| T-8 | 梱包スキル | 段ボールの開封を含む梱包関連操作 |
| T-9 | ピック失敗リカバリー | リトライ・代替手順。テレオペ介入・UMI操作ログ記録の導入検討 |
| T-10 | 現場固有安全 | 異常時自動停止、人介在データ記録フロー |

#### 成功基準

| 指標 | 目標 | 備考 |
|------|------|------|
| **タスク成功率** | TBD | β版はピック単体ではなく、個装・梱包を含むサイクル全体での成功率 |
| 安定稼働 | 継続運用可能 | 「ちゃんと動くこと」が最優先 |

※ 継続学習サイクルや人介在頻度の削減は、β版の成功定義とは切り離して考える

### 2.3 GA（General Availability）— FY27〜

**ゴール:** 正式製品版。QMSを整備し、汎用顧客向けに提供可能な状態。

#### スコープ

| 項目 | 内容 |
|------|------|
| **コア** | ピック → 個装 → 梱包 → 配置（出庫作業サイクル全体） |
| **QMS** | 品質マネジメントシステムの整備・監査対応 |
| **提供形態** | RoboSyncオプション（有償） |

#### オプション（GAのコアとは別に段階的に追加）

| オプション | 内容 | 備考 |
|-----------|------|------|
| **指の制御** | デクスタラスハンドによる繊細な操作 | 長期研究成果の統合 |
| **双腕** | 双腕協調制御 | 個装・梱包で特に有効 |
| **Agentic制御** | 複数スキル切替、自律的再計画 | RFAアーキテクチャの本格活用 |
| **移動ロボット** | ステーション間移動、台車配置 | 移動系の複雑さ軸 |

#### 技術要件

| # | 要件 |
|---|------|
| T-11 | QMS整備（安全要件文書化・監査対応・品質保証プロセス確立） |
| T-12 | SLAとしての成功率定義 |
| T-13 | スケーラブルな学習基盤（継続学習の本格運用） |

#### 成功基準

| 指標 | 目標 |
|------|------|
| タスク成功率 | SLAとして定義 |
| 提供形態 | RoboSyncオプション（有償）として提供可能 |
| QMS | 監査対応完了 |
