## テーマ① 顧客に刺さるユースケース

### 指摘の核心（宮嶋さん）
> 顧客に刺さるユースケースが見えない。PoTレベルではないか。技術積み上げだけになっていないか。

### 回答

**彩都ピッキングを選ぶ理由（3点）:**

1. **RoboSyncが既に稼働** → 統合基盤があり、VLAスキルブロックとして乗せるだけ。ゼロからではない
2. **VLAの価値が最も出る環境** → 12万品番・多様な形状・包装状態。ルールベースでは対応できない多様性がある。だからこそVLAの意義が示せる
3. **顧客価値が定量化できる** → 省人化効果（処理件数・人手削減率）として測定可能。「技術デモ」ではなく「業務の一部を代替する」

**α版（9月末）のゴール:**

> RoboSyncの現場でVLAが実際に動き、省人化効果を定量計測できる状態

| IN（やること） | OUT（やらないこと） |
|--------------|-----------------|
| ビン内パーツのピック → 作業台への配置 | 個装・梱包・移動・Agentic制御 |
| 箱物・袋入り・ケーブル袋ごとが対象（★★☆以下） | 袋in袋・段ボール開封・バラ選別はβ版 |

詳細: `leading-task-alpha-beta-definition.md`参照

**FY26末（2027年3月）にどう判断するか:**
- 現場省人化の効果が定量的に示せている
- RoboSync顧客への提案ストーリーが成立している
→ この状態に至らなければ旗を降ろす判断をする

## テーマ② 活動スピード

### 指摘の核心（松山・宮嶋さん）
> 取り組みが多岐。α版の定義が曖昧。スケジュールを再検討すべき。テーマを分割して再検討を。

### 回答

**1PJ・2ストリーム体制（奥村さんと事前合意済み）:**

```
RFAプロジェクト（PM: 黄瀬）
├─ 共通基盤: DataOps / RoboSync / MS連携（横断）
├─ Stream A: α版リリース
│   リーダー: 奥村　メンバー: 桑田・鎌田　拠点: 京橋
│   権限: α版スコープ内の技術判断・現場調整を自律的に決定
└─ Stream B: VLA/Agent技術深化（β版以降）
    リーダー: 黄瀬　メンバー: 小栗・Zhu・黒川・村上・加賀屋　拠点: 横浜
```

**完全分割しない理由:**
- DataOps/MLOps/RoboSync基盤が共通 → 分けると二重化
- α版にStream Bの技術（VLAモデル）が不可欠 → 分断できない
- 「管理単位」ではなく「責任・権限のオーナー」を分離する

**スケジュール（修正）:**

| 時期 | Stream A（α版） |
|------|--------------|
| Q1末（6月） | 実験室での検証・数値評価完了 |
| Q2末（9月） | **彩都現場での実証・CA判定** |
