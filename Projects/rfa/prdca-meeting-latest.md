## 背景: Problem Statement

現在の模倣学習パイプラインの3つの課題：
1. **テレオペレーションコスト**: タスクごとに数千のエキスパートデモが必要。新環境/製品のたびにゼロから再収集
2. **汎化失敗**: 狭い分布で訓練したポリシーはわずかな変動で失敗。同カテゴリ内の物体間でも転移不可
3. **エラーリカバリなし**: デプロイ済みポリシーに失敗の概念がない。全失敗に人間介入が必要

> *"Deploying physical robots to many new environments to collect data with enough coverage and diversity is economically and practically intractable."* — EgoBridge, NeurIPS 2025

## 提案1: EgoBridge + Steering Vectors（メイン提案・野心的）

### EgoBridge（ベース論文）
- **出典**: Georgia Institute of Technology, NeurIPS 2025
- **成果**: ベースライン比+44%改善、33%ゼロショット行動汎化、3つの実世界タスクでバイマニュアル検証

### 課題: 4つのドメインギャップ
| ギャップ | 内容 |
|---------|------|
| **Visual Gap** | 人間の肌 vs 金属グリッパー。エゴ視点でも根本的に異なる |
| **Sensor Gap** | ロボットはRealSense D405手首カメラ。人間データには無いモダリティ |
| **Kinematic Gap** | 人間: xyz位置のみ。ロボット: xyz + オイラー角 + グリッパー状態。関節制限・到達範囲が異なる |
| **Speed Gap** | 人間は2-3倍速く操作。単純なフレームペアリングでは意味的に不整合 |

→ **結果**: 共変量シフト（Covariate Shift）が発生。共訓練しても別クラスタを学習してしまう

### EgoBridgeの手法（3ステップ）
1. **DTW Behavioral Pairing**: Dynamic Time Warping で速度差に関係なくアクション系列をアライン。各ロボットデモに最も類似した人間軌道を疑似ペアとして発見
2. **Shaped OT Cost Matrix**: DTWペアにコスト割引 → Optimal Transport で輸送計画T*を発見（Sinkhorn, ε=0.05）
3. **Joint Distribution Alignment**: 標準手法は周辺分布P(f_φ(O))のみアライン → EgoBridgeは同時分布P(f_φ(O), A)をアラインし、行動対応を保存。OTロスをBCロスに加算（α=0.7）

### Steering Vectors（独自拡張提案）
- **着想**: LLMの表現ステアリングから応用
- **原理**: 特徴空間内で「Sim→Real」「人間映像→ロボット」の方向に対応するベクトルを発見
- **手順**:
  1. EgoBridgeで人間/ロボットの特徴分布を統合
  2. シミュレーションデータと実データの差に対応する方向ベクトルを特定
  3. そのベクトルを適用して、新しいソースドメインデータを実ロボット分布にシフト
- **利点**: ワンショットドメイン転移の可能性
- **制限**: 環境が大きく変わると汎化しない可能性。環境固有のFTが必要かもしれない

### 技術計画（3ステップ）
1. **Train Base Policy**: ロボットテレオペデータのみで訓練。エンコーダf_φがタスク特徴空間を学習。以後凍結。アーキテクチャ変更不要
2. **Compute Δz**: 5-10のペア（ソース, ロボット）観測を収集。Δz = mean(f_φ(O_robot)) - mean(f_φ(O_source))。1回のフォワードパスのみ、勾配不要
3. **Steered Inference**: z_steered = f_φ(O_source) + α·Δz（αは推論時に調整可能）。ポリシーπ_θ(z_steered)がソースドメイン観測から直接ロボットアクションを生成

### 検証する転移方向
| 方向 | 内容 |
|------|------|
| sim → real robot | シミュレーション訓練ポリシーをxArm 7にデプロイ |
| human egocentric → robot | Aria映像 → ロボット実行 |
| new scene / lighting | 同タスク・異なる視覚コンテキスト → 同ポリシー |
| robot A → robot B | Δzによるエンボディメントシフト |

### 拡張: Pure Videoからのステアリング
- ΔzをTAPIR/CoTrackerによるポイントトラッキングで推定 → 疑似アクションラベル生成
- **Ariaグラス不要** → 既存の工場映像がデータソースとして使用可能に

## 提案2: Human Hand Grasp Transfer to Robot Gripper（低リスク・把持特化）

### コンセプト
- 人間の手の把持を「テンプレート」として使用 → ロボットグリッパー・新オブジェクトへ転移
- **ロボット把持データ不要** / ゼロショットで新オブジェクトに対応 / 低リスク・インクリメンタル

### CMT Diffusion の手法と成果
- **核心**: Object-Centric表現 — Contact map (c ∈ R^{m×1}), Part map (p ∈ R^{m×16}), Direction map (d ∈ R^{m×3}) すべて物体表面上に定義。手/グリッパー形状に非依存
- **Dual-Branch Conditional Diffusion**:
  - Template branch: 既知把持をPointNet++でエンコード → h_e
  - Target branch: 物体点群 + h_e + BERT言語条件でデノイズ
  - Bidirectional adaptation moduleで幾何的類似性を学習
  - Cascaded生成: contact → part → direction
- **成果**: Seen SR 79% / Unseen SR 74%（RealDex 42%/30%, DexGYS 42%/39%を大幅上回る）

### 技術計画
- **データ収集**: 人間がAriaグラス着用→対象物を把持。3Dハンドキーポイント → MANOで物体点群に投影。ロボット知覚と同じハードウェア
- **スコープ**: 2-3のPCO物体カテゴリ。カテゴリあたり5-10の人間把持を収集。人間テンプレートで訓練 → Seen/Unseen物体で評価
- **拡張 — Pure Video転移**: 3Dオブジェクトモデル（CAD/スキャン/VGGT再構築）があれば、通常のRGB映像からハンドポーズ推定で接触マップを抽出可能。工場にはCADモデルが多い
- **拡張 — Error Recovery**: Ariaで連続記録 → (失敗→修正)ペアが自動で得られる。リカバリデータでの共訓練で明示的な失敗回復行動を獲得

## 主要な議論ポイント

### 1. 両提案の課題認識（大坪さんの質問）
- **共通目的**: テレオペレーションの最小化
- **データソース優先順位**: ①Ariaグラス等のエゴ視点 > ②通常映像 > ③シミュレーション
- **Steering Vectors**: 野心的だがリスクあり。スコープを限定（固定環境・特定タスク）すればリスク軽減可能
- **CMT**: 低リスクだが把持限定

### 2. データ収集戦略（黄瀬・大坪・Konstantinos・Luca）
- **現状**: 神戸工場と大阪彩都倉庫でのデータ収集を検討中
- **カメラ**: 主にサードパーソンビュー（固定カメラ）。エゴセントリックは作業者への負担で困難
- **黄瀬の見解**: サードパーソンビューはエゴセントリックより情報量が少なく、オクルージョンも多い → **追加デバイスの検討が必要**
- **PRDCA推奨**: Ariaグラスまたは同等品 > ZED2iカメラ（深度+3Dマッピング、$500程度） > 安価なWebカメラは避けるべき
- **最低構成**: 右カメラ + 左カメラ + 手首/頭部カメラの3台。オクルージョン最小化のため

### 3. タスクスコープ（Konstantinos・奥村）
- **ラップトップ組立**が候補タスクとして言及
- **スコープ限定が重要**: 固定ロボット＋特定パーツ＝成功率向上。複数パーツ/環境は指数的に困難に
- **シングルアーム vs バイマニュアル**: 未確定（要回答）
- **オブジェクト情報**: サイズ・形状・新規物体の出現頻度 → 要共有

### 4. VLA互換性（奥村の重要質問）
> **奥村**: 「この技術は任意のVLAモデル（π₀, Gemini Robotics等）に適用可能か？」
> **Konstantinos**: 「DTWはモデル非依存。Optimal Transportもロス関数なので、他のエンコーダに適応可能。大規模事前学習済みエンコーダから始めることもできる」
> **奥村**: 「最新VLAモデルを使えることはConnectにとって最重要基準」

### 5. 環境固有FTのコスト（奥村の質問）
- **奥村**: 環境固有FTのコストが十分小さければ非常に価値ある技術
- **Konstantinos**: 訓練コストは問題ないと考える。GPUアクセスあり。前回ICLR投稿時と同程度

## PCI視点での評価メモ

### ポジティブ
- **EgoBridge + Steering Vectors**は前回議論のデータスケール問題に直接対応
- VLA非依存（任意のVLAモデルに適用可能）→ PCIの「OSS VLA活用」戦略と整合
- 人間映像→ロボットの意味的転移 → 前回フィードバックの核心に合致
- 環境固有FTのコストが小さければ実用価値大

### 懸念・要確認
- Steering Vectorsは環境汎化の限界が不明 → 「特定環境向け」に留まるリスク
- CMTは把持限定 → 組立等の複雑タスクには不十分
- エゴセントリックデータ収集の現場実現性 → 作業者への負担・データ合意問題
- **Gemini Robotics-ER等との差別化**（前回と同じ質問が依然有効）

### 前回からの進展
| 項目 | 前回 | 今回 |
|------|------|------|
| アプローチ | GRASP-VLM（4D軌道基盤モデル） | EgoBridge + Steering Vectors（ドメイン適応） |
| データ戦略 | 大規模実データ（NVIDIA等） | 少量テレオペ + 人間映像の転移学習 |
| VLA互換性 | 不明確 | 明確に「任意のVLAに適用可能」 |
| 実用性 | 研究段階 | 環境限定なら比較的実用的 |
