## 1. リーディングタスクの工程分解と分析

### 1.1 α版スコープの再確認

**対象作業:** 彩都パーツセンター出庫作業のうち「ビン内パーツのピック → 作業台配置」

```
  オートストアから       UR5e（単腕）       作業台
  ビンが到着            + 吸着/グリッパ
  ┌──────────┐         ┌──────┐         ┌──────────┐
  │ 多様な    │  ピック  │      │  配置   │          │
  │ パーツ群  │ ──────→ │ 把持 │ ──────→│ パーツを  │
  │ (12万品番)│         │ 移動 │         │ 安全に置く│
  └──────────┘         └──────┘         └──────────┘
```

**制約条件:**
- 単腕（UR5e）、吸着/平行グリッパ（手動交換）
- 指の制御なし
- Agentic制御なし（単一スキル）
- 人×ロボット協働（リカバリーは人が実施）

### 1.2 工程分解 — 5フェーズ×サブタスク

α版の「ビンからパーツをピック→作業台に配置」を、認知-判断-計画-実行-確認の5フェーズに分解する。

```
Phase 1        Phase 2         Phase 3          Phase 4         Phase 5
知覚            判断             把持計画          実行             確認
(Perception)   (Decision)      (Grasp Planning)  (Execution)     (Verification)
───────────    ──────────      ──────────────    ──────────      ──────────
シーン認識      パーツ同定       把持タイプ選定     アプローチ       把持状態確認
↓              ↓               ↓                ↓               ↓
ビン内状態把握  アクセス順序判断  把持点決定        把持実行         持ち上げ確認
↓              ↓               ↓                ↓               ↓
障害物検出      包装状態判断     軌道生成          移動（搬送）     配置位置確認
                ↓               ↓                ↓               ↓
                EE選択判断       衝突回避計画      配置実行         完了判定
```

#### Phase 1: 知覚（Perception）

| サブタスク | 内容 | 入力 | 出力 |
|-----------|------|------|------|
| **1-1. シーン認識** | ビン俯瞰カメラでビン内全体を把握。仕切り構造（4区画）、パーツ配置、包装材の有無を認識 | RGB画像（ビン俯瞰カメラ） | シーン記述（パーツ数、配置、包装状態） |
| **1-2. 対象パーツ特定** | PMSからのパーツ情報（品番・名称・ビン内位置）と視覚情報を照合し、ピック対象を特定 | PMS情報 + RGB画像 | 対象パーツの位置・領域 |
| **1-3. 包装状態認識** | 対象パーツの包装状態を認識（裸、プチプチ、袋入り、段ボール内等） | RGB画像（Hand-Eye） | 包装状態ラベル |
| **1-4. 障害物・遮蔽認識** | パーツが他のパーツや緩衝材に埋もれていないか、立てかかっていないかを認識 | RGB画像（Hand-Eye + ビン俯瞰） | アクセシビリティ評価 |

#### Phase 2: 判断（Decision）

| サブタスク | 内容 | 入力 | 出力 |
|-----------|------|------|------|
| **2-1. アクセス順序判断** | 複数パーツが重なっている場合、どの順序でアクセスするか（上にあるものを先にずらす等） | シーン認識結果 | アクセス手順（前処理が必要か） |
| **2-2. 包装状態→把持戦略マッピング** | 包装状態に応じた把持アプローチの大方針を決定（直接把持 / 袋ごと把持 / 包装除去後把持） | 包装状態ラベル + パーツ情報 | 把持戦略（直接/間接/前処理要否） |
| **2-3. EE選択判断** | 吸着 vs 平行グリッパの選択。α版では手動交換のため、事前にEEが装着済みの前提で、そのEEで把持可能かの判断 | パーツ形状・包装状態・装着中のEE | 把持可否判定 + 必要ならEE交換要求 |

#### Phase 3: 把持計画（Grasp Planning）

| サブタスク | 内容 | 入力 | 出力 |
|-----------|------|------|------|
| **3-1. 把持タイプ選定** | Grasp Taxonomy に基づき、具体的な把持タイプを選定（後述セクション2参照） | パーツ形状・包装状態・EE種別 | Grasp Type |
| **3-2. 把持点決定** | 対象パーツ上のどこを把持するか（吸着面の位置、グリッパの挟み位置）を決定 | RGB-D画像 + パーツ形状 | 把持点座標（Keypoint / Index） |
| **3-3. アプローチ軌道生成** | 把持点へのアプローチ方向・角度・軌道を生成。ビン壁面・他パーツとの衝突回避を含む | 把持点 + シーン情報 | アプローチ軌道（Trajectory） |
| **3-4. 退避軌道生成** | パーツ把持後のビンからの引き上げ軌道を生成 | 把持点 + ビン形状 | 退避軌道 |

#### Phase 4: 実行（Execution）

| サブタスク | 内容 | 入力 | 出力 |
|-----------|------|------|------|
| **4-1. アプローチ実行** | 生成された軌道に沿ってEEを把持点へ移動 | アプローチ軌道 | EEが把持位置に到達 |
| **4-2. 把持実行** | 吸着ON / グリッパ閉 を実行 | 把持タイプ + 把持点 | 把持状態 |
| **4-3. 持ち上げ** | パーツをビンから持ち上げ | 退避軌道 | パーツがビンから離脱 |
| **4-4. 搬送** | ビンから作業台への移動 | 搬送軌道（P2P） | パーツが作業台上空に到達 |
| **4-5. 配置実行** | 作業台上にパーツを安全に配置（吸着OFF / グリッパ開） | 配置位置 + 配置軌道 | パーツが作業台上に配置完了 |

#### Phase 5: 確認（Verification）

| サブタスク | 内容 | 入力 | 出力 |
|-----------|------|------|------|
| **5-1. 把持状態確認** | 持ち上げ直後にパーツが正しく把持されているか確認（落下・ずれの検知） | Hand-Eye画像 + 力覚データ | 把持成功/失敗判定 |
| **5-2. 配置確認** | 作業台上にパーツが正しく配置されたか確認（位置・姿勢の妥当性） | 作業台俯瞰カメラ画像 | 配置成功/失敗判定 |
| **5-3. ビン内状態確認** | ピック後のビン内の状態を確認（他パーツを散乱させていないか） | ビン俯瞰カメラ画像 | 正常/異常判定 |
| **5-4. エラーハンドリング** | 失敗判定時のリカバリー発動（α版では人に通知） | 失敗判定結果 | リカバリー要求 or 次タスクへ |

### 1.3 各工程の難易度分析

#### 制御方式の判定基準

| 方式 | 適用条件 |
|------|---------|
| **ルールベース** | 入力が構造化されており、条件分岐で対応可能。状態空間が離散的・有限 |
| **VLA必要** | 入力が非構造的（画像ベース）、状態空間が連続的で事前に列挙不可能 |
| **VLA + RL** | VLAの模倣学習だけでは不十分で、試行錯誤による微調整が必要 |

#### 工程別難易度マトリクス

| Phase | サブタスク | 難易度 | 制御方式 | 何が難しいか |
|-------|----------|--------|---------|------------|
| **1. 知覚** | 1-1. シーン認識 | ★★☆ | VLM/VLA | ビン内の乱雑な状態を理解する必要。ただし構造化されたビン（仕切りあり）のため比較的やさしい |
| | 1-2. 対象パーツ特定 | ★★☆ | VLM + ルール | PMS情報（品番・位置）との照合はルールベース。視覚的なマッチングにVLMが必要 |
| | 1-3. 包装状態認識 | ★★★ | VLM/VLA | プチプチ、袋、段ボール内等の状態は多様。セマンティックな理解が必要 |
| | 1-4. 障害物・遮蔽認識 | ★★☆ | VLA | パーツ間の重なり・遮蔽の3D的理解が必要 |
| **2. 判断** | 2-1. アクセス順序 | ★★★ | VLA | 「何をどかせばアクセスできるか」の空間推論。ルールベースでは困難 |
| | 2-2. 把持戦略 | ★★★ | VLA | 包装状態×形状の組み合わせが膨大。暗黙知が多い |
| | 2-3. EE選択 | ★☆☆ | ルール | パーツカテゴリ→EE対応テーブルで事前に定義可能 |
| **3. 把持計画** | 3-1. 把持タイプ選定 | ★★★ | VLA | パーツ形状・包装・姿勢に応じた最適把持タイプの選択。後述のGrasp Taxonomy参照 |
| | 3-2. 把持点決定 | ★★★★ | **VLA (IndexVLA)** | **最も重要な工程。** 形状が一意でないパーツ上の最適把持点を、画像から直接推論する必要がある。IndexVLAのKeypoint生成が本領を発揮する箇所 |
| | 3-3. アプローチ軌道 | ★★★ | VLA | ビン壁面・他パーツを避けた軌道生成。特に深いビン（404mm）内での操作が課題 |
| | 3-4. 退避軌道 | ★★☆ | VLA + ルール | 基本は垂直引き上げだが、形状によっては斜めの退避が必要 |
| **4. 実行** | 4-1. アプローチ実行 | ★★☆ | VLA | 軌道追従。VLAの連続制御で10Hzのアクション生成 |
| | 4-2. 把持実行 | ★★★ | VLA | 吸着の場合は面の角度合わせ、グリッパの場合は挟み力の調整 |
| | 4-3. 持ち上げ | ★★☆ | VLA | パーツ重量に応じた速度・加速度の調整 |
| | 4-4. 搬送 | ★☆☆ | ルール | ビン→作業台のP2P移動。経路は固定的 |
| | 4-5. 配置実行 | ★★☆ | VLA + ルール | 配置位置はおおよそ決まっているが、パーツ姿勢の調整にVLAが有用 |
| **5. 確認** | 5-1. 把持状態確認 | ★★☆ | VLM + ルール | 力覚データ（閾値判定）+ 画像での確認 |
| | 5-2. 配置確認 | ★☆☆ | VLM | 作業台俯瞰カメラで配置を確認 |
| | 5-3. ビン内状態確認 | ★☆☆ | VLM | ビン俯瞰カメラで確認 |
| | 5-4. エラーハンドリング | ★☆☆ | ルール | α版では人に通知するだけ |

**凡例:** ★☆☆ = 低難易度、★★☆ = 中難易度、★★★ = 高難易度、★★★★ = 最高難易度

#### VLAが本質的に必要な箇所（まとめ）

```
知覚 ──→ 判断 ──→ 把持計画 ──→ 実行 ──→ 確認
                     ↑↑↑
                  ここが核心

VLAが最も価値を出す工程:
  ① 把持点決定（3-2）: IndexVLAのKeypoint/Index生成
  ② 把持タイプ選定（3-1）: 形状×包装×姿勢からの最適選択
  ③ 把持戦略判断（2-2）: 暗黙知の学習
  ④ アプローチ軌道（3-3）: ビン内の複雑な空間での操作

ルールベースで十分な工程:
  ① EE選択（2-3）: カテゴリテーブル
  ② 搬送（4-4）: P2P固定経路
  ③ エラー通知（5-4）: 閾値ベース
```

### 1.4 池内先生のGMR/ICF/PTG/STGフレームワークとのマッピング

#### GMRフレームワークとの対応

α版の「ピック→配置」は **1つのGMR（Grasp → Manipulation → Release）** として構造化できる。

| GMRフェーズ | 対応する工程Phase | 具体的な内容 |
|------------|-----------------|-------------|
| **Pre-Grasp** (前処理) | Phase 1-2: 知覚・判断 | ビン内の状況認識、パーツ特定、把持戦略決定 |
| **Grasp** (把持) | Phase 3-4 (前半): 把持計画・アプローチ・把持実行 | 把持点決定→アプローチ→把持→持ち上げ |
| **Manipulation** (操作) | Phase 4 (後半): 搬送 | ビンから作業台への移動 |
| **Release** (解放) | Phase 4-5: 配置・確認 | 作業台への配置→完了確認 |

```
   GMR構造とPhaseの対応

   ┌──Pre-Grasp──┐┌────Grasp────┐┌─Manipulation─┐┌──Release──┐
   │ P1    P2    ││ P3     P4前 ││   P4後        ││ P4末  P5  │
   │知覚  判断   ││計画   実行前││   搬送        ││ 配置  確認│
   └─────────────┘└─────────────┘└───────────────┘└───────────┘

   ※ Pre-GraspはGMR原論文では明示されていないが、
     VLAベースの実装では不可欠なフェーズ
```

**重要な論点:** α版でビン内のパーツが「プチプチに包まれて埋まっている」場合、**単一GMRでは表現しきれない可能性**がある。

```
  ケース1: 直接把持可能 → 1 GMR
    Grasp(パーツ) → Manipulation(搬送) → Release(配置)

  ケース2: 障害物の除去が必要 → 2 GMR
    GMR-0: Grasp(障害物) → Manipulation(ずらす) → Release(仮置き)
    GMR-1: Grasp(パーツ)  → Manipulation(搬送)   → Release(配置)

  ケース3: 包装ごと把持 → 1 GMR（ただし把持対象が「パーツ+包装」複合体）
    Grasp(パーツ+プチプチ) → Manipulation(搬送) → Release(配置)
```

→ **α版では「ケース1 + ケース3」を主対象とし、ケース2（前処理GMR）はスコープ外とすべきか、池内先生に議論したい。**

#### ICF（接触状態）遷移としての表現

ピック→配置の接触状態遷移:

```
  時間 →

  パーツ:    [ビン底面に面接触(PC)]  →  [ロボットが把持(PC/PR)]  →  [空中移動(NC)]  →  [作業台に面接触(PC)]
  ロボット:  [非接触(NC)]           →  [パーツに接触(PC)]       →  [把持維持(PC)]  →  [非接触(NC)]

  ICF遷移:
    ① NC → PC (ロボットがパーツに接触) ← Grasp
    ② PC → NC (パーツがビンから離脱)   ← Detach (PTG11)
    ③ NC → PC (パーツが作業台に接触)   ← Attach (PTG13)
    ④ PC → NC (ロボットがパーツを解放) ← Release
```

#### PTG/STGとの対応

| 工程 | タスクグループ | コード | 説明 | 物理/意味 |
|------|--------------|--------|------|-----------|
| ビンからのピック | PTG | **PTG11 (Detach/Picking)** | 物理的な分離。ビン底面からパーツを引き離す | 純物理制約 |
| 搬送 | PTG | **PTG12 (Bringing)** | パーツを持って移動する | 純物理制約 |
| 作業台への配置 | PTG | **PTG13 (Attach/Placing)** | 作業台上に配置する | 純物理制約 |
| 緩衝材からの慎重な取り出し | STG | **STG11 (CarefullyDetach)** | パーツを傷つけないよう慎重に取り出す | **意味的制約** |
| プチプチ内パーツの取り出し | STG | **STG11 (CarefullyDetach)** | 包装を壊さず、パーツを傷つけず取り出す | **意味的制約** |
| 壊れ物の慎重な配置 | STG | **STG13 (CarefullyAttach)** | パーツの向き・衝撃を考慮した配置 | **意味的制約** |

**重要な示唆:**
- α版は一見「単純なピック＆プレース」だが、包装状態の多様性により **STG領域（意味的制約）が支配的**
- これは梱包タスクの分析（iREXデモ以降の議論）と同じ構造：「物理的には単純だが意味的には複雑」
- **VLAが価値を出す本質的な理由:** STG11/STG13のSemantic Constraintを暗黙的に学習できる

#### Semantic Constraint の具体例（α版）

| 制約タイプ | 彩都ピッキングでの具体例 | 物理的には可能だが避けるべき動作 |
|-----------|------------------------|-------------------------------|
| **Semantic Pin** | ランプ・光学部品のガラス面を触らない | ガラス面を直接把持する（物理的には可能だが傷がつく） |
| **Semantic Wall** | プチプチ包装を破かない | プチプチに強い力をかけて引っ張る（物理的には取り出せるが包装が破損） |
| **Semantic Constraint（暗黙）** | 袋入り小型部品は袋ごと持つ | 袋を開けて中身だけ取り出す（物理的には可能だが手順違反） |

### 1.5 難易度グラデーション — パーツ形状・包装状態別マトリクス

#### パーツカテゴリ × 包装状態 の工程難易度マトリクス

各セルは「工程全体の総合難易度」を示す。★が多いほど難しい。

| パーツカテゴリ | 裸（包装なし） | プチプチ包装 | 袋入り | 段ボール小箱 |
|--------------|--------------|-------------|--------|-------------|
| **A: ランプ・光学部品** | ★★★ (把持点が繊細) | ★★★★ (慎重さ+包装認識) | ★★☆ (袋ごと把持) | ★★☆ (箱ごと把持) |
| **B: 平面パーツ（天板等）** | ★★★ (把持点不定) | ★★★★ (薄い+包装) | ★★★ (袋が滑る) | ★★☆ (箱ごと把持) |
| **C: 箱物・ユニット** | ★☆☆ (直方体=容易) | ★★☆ (包装除去判断) | ★★☆ (袋ごと把持) | ★☆☆ (箱ごと把持) |
| **D: ケーブル・フレキ** | ★★★★ (不定形) | ★★★★★ (最高難度) | ★★★ (袋ごと把持) | ★★☆ (箱ごと把持) |
| **E: 小型部品** | ★★★ (小さい) | ★★★ (認識困難) | ★★☆ (袋ごと把持) | ★☆☆ (箱ごと把持) |
| **F: ゴム・シール類** | ★★★★ (変形) | ★★★★ (変形+包装) | ★★★ (袋ごと把持) | ★★☆ (箱ごと把持) |

#### 工程Phase別の難易度集中パターン

```
        Phase1   Phase2    Phase3     Phase4    Phase5
        知覚     判断      把持計画    実行      確認

裸の箱物  ★☆     ★☆       ★☆        ★☆       ★☆     ← ルールベースでほぼ対応可
プチプチ  ★★★    ★★★      ★★★★      ★★★      ★★     ← VLAが全フェーズで必要
ケーブル  ★★★    ★★★★     ★★★★★     ★★★★     ★★★    ← 最も困難
袋入り    ★★     ★★       ★★        ★★       ★☆     ← 「袋ごと」戦略で難易度緩和
段ボール箱 ★☆    ★☆       ★★        ★★       ★☆     ← 比較的容易
```

**示唆:**
- 「袋ごと把持」戦略は多くのカテゴリで難易度を大幅に下げる → α版の初期フェーズで有効
- プチプチ包装とケーブル類がVLA制御の本質的な検証対象
- 裸の箱物はルールベースで十分 → VLAの価値証明にはならない
- **α版の検証パーツ選定は「難易度★★★以上のセル」を重点的にカバーすべき**

## 2. Grasp Taxonomy（把持分類）の分解

### 2.1 Cutkosky Grasp Taxonomy の概要

Cutkosky (1989) が提唱した把持分類体系。人間の手の把持を16種類に分類し、階層的なツリー構造で整理した。

```
                    Human Grasp
                    ├── Power Grasp（力把持）
                    │   ├── 大型物体把持
                    │   │   ├── Large Diameter (円柱把持)
                    │   │   ├── Small Diameter (棒状把持)
                    │   │   └── Medium Wrap (中型包み込み)
                    │   ├── 平面把持
                    │   │   ├── Adducted Thumb (親指添え)
                    │   │   └── Light Tool (軽量ツール)
                    │   └── 側面把持
                    │       ├── Lateral Pinch (側面つまみ)
                    │       └── Hook (フック)
                    │
                    └── Precision Grasp（精密把持）
                        ├── 円筒精密把持
                        │   ├── Prismatic (角柱精密)
                        │   │   ├── 2-Finger Pinch
                        │   │   └── 3-Finger (Tripod)
                        │   └── Circular (円形精密)
                        │       ├── Thumb-2 Finger
                        │       └── Thumb-3 Finger
                        ├── Disk (円盤把持)
                        ├── Sphere (球把持)
                        └── Tip Pinch (指先つまみ)
```

#### 基本2分類の特性

| 特性 | Power Grasp（力把持） | Precision Grasp（精密把持） |
|------|---------------------|--------------------------|
| **目的** | 安定性・力の伝達 | 精密性・器用さ |
| **接触面積** | 大（手のひら全体） | 小（指先のみ） |
| **自由度** | 低（物体を拘束） | 高（微調整可能） |
| **力** | 大 | 小 |
| **ロボットでの実現** | 平行グリッパ / 吸着 | 多指ハンド / 精密グリッパ |
| **α版での対応** | ✅ 平行グリッパ / 吸着 | △ 限定的（2F-85のストローク内） |

### 2.2 彩都パーツへの Grasp Taxonomy 適用

#### α版のエンドエフェクタで実現可能な把持タイプ

α版で使用するEEは **Robotiq 2F-85（平行グリッパ）** と **SQUARE Hand SQHW-01（吸着パッド）** の2種。

```
  Cutkosky Taxonomy          α版EEでの対応

  Power Grasp
    Large Diameter  ──→  2F-85（ストローク85mmまで）
    Medium Wrap     ──→  2F-85（中型物体の挟み込み）
    Hook            ──→  ❌ 非対応
    Lateral Pinch   ──→  2F-85（薄い物体の側面把持）

  Precision Grasp
    2-Finger Pinch  ──→  2F-85（小型部品のつまみ）
    Tripod          ──→  ❌ 非対応（3指なし）
    Tip Pinch       ──→  ❌ 非対応（指先制御なし）

  吸着（Cutkoskyの範囲外）
    Vacuum Grasp    ──→  SQHW-01（平面・半平面パーツ）
```

#### パーツカテゴリ × Grasp Type マッピング

| パーツカテゴリ | 主要Grasp Type | EE | 把持の詳細 | 難易度 |
|--------------|---------------|-----|----------|--------|
| **A: ランプ・光学部品** | Medium Wrap / Lateral Pinch | 2F-85 | 円柱部分をグリッパで挟む。ガラス面を避ける | ★★★ |
| **B: 平面パーツ（天板）** | Vacuum Grasp | SQHW-01 | 平面に吸着。端の把持点が定まらない問題 | ★★★ |
| **B': 平面パーツ（プチプチ付）** | Lateral Pinch → Vacuum | 2F-85 → SQHW-01 | プチプチ表面に吸着不可の場合、端を挟む | ★★★★ |
| **C: 箱物・ユニット** | Medium Wrap / Large Diameter | 2F-85 | 直方体を挟む。最も標準的なケース | ★☆☆ |
| **D: ケーブル（袋入り）** | Lateral Pinch | 2F-85 | 袋の端を挟んで持ち上げる | ★★★ |
| **D': ケーブル（裸）** | — | — | 不定形。α版では対象外とする判断が妥当 | ★★★★★ |
| **E: 小型部品（袋入り）** | Lateral Pinch / 2-Finger Pinch | 2F-85 | 小袋ごと把持 | ★★☆ |
| **F: ゴム・シール類** | Vacuum Grasp / Lateral Pinch | SQHW-01 / 2F-85 | 変形するため吸着が安定しにくい | ★★★★ |

### 2.3 エンドエフェクタ選択基準とGrasp Taxonomyの対応

#### 吸着 vs グリッパの選択判断フロー

```
                対象パーツの特性
                      │
            ┌─────────┴─────────┐
            │                   │
       平面あり？            平面なし
       ┌────┴────┐              │
       │         │              │
   吸着面が      吸着面が     グリッパへ

```

#### Grasp Taxonomy ↔ EE選択 対応表

|-----------------------|---------|--------|--------|
| **Vacuum Grasp** (拡張) | 平面あり、表面が滑らかで吸着可能 | SQHW-01 | — |

| **Medium Wrap** | 中型の箱物・ユニット | 2F-85 | SQHW-01 |

| **2-Finger Pinch** | 小型部品、小袋 | 2F-85 | — |

### 2.4 VLAとGrasp Taxonomyの関係

#### VLA PolicyにおけるGrasp Typeの扱い方

現在のVLAアーキテクチャ（Pi0ベース）は、**画像入力→関節角度出力** の端到端（end-to-end）学習を行う。Grasp Typeは**明示的に入力されない**。

```
  現状のVLAアプローチ:
  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ RGB画像   │────→│ VLA Policy│────→│ 関節角度  │
  │ + 言語指示 │     │ (Pi0)    │     │ (q1-q6)  │
  │ + ロボ状態 │     │          │     │ + EE指令  │
  └──────────┘     └──────────┘     └──────────┘
                        ↑
                  Grasp Typeは
                  暗黙的に学習される
```

**問題点:**
1. **Grasp Typeが暗黙的** → 「なぜその把持を選んだか」の説明性がない
2. **学習データの偏り** → 特定のGrasp Typeに偏ると、他のタイプで失敗しやすい
3. **EE切替の判断** → 吸着/グリッパの選択がVLAの推論に含まれない

#### IndexVLAによるGrasp Typeの明示的表現

IndexVLAは、LLMとVLAの「通訳」として、**Grasp Typeを含むIndex情報**をVLAに提供できる。

```
  IndexVLAアプローチ:
  ┌──────────┐     ┌──────────────┐     ┌──────────┐     ┌──────────┐
  │ LLM Agent │────→│ IndexVLA     │────→│ VLA Policy│────→│ 関節角度  │
  │ 「天板を  │     │              │     │ (Pi0)    │     │ + EE指令  │
  │  取って」 │     │ Index生成:   │     │          │     │          │
  │           │     │ ・Grasp Type │     │          │     │          │
  │           │     │ ・把持点KP   │     │          │     │          │
  │           │     │ ・軌道方向   │     │          │     │          │
  └──────────┘     └──────────────┘     └──────────┘     └──────────┘
```

**IndexVLAでGrasp Typeを扱う2つのアプローチ:**

| アプローチ | 方式 | メリット | デメリット |
|-----------|------|---------|-----------|
| **A: Grasp Type as Language** | 「Lateral Pinchで袋の端を掴め」と言語指示に含める | シンプル。既存LLMで対応可。解釈可能 | 言語の曖昧性。正確な把持点には不十分 |
| **B: Grasp Type as Index** | Grasp Typeに対応するKeypoint/Vector（把持点、接近方向、把持面法線）を明示的にIndex化 | 精密。EE選択との対応が明確。物理的に整合的 | Index設計が複雑。学習データへのアノテーションが必要 |

**推奨:** α版では **アプローチA（言語ベース）から開始** し、精度が不十分な場合にアプローチB（Indexベース）に拡張する。

#### 学習データとGrasp Taxonomyの関係

| 観点 | 詳細 |
|------|------|
| **データ収集時の分類** | テレオペデータ収集時にGrasp Typeをメタデータとして記録すべき。後のファインチューニングでGrasp Type別の性能分析が可能になる |
| **データバランス** | Power Grasp系のデモが多くなりがち。Precision Grasp系、Vacuum系のデモを意識的に増やす必要あり |
| **Grasp Type別のPolicy分離 vs 統合** | iREXデモでは物体別にPolicy分離 → 池内先生からGrasp Type別が本質的と指摘。「Lateral Pinch Policy」「Vacuum Policy」のようにGrasp Type軸で分離する選択肢 |
| **Cross-Grasp Transfer** | あるGrasp Typeで学習した知識が、別のGrasp Typeに転移するか。VLAのプレトレーン知識がどこまで活用できるか |

### 2.5 IndexVLA論文へのインプリケーション

Grasp TaxonomyがIndexVLAの設計に与える影響:

#### (1) Indexカテゴリの拡張

現在のIndexVLAのIndexカテゴリ（P2P / Trajectory / Vector）に対して、**Grasp Type固有のIndex構造**を追加定義する可能性:

| 現Indexカテゴリ | Grasp Taxonomyとの対応 | 拡張の可能性 |
|---------------|----------------------|-------------|
| **P2P** (始点→終点) | 基本的なPick-and-Place。Grasp Typeは暗黙 | Grasp Typeを第3のパラメータとして付与 → (start, end, **grasp_type**) |
| **Trajectory** (軌道) | 拭き・なぞり系。把持というよりは操作系 | そのまま |
| **Vector** (object→target→action) | コネクタ挿入等。Vector方向がGrasp方向と一致しうる | Grasp方向ベクトルを明示化 → (p_object, p_target, v_action, **v_grasp**) |

→ **新カテゴリ "Grasp Index" の提案:** (把持点, 接近方向, 把持面法線, Grasp Type) の4要素で構成

#### (2) Affordance → Grasp Type → Index の3段階推論

```
  [物体認識]        [Grasp推論]          [Index生成]         [Policy実行]
  ┌─────────┐     ┌─────────────┐     ┌─────────────┐     ┌──────────┐
  │ VLM:     │────→│ VLM/LLM:    │────→│ IndexVLA:    │────→│ VLA:     │
  │ 物体の   │     │ Affordance  │     │ 把持点KP     │     │ 操作軌道 │
  │ 形状認識 │     │ → Grasp Type│     │ 接近方向     │     │ 生成     │
  │ 材質推定 │     │ → EE選択    │     │ 退避方向     │     │          │
  └─────────┘     └─────────────┘     └─────────────┘     └──────────┘

  例: 天板（平面パーツ）
    物体認識: 薄い平面、プチプチ包装あり
    → Grasp推論: Lateral Pinch（プチプチ表面は吸着不可）、2F-85使用
    → Index生成: 把持点=(端から2cm), 接近方向=(横方向), 退避=(垂直上方)
    → Policy実行: 2F-85で端を挟み、慎重に持ち上げる軌道を生成
```

#### (3) 論文への貢献ポイント

| 貢献 | 内容 | ノベルティ |
|------|------|----------|
| **Grasp-Aware Index** | Grasp TaxonomyをIndex空間に統合し、把持タイプに応じた中間表現を生成する機構 | 既存VLAはGrasp Typeを暗黙に扱う。明示的なIndex化は新規性あり |
| **EE-Conditioned Policy** | エンドエフェクタの種類をIndex情報として含め、EE切替を含む把持戦略を学習 | マルチEE環境でのVLA学習は未開拓 |
| **Industrial Grasp Benchmark** | 彩都パーツセンターの12万品番をベースにした産業用把持ベンチマーク | 既存のGrasp Benchmarkは日用品中心。産業補修部品での評価は新規 |

## 3. 池内先生へのディスカッション論点

### 論点①: GMRの粒度設計 — 前処理GMRの分離基準

**背景:** α版の「ピック→配置」を1GMRとして設計しているが、ビン内の状態（パーツが埋まっている、プチプチに包まれている等）によっては、把持前に「障害物をずらす」「緩衝材を除ける」等の前処理が必要になるケースがある。

**質問:**
- VLA Policyの粒度として「1 GMR = 1 Policy」の仮説は、前処理を含む場合にどう拡張すべきか？
- 前処理GMR（ずらす、除ける）と本体GMR（ピック→配置）を**別Policy**にすべきか、あるいは1つのPolicyが前処理を含めて端到端で学習すべきか？
- 池内先生の論文における「Pre-Grasp操作」の概念と、GMR内部での前処理の位置づけはどうなるか？

### 論点②: Grasp TaxonomyをIndexVLAにどう統合するか

**背景:** Cutkosky Grasp Taxonomyの分類は人間の手を前提としており、ロボット（特に平行グリッパ+吸着の構成）には直接適用しにくい。一方で、把持の「意図」（安定性重視 vs 精密性重視）の区別はロボットでも重要。

**質問:**
- IndexVLAのIndex空間にGrasp Typeを含めるべきか？含めるとしたらどのレベルの粒度（Power/Precision の2分類？ さらに細かい16分類？）が適切か？
- 「Grasp-Aware Index」のような新しいIndexカテゴリを定義することは、論文としての貢献になりうるか？
- 吸着（Vacuum Grasp）はCutkoskyの分類に含まれないが、産業用ロボットでは最も頻用される。拡張分類をどう設計すべきか？

### 論点③: PTG/STGの境界とVLAの暗黙的学習の範囲

**背景:** α版のピッキングは一見PTG11（物理的Detach）だが、包装状態によってはSTG11（CarefullyDetach）の意味的制約が支配的。VLA Policyがこの境界を暗黙的に学習できるとすれば、明示的なICF遷移の構造化は不要かもしれない。

**質問:**
- VLAの暗黙的学習に委ねてよい意味的制約の範囲と、明示的にICFとして構造化すべき制約の範囲の**判断基準**は何か？
- STGのSemantic Constraint（例: 「ガラス面を触らない」「プチプチを破かない」）は、言語指示としてVLAに伝えるのが効果的か、それともデモデータから暗黙的に学習させるべきか？
- α版の検証で「VLAがSemantic Constraintを暗黙的に学習できているか」を測定する方法はあるか？

### 論点④: α版のパーツ選定 — 難易度グラデーションの検証戦略

**背景:** セクション1.5で示した難易度マトリクスに基づき、α版で最初に扱うパーツの選定戦略を議論したい。

**質問:**
- 最初のデモ/検証では、どの難易度レベルから攻めるべきか？
  - **保守的:** 箱物（★☆☆）から始めて段階的に難易度を上げる → ルールベースでも対応可能なため、VLAの価値証明が弱い
  - **攻撃的:** プチプチ包装（★★★★）をいきなり攻める → VLAの価値は明確だが、失敗リスクが高い
  - **バランス:** 袋入り小型部品（★★☆）を起点に、Lateral Pinch系の把持を確立してから拡張
- Grasp Typeの多様性を検証するために、最低何種類のGrasp Typeをカバーする必要があるか？

### 論点⑤: iREXデモの知見 — 物体別Policy分離 vs Grasp Type別Policy分離

**背景:** iREXデモではスキルを「物体別」（シェーバー、チャージャー、マニュアル）に分離した。池内先生から「物理特性による分類」が本質と指摘あり。

**質問:**
- 彩都ピッキングのα版では、Policy分離の軸を「物体別」ではなく「**Grasp Type別**」にすべきか？
  - 例: 「Vacuum Policy」「Lateral Pinch Policy」「Medium Wrap Policy」の3ポリシー
  - 各ポリシーが複数パーツカテゴリに横断的に適用される構造
- この設計は、池内先生の「シェーバーとチャージャーは同じGMR」「片手と両手は別GMR」の指摘と整合するか？
- Grasp Type別Policy分離の場合、IndexVLAは「どのGrasp Type Policyを呼ぶか」の選択を行うことになるが、これはiREXでのVLMによるスキル切替と同じ構造か？
