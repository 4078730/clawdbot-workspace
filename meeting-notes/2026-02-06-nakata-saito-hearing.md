**日時:** 2026年2月6日  
**参加者:** 黄瀬、中田真一（SCMソリューション部）、奥村亮、若杉勇作  
**目的:** 彩都パーツセンターのオートストア出庫作業の詳細理解、VLA/ロボットアーム導入可能性の検討

## 5行サマリー

1. **オートストア出庫はオーダーピッキング方式**：伝票1枚につき顧客オーダー1つが紐づいており、トータルピッキングではない（PMS起因）
2. **ロボット導入にはプロパティ登録が鍵**：入庫時にロボット把持可能フラグを立て、専用ビンに格納→専用ポートに流す制御が可能
3. **事前個装による荷姿変動が課題**：作業者が事前に個包装すると、マスターデータと実際の荷姿が異なり、ロボットの把持難易度が上がる
4. **WMSへの移行で改善の可能性**：現在のPMS（パーツマネジメントシステム）からBlue Yonder WMSへ移行中、トータルピッキング対応も可能に
5. **若杉さんのシミュレーションと連携可能**：倉庫全体のスループットシミュレーションとVLAの物理シミュレーションを連携させる構想

## 詳細サマリー

### 1. 会議概要

黄瀬が彩都パーツセンター見学後のフォローアップとして、中田真一さん（SCMソリューション部、オートストア導入経験者）にオペレーションの詳細をヒアリング。VLA適用に向けた技術的課題と対応策を整理した。

### 2. 議題別サマリー

#### 2.1 オートストアのビン構造とプロパティ登録

**ビンの間仕切り:**
- ビンには溝があり、スリットを入れて棚分割が可能
- 分割ごとにビンIDが付与され、WMSと紐づく
- 細かすぎる分割はロボットアームのアクセス精度が課題

**プロパティ登録の仕組み:**
- 入庫時に「ロボット把持可能」フラグを立てる
- WMSに商品のプロパティが登録され、出庫時に制御可能
- ロボット専用ビンを指定 → 専用ポートに流れるよう制御

**中田さんコメント:**
> 「入荷の方でプロパティ登録をしてあげることによって、ピッキング指示をかけた時に、その特定のロボットアーム用ポートにしか、その掴める商品が流れてこないという仕組みができる」

#### 2.2 彩都パーツセンターの倉庫構成

**4種類の保管エリア:**

| エリア | 特徴 | 対象商品 |
|--------|------|----------|
| **ネステナー** | 赤い棚、フォークリフトで運搬 | 大型・重量物（モニター等） |
| **一般棚** | 高頻度品エリア | 常に出る商品 |
| **オートストア** | AGVによる自動搬送 | 中頻度、小型・軽量品 |
| **ラピュタASRS** | 2月から稼働予定 | （詳細未確認） |

**配置基準:**
- 商品の**大きさ**、**重さ**、**出荷頻度**で保管場所を決定
- 高頻度品 → 一般棚（オートストア左手のエリア）
- 中〜低頻度品 → オートストア

#### 2.3 出荷指示と伝票の流れ

**出荷指示のフロー:**
1. コンピュータから出荷指示が流れる
2. 伝票発行係が紙に印刷（コピー機）
3. 伝票を棚に分類（エリア別、時間帯別、行き先別）
4. 作業者が自分の担当エリアの伝票を取得
5. 伝票のバーコードをスキャン → WMSに作業開始を通知

**伝票の仕分け基準:**
- **時間帯**: 遠方地（北海道、沖縄、九州）はトラック切り離しが早いため先に処理
- **行き先**: 京橋行き、関東行きなど
- **保管場所**: オートストア / 一般棚 / ネステナー

#### 2.4 オートストア出庫の具体的オペレーション

**作業フロー:**
```
①伝票棚から売上伝票を取得
  ↓
②タブレットにログイン（作業者識別）
  ↓
③伝票のバーコードをスキャン → WMSに作業開始を通知
  ↓
④AGVがビンをポートに搬送
  ↓
⑤商品をピック、商品バーコードをスキャン（照合）
  ↓
⑥個装作業（袋入れ、輪ゴムくくり等）
  ↓
⑦完了ボタン → 次のビンが流れてくる
  ↓
⑧全伝票処理完了 → 梱包エリアへ
```

**重要ポイント:**
- **伝票1枚 = 顧客オーダー1つ**（オーダーピッキング方式）
- 商品バーコードが異なればエラー（誤ピッキング防止）
- 流通加工（個装）はピッキングと同時に実施

#### 2.5 オーダーピッキング vs トータルピッキング

| 方式 | 特徴 | 彩都での採用 |
|------|------|-------------|
| **オーダーピッキング** | 顧客ごとにピッキング、動線が長い | ✓ 現在採用 |
| **トータルピッキング** | エリアごとに商品を集約、後で仕分け | △ WMS移行後に可能性 |

**PMSを使っている理由:**
- 彩都は「工場物流」の位置づけ（パソコン工場の出荷前倉庫が分離したもの）
- WMSではなくPMS（パーツマネジメントシステム）を使用
- Blue Yonder WMSへの移行プロジェクトが進行中

#### 2.6 ロボット導入時の技術的課題

**課題①: 事前個装による荷姿変動**
- 作業者が暇な時に事前個装 → プチプチ包装されると荷姿が変わる
- マスターデータは「標準荷姿」のサイズ → 実際と異なる
- 中身が見えない状態でバーコード読み取りで照合

**課題②: バーコード読み取り**
- 商品のバーコード位置はランダム（上面、側面など）
- ロボットがスキャナーにバーコードを向ける角度調整が必要
- 伝票バーコードは指定位置に置く運用が必要

**課題③: 仕切り内の複数SKU**
- 1つの仕切りに複数SKU（最大3つ程度）が入る場合あり
- どの商品を取るかはバーコード照合で判断

**課題④: ピース/ボール/ケースの単位**
- ネジ等は袋から1個取り出す場合あり（ピース単位）
- 人でも間違えやすい作業

#### 2.7 ロボット導入に向けた対応策

**Phase 1: 簡単なものから**
- 個装なし、マスターデータ通りの荷姿の商品を対象
- ABCランク分けでロボット把持可能な商品を選別
- 専用ポートに流す制御を実装

**Phase 2: 複雑な荷姿への対応**
- 事前個装されたものも把持できるよう拡張
- 置き方の最適化（充填率 vs ロボット把持しやすさのトレードオフ）

**レイアウト案:**
- ロボット専用ポートを設置
- ピッキング → 梱包まで一貫してロボットで処理
- コンベアで商品+伝票をセットで次工程へ

#### 2.8 梱包工程

**現状:**
- 別エリアの作業台で実施
- 箱サイズは作業者が選定（属人的）
- 梱包材（発泡スチロール、緩衝材）の使用もマニュアルベース

**課題:**
- 現在は属人的な判断が多い
- Amazon等は箱サイズ・テープ長さを自動化済み
- PMS → WMSへの移行で改善の余地あり

#### 2.9 若杉さんのシミュレーションとの連携

**若杉さんの取り組み:**
- 出庫を想定した物理シミュレーション
- 彩都倉庫全体のシミュレーションモデルも開発中
- ロボット導入時の効果検証（スループット等）

**連携の可能性:**
- VLAの物理シミュレーション + 倉庫全体のマクロシミュレーション
- 彩都はデータ取得しやすく、連携に適している

## 参考：技術用語

| 用語 | 説明 |
|------|------|
| **PMS** | Parts Management System。パーツマネジメントシステム。工場物流向け |
| **WMS** | Warehouse Management System。倉庫管理システム。Blue Yonderへ移行予定 |
| **オートストア** | AutoStore。AGVがビンを自動搬送する自動倉庫システム |
| **ラピュタASRS** | Rapyuta社のAutomatic Storage and Retrieval System |
| **ネステナー** | 重量物保管用の棚。フォークリフトで運搬 |
| **GTP** | Goods to Person。商品が人のところに来るオペレーション |
| **プロパティ登録** | 入庫時に商品の属性（ロボット把持可能等）を登録すること |
| **SKU** | Stock Keeping Unit。在庫管理単位 |
| **個装** | 個別包装。袋入れ、輪ゴムくくり等 |
| **事前個装** | 出庫前に予め個包装しておくこと |
