### S1-1. ビジョンとプロジェクト概要（7分）

**スライド案:** 1枚目=ビジョンステートメント、2枚目=RFAスタック概要図

**台本（日本語）:**

RFA — Robotics Foundation Agent は、人間の言語指示を理解して、物流・製造現場の非定型タスクをロボットが自律実行する知能システムです。

対象としているのは、物流の入出庫・梱包と、製造の組立工程。具体的にはEC倉庫でのバラ積みピッキング・箱詰め・緩衝材投入や、Let's Note組立工程でのコネクタ挿入・ケーブル配索・テープ貼付などです。

これらの現場には共通する3つの課題があります。
- 形状・包装状態の多様性（12万品番をすべてプログラムするのは非現実的）
- 暗黙知に基づく判断（「袋から1個だけ取り出す」「壊れ物は上に」）
- 人手不足の深刻化

RFAはこれを、4レイヤーのスタックで解決します。

- **Task Planning Hub** — LLM Agentによるタスク理解・計画・再計画
- **Symbol Hub** — 言語と物理世界の記号接地、VLMによる状態認識・成功/失敗判定
- **Sensorimotor Hub** — VLA（π0ベース）による視覚→行動の直接生成
- **RoboSync + Data Platform** — ロボット制御基盤、データ収集・継続学習サイクル

---

### S1-2. ビジネス課題と差別化（7分）

**スライド案:** 1枚目=現場課題、2枚目=Deploy-Time Scaling vs Training-Time Scaling

**台本（日本語）:**

ロボティクスAI分野では、BigTechが大規模テレオペ施設で力技のデータ収集を行うTraining-Time Scalingを取っています。初期投資は数百億円規模、データは管理された施設環境の成功データが中心です。

我々のアプローチはDeploy-Time Scaling。RoboSyncを通じて現場に導入した瞬間からデータ収集が始まる構造です。顧客への省人化価値の提供とデータ蓄積が同時に起きるため、データ収集の追加コストはほぼゼロ。そして現場の失敗データ・リカバリーデータは、テレオペ施設では得られない固有の学習資産になります。

もう1つの重要な差別化は「段階的自律化」です。完全自律を待たずに、ルールベースの既存RoboSyncを主力とし、VLAがその限界を補完する。失敗したら人が介在する。この「不完全だが価値がある」構成で早期に現場投入し、データを蓄積しながら自律度を高めていきます。

### S1-3. FY25実績とFY26目標（6分）

**スライド案:** 1枚目=FY25成果、2枚目=FY26目標（α/β + 4つの柱）

**台本（日本語）:**

FY25の主な成果をまとめます。

- RFAコアスタック（LLM Agent × VLA × RoboSync）の統合アーキテクチャを確立
- CXC常設展示にVLAデモを設置、RFAデモは第8回まで更新
- iREX 2025で4スキル連続実行による梱包デモを実現
- 要素技術としてKeypointVLA、IndexVLA、DRIP、MTPを確立し、トップ会議に投稿
- Responsible AI Workshopの完了

FY26は研究フェーズから現場実証フェーズへの転換年です。

**α版（9月）:** VLAピックスキルがRoboSync上で動作。ビン内パーツのピック→作業台配置。成功率90%目標。

**β版（3月）:** 出庫サイクル全体（スキャン→ピック→個装→配置）をAgentic制御で動作。定量評価完了。

4つの柱でリソースを集中：
1. RoboSync × VLA統合
2. DataOps（継続学習パイプライン）
3. Agent Operation（失敗検知・再計画・スキル切替）
4. Field PoC（彩都パーツセンター）

Microsoftとの協業は、特にDataOps基盤、シミュレーション、Responsible AIの領域で大きなシナジーがあります。

### S2-1. 技術アーキテクチャ詳細（25分）

**スライド案:** 全体構成図 → 各Hub詳細（4枚）→ 統合ルート

**台本（日本語）:**

RFAの4レイヤーをそれぞれ詳しく説明します。

**Task Planning Hub**

LLM Agentがタスクの「段取り」を担います。

自然言語指示（「この商品を梱包して」）を受け取り、サブタスク列に分解する。どの箱を使うか、どの順番で詰めるか、緩衝材はどう入れるか。

Skill Alignmentによって、各ステップでVLAスキルを使うのかRoboSyncの既存ブロックを使うのかを選択し、パラメータを決定します。

失敗時にはRe-Planningを実行。論理的な計画ミスはLLMが再計画し、局所的な制御ミスはVLAにパラメータ修正を指示する。

関連要素技術としてDRIP（前提条件→サブタスク→ゴールの構造化）とMTP（マルチステップ計画）があります。

**Symbol Hub**

言語世界と物理世界の間を接続する中間層です。

LLMが扱う記号表現（「重い箱」「壊れ物」「左側の棚」）を、VLAが扱える座標・Keypoint・マスクに変換する。逆に、VLAの観測結果を「緩衝材がまだ不足している」のようなLLMが理解できる状態記述に変換する。

VLMによる状態認識で、成功/失敗の判定や次に実行すべきスキルの決定を行います。

IndexVLAはここの核心技術で、池内先生（京大名誉教授）のパースの記号論をベースに、タスクやオブジェクトの種類に応じた適切なスキル選択のための「Index」表現を導入しています。RA-L投稿予定。

**Sensorimotor Hub**

実際の物理動作を生成するレイヤーです。

VLA Policy：π0をベースモデルとして、カメラ画像＋ロボット状態＋タスク指示から連続的なロボット動作を生成します。学習コスト目安は約200エピソード/90分のデータ収集、スキルあたり約1日の学習です。

KeypointVLA：物体や手先のKeypointを中間表現として用いるVLA。コネクタ挿入や微細な位置調整など、高精度が求められるタスク向けです。

HIL Interface：自律リカバリーが困難な場合は人間の教示を受け入れ、その教示データを継続学習に利用します。

**RoboSync統合**

2つのルートがあります。

ルート①（Agent経由）：LLM AgentがMCP経由でRoboSyncに接続。将来的な完全自律化ルート。
ルート②（VLAスキルブロック）：VLAをRoboSyncのスキルブロックとして提供し、GUIで配置・組み合わせ。**FY26最優先の統合ルート。**

MCPを共通プロトコルとして、RoboSync、WMS/MES、将来的にはGemini-ERなどと疎結合に接続します。

### S2-2. データ戦略と学習パイプライン（20分）

**スライド案:** データピラミッド → ライフサイクル図 → 現状と課題

**台本（日本語）:**

RFAのデータ戦略は「使いながら賢くなる」継続学習サイクルが核心です。

**データソースの3層構造：**

| 層 | ソース | 特徴 | 役割 |
|----|--------|------|------|
| 上位 | Human Teleoperation | 最高価値・高コスト | 初期スキル獲得、高精度Fine-tuning |
| 中位 | Synthetic Data | スケール可能、Sim2Realギャップあり | バリエーション拡張、ロバスト化 |
| 下位 | Human Videos | 大量に存在、ロボットコマンド含まず | 事前学習、意味表現の土台 |

**データライフサイクル（RoboSync経由）：**

1. **Model Deploy** — 学習済みポリシーをRoboSync経由でデプロイ
2. **Logging** — 実行中の画像・ロボット状態・タスク結果をログ収集
3. **Data Export** — CECクラウドへ転送、RLDS/LeRobot形式に変換
4. **Train / Fine-tune** — 現場ログ＋テレオペ＋合成データでモデル更新
5. **Re-Deploy** — 改善モデルを再デプロイ

このサイクルで、現場実行→失敗検知→人間教示→データ蓄積→学習→Policy更新→再デプロイ、というループを回します。

**現在の課題（Microsoftとの協業ポイント）：**
- データフォーマットの標準化（RLDS/LeRobot間の変換自動化）
- Fine-Tuning→再デプロイのサイクルがまだ手動 → **Azure ML Pipelines**との統合
- モデルバージョン管理の仕組みが未整備 → **MLOps基盤**
- シミュレーションによる合成データ生成 → **Microsoft Simulation環境**

### S2-3. ユースケースと実証ターゲット（20分）

**スライド案:** 物流ユースケース → 製造ユースケース → 彩都の具体 → ラボ↔現場のマッピング

**台本（日本語）:**

RFAは2つのリーディングユースケースを持っています。

**物流ユースケース（梱包）：**

EC倉庫での注文単位の梱包作業。ばら積みピッキング→箱選定→箱詰め→緩衝材投入→封緘。

RFAスタックの適用：
- LLM Agentが注文情報から段取りを生成（箱選定→詰め順序→レイアウト決定）
- Symbol Hubが「重い箱」「壊れ物」「空きスペース」を認識し座標に変換
- VLAが非定型物のピッキング・箱内配置・緩衝材投入を実行
- RoboSyncがコンベア連携等を安全に制御

**製造ユースケース（Let's Note組立）：**

コネクタ挿入、ケーブル配索、テープ貼付など。寸法公差が厳しく、力加減を間違えると破損する精密作業。

- LLM Agentが図面・BOMから工程フローを生成
- Symbol Hubがコネクタ・ビス穴・ケーブルガイドの部位を特定
- KeypointVLAが「最後の数mm」の精密位置合わせを担当
- 品種変更時は差分学習で対応

**FY26の実証ターゲット：彩都パーツセンター**

彩都パーツセンターではオートストアの出庫作業を対象にします。

作業フロー：納品書スキャン→オートストアからビン到着→パーツ取り出し→個装→台車配置。1オーダー1-3分。

動画分析の結果、ビン内は段ボール＋プチプチ＋多種多様なパーツが混在していて、「箱物だけ」というシンプルなケースはほぼ存在しない。VLAによる学習ベースの操作制御が最初のステップから必要です。

**ラボタスク↔現場タスクのマッピング：**

| ラボタスク | 現場タスク |
|-----------|----------|
| ランダム配置ビンピッキング | 商品ピッキング（物流） |
| 箱内配置タスク | 箱詰め配置（物流） |
| 柔軟物の詰め込み | 緩衝材投入（物流） |
| コネクタ挿入 | FPC挿入（製造） |
| 簡易配線タスク | ケーブル配索（製造） |

### S2-4. 要素技術と研究成果（15分）

**スライド案:** 各技術の1枚サマリー（VLA/KeypointVLA/IndexVLA/DRIP/MTP/ViReSkill）

**台本（日本語）:**

FY25で確立した要素技術を説明します。これらはすべてRFAスタックのどこかを強化する技術です。

| 技術 | Hubの位置づけ | 概要 |
|------|-------------|------|
| **VLA（π0ベース）** | Sensorimotor | 視覚→行動の汎用ポリシー。200エピソード/1日学習で1スキル |
| **KeypointVLA** | Sensorimotor | Keypoint中間表現を使った高精度VLA。コネクタ挿入等向け |
| **IndexVLA** | Symbol | パースの記号論に基づくスキル選択Index。RA-L投稿予定 |
| **DRIP** | Task Planning | 前提条件→サブタスク→ゴールの構造的タスク分解 |
| **MTP** | Task Planning | マルチステップの階層的タスク計画 |
| **ViReSkill** | Sensorimotor | スキルの視覚的表現学習 |

iREX 2025では、4スキル連続実行（ピッキング→箱詰め→緩衝材→封緘）の梱包デモを実現しました。

### S2-5. 技術的チャレンジとMicrosoftへの期待（15分）

**スライド案:** チャレンジ一覧 → 各チャレンジの詳細 → Microsoft支援マッピング

**台本（日本語）:**

現在の主要な技術チャレンジを挙げます。

**Challenge 1: 非定型物品の汎用操作**
- 12万品番の形状・包装への対応。袋物、プチプチ包装、柔軟物
- エンドエフェクタ戦略（吸着/平行グリッパ/指の選択）
- 段ボール開封、緩衝材除去などの付随作業

**Challenge 2: DataOps自動化**
- テレオペデータ＋実行ログの統合・変換の自動化
- Fine-Tuning→評価→再デプロイの自動化
- **→ Microsoft支援期待: Azure ML Pipelines、MLOps基盤**

**Challenge 3: Agentic制御**
- 複数スキルの動的切替（ピック→個装→梱包→配置）
- 失敗検知→原因分析→自律リカバリー
- バーコードスキャン・モニター操作との連携
- β版の核心チャレンジ

**Challenge 4: シミュレーション連携**
- Real2Sim / Sim2Realのギャップ
- 合成データによるデータ拡張（特にバリエーション強化）
- **→ Microsoft支援期待: シミュレーション環境構築**

**Challenge 5: Responsible AI**
- VLA推論の不確実性下での安全性保証
- OSSモデルのライセンス管理（BlackDuck）
- 倫理的データ利活用
- **→ Microsoft支援期待: RAIフレームワーク適用**
