| 項目 | 内容 |

| **拠点** | 彩都パーツセンター |

### 1.2 出庫作業フロー（動画分析 2026-02-16）

```
[1] 準備・登録
    納品書の束を作業台に持ってくる → モニター操作で作業者登録

[2] スキャン・チェック                   ← β版の対象範囲
    バーコードリーダーで納品書を1枚ずつスキャン（〜10枚/束）
    ペンでチェック記入

[3] ビン到着・ピック                     ← α版の対象範囲
    オートストアからビンが右側のポートに到着
    ビンは仕切りで4区画に分かれており、各区画に種類ごとのパーツが格納されている
    納品書に対応するパーツをピックし、作業台に配置

[4] 個装（物による）                     ← β版の対象範囲
    パーツと納品書を適切なサイズの袋に同封
    袋を閉じ、左側のワゴンに配置

[5] 次のオーダーへ
    1オーダー: 約1〜3分
```

### 1.3 α版が扱う複雑さ

α版では「オペレーションの標準化が不可能」な部分はスコープ外とし、
**商品の形状・包装状態の複雑さに対するVLA制御の有効性**を実証する。

#### α版が扱う課題

| 課題 | 具体例 |
|------|--------|
| **形状の多様性** | 天板（平面パーツ：端の把持点が一意に定まらない）、不定形パーツ、小型部品 |
| **包装状態の多様性** | 段ボール内のプチプチ包装、袋入り、仕切り付きビン |
| **取り出しにくさ** | 立てかかってつかみにくい、一度で掴めない、下に埋まっている |

> → VLAによる操作制御ポリシーで、形状・状態に応じた把持・取り出し動作を生成する

#### α版のスコープ外（β版以降）

- 個装（袋入れ・納品書同封）
- 梱包（段ボール開封含む）
- 移動（台車配置）
- 複数スキル切替のAgenticな制御

## 2. フェーズ定義

### 2.1 全体構造 — 複雑さの3軸で段階的に拡張

**設計原則:** VLAはルールベースの「置き換え」ではなく「段階的補完」。
ルールベースで対応可能なものはルールベースで処理し、VLAが困難なタスクを補完。
複雑さを3軸に分け、時間的に分離して取り組む。

| 軸 | α版（CA） | β版 | GA（参考） |
|----|----------|-----|----------|
| **手** | 吸着/平行グリッパ（手動交換） | 同左 + 自動チェンジャー検討 | 指の制御（研究成果統合、GA オプション） |
| **腕** | 単腕・VLA制御 | 双腕・Agentic制御・個装/段ボール開封 | 双腕高度化（GA オプション） |
| **移動** | なし（固定ステーション） | TBD（台車配置まで） | 移動ロボット（GA オプション） |

### 2.2 α版（CA: Controlled Availability）— Q2末/9月

**ゴール:** RoboSync上でVLA制御ポリシーが動作し、形状・包装状態が複雑なパーツの
ビン内ピック → 作業台配置を実現する

#### スコープ

| 項目 | 内容 |
|------|------|
| **IN** | ビン内パーツのピック → 作業台に配置 |
| **OUT** | 個装、梱包、台車への移動、納品書処理、Agentic制御 |
| **対象パーツ** | 形状・包装状態が複雑なパーツ（具体ケースはRoboSyncチームと選定） |
| **人の役割** | 個装・梱包・納品書処理は人が担当。リカバリー作業も人が実施（人×ロボット協働） |

> **人の介在について:** ロボットのピック失敗・異常時には人が手動でリカバリーを行う。
> β版以降ではテレオペ介入や操作ログ記録によるリカバリーデータ蓄積を目指すが、
> α版では開発工数を考慮し手動リカバリーとする。

#### 成功基準（仮）
*期初時点の目安。現場・実際のタスク・技術レベルに合わせて調整。*

| 指標 | 目標 | 備考 |
|------|------|------|
| **ピック成功率** | **90%** | 対象パーツを絞った上での成功率 |
| **リトライ込み成功率** | **95%** | 1回失敗後のリトライを含む |
| **ログ蓄積** | 稼働 | 実行ログが記録される状態 |
| **安全性** | **インシデント0** | 物理的な怪我に繋がる事故は絶対にゼロ |

#### 技術要件

| # | 要件 | 詳細 |
|---|------|------|
| T-1 | **VLA制御ポリシー** | 形状・包装状態が複雑なパーツに対して、操作軌道を生成・実行 |
| T-2 | **RoboSync統合** | VLAスキルブロックとしてRoboSync上で実装・実行・管理 |
| T-3 | **ログ収集** | 実行ログ（画像・ロボット状態・成功/失敗）の記録 |
| T-4 | **安全制御** | RoboSync標準安全制約に準拠。物理的な怪我につながる危険箇所は絶対に対処 |
| T-5 | **ルールベース連携** | ルールベースで対応可能なケースはルールベースで処理。VLAは困難なケースを補完 |

*※ Agentic要素（複数スキル切替、自律的再計画等）はα版には含まない*

#### 事業的価値（内部レビュー向け）

彩都パーツセンターの出庫作業では、ビン内のパーツが多様な形状・包装状態で存在する。
例えば、平面パーツ（天板等）は端の把持点が一意に定まらず、
不定形パーツやプチプチに包まれた状態のパーツは吸着のみでは対応不可能。
α版では、こうした形状・状態の複雑さに対してVLA制御ポリシーが有効であることを、
「ビンからパーツをピックして作業台に置く」という限定されたスコープで実証する。
個装は人が担当し、人×ロボット協働でオペレーションを成立させる。

### 4.2 α版の制御フロー

α版では **Sensorimotor Hub（VLA Policy）+ Robo Sync** のみ。Task Planning Hub、Agentic制御は不使用。

```
入力                         処理                          出力
─────                       ────                          ────
カメラ画像（RGB）    ──→  VLA Policy  ──→  関節角度制御
  640×480 / 30fps            （推論）         （q1〜q6）
  + ロボット関節角度                        + グリッパ開閉指令
  + パーツ情報（PMS）
    パーツ名・品番・
    ビン内ピック位置
        │
        ▼
    Robo Sync
    ・安全チェック（速度・力・範囲制限）
    ・UR5eへの指令送信
    ・実行ログ記録
```

#### VLAモデル（確定）

**α版採用モデル: π0 / π0.5 (Physical Intelligence)**

| モデル | 特徴 | α版での位置づけ |
|--------|------|---------|
| **π0** | Physical Intelligence社。Flow Matching + VLA。汎化性能が高い | **採用確定**。Fine-tuningベースライン |
| **π0.5** | π0の後継。マルチタスク・スケーラビリティ向上版 | **採用確定**。π0と並行評価 |

#### VLA Policy の入出力仕様（α版）

| 項目 | 仕様 |
|------|------|
| **入力: 画像** | RGB 画像（Hand-Eye カメラ）/ **640×480 / 30fps** / 深度は使用しない |
| **入力: ロボット状態** | 関節角度 6次元（q1〜q6）+ グリッパ状態 1次元 = 7次元 |
| **入力: パーツ情報** | PMSから取得するパーツ仕様（パーツ名・品番・ビン内ピック位置 ※仕切りがある場合）。言語指示ではなくパーツ仕様書として読み込む |
| **出力: アクション** | **関節角度制御** — 各関節の目標角度 6次元（q1〜q6）+ グリッパ指令 1次元 = 7次元 |
| **制御周期** | **10Hz**（確定） |

### 4.3 ルールベースとVLAの切替条件

```
                パーツ認識
                    │
            ┌───────┴───────┐
            │               │
     ルールベースで       ルールベースでは
     対応可能か？         対応不可
            │               │
            ▼               ▼
     ┌────────────┐   ┌────────────┐
     │ルールベース │   │ VLA Policy │
     │ピック       │   │ ピック      │
     │（既存）     │   │（新規）     │
     └────────────┘   └────────────┘
```

| 条件 | 制御方式 | 判定方法 |
|------|---------|---------|
| 箱物・整列済み・吸着面が明確 | ルールベース | 画像認識で物体カテゴリ判定 |
| 不定形・包装状態が複雑・把持点不明確 | VLA Policy | 上記以外（α版初期は全VLAも可） |
| VLA推論が異常値を返した場合 | **停止→人介在** | アクション値の範囲チェック |
| VLAが一定時間内にタスク完了しない場合 | **タイムアウト→人介在** | タイマー（閾値TBD） |

> **α版初期の方針:** 最初はVLAのみで動かし性能データを蓄積する。
> ルールベースとの切替は、VLAの得意/不得意が明確になった段階で導入。

### 4.4 β版の制御フロー

β版では **Task Planning Hub（LLM Agent）** が加わり、8工程のオーケストレーションを行う。

```
LLM Agent（Task Planning Hub）
│
├─ [1] バーコードスキャン → スキャンスキル呼出
├─ [2] モニター操作      → UI操作スキル呼出
├─ [3] 段ボール開封       → 開封スキル呼出（双腕）
├─ [4] パーツピック       → ピックスキル呼出
├─ [5] ビンに戻す        → 配置スキル呼出
├─ [6] 個装              → 個装スキル呼出（双腕）
├─ [7] 台車配置          → 配置スキル呼出
└─ [8] 次のオーダー      → サイクル管理

各スキルは Robo Sync 上のスキルブロックとして実装
スキル間の遷移は BT（ビヘイビアツリー）で定義
```

### 4.5 RoboSyncとの統合

| レイヤ | 責任 | 実装場所 |
|--------|------|---------|
| **タスク管理** | スキル選択・切替・再計画 | Task Planning Hub（β版） |
| **スキル実行** | VLA推論 → アクション生成 | VLAスキルブロック（RoboSync上） |
| **安全制御** | 速度制限・力制限・範囲制限・緊急停止 | RoboSync標準安全レイヤ |

### 6.3 α版の対象パーツ選定基準

| 基準 | 条件 | 理由 |

## 8. 検証計画

### 8.1 α版の検証

#### 成功/失敗の定義

| 判定 | 定義 |
|------|------|
| **ピック成功** | 対象パーツをビンからピックし、作業台上の指定位置に配置できた |
| **ピック部分成功** | ピックは完了したが、配置位置・向きが不適切。VLM判定でフラグ立て |
| **ピック失敗** | パーツを把持できなかった / 落下した / タイムアウト |

#### 評価指標

| 指標 | 定義 | α版目標 |
|------|------|---------|
| **ピック成功率** | 成功 / 総トライアル数 | **90%**（対象パーツを絞った上で） |
| **リトライ込み成功率** | 1回リトライを含む成功率 | **95%** |
| **ログ蓄積** | 実行ログが記録される状態 | 稼働 |
| **安全性** | インシデント件数 | **0件**（物理的な怪我につながる事故は絶対にゼロ） |

#### 検証ステージ

| ステージ | 場所 | 目的 |
|---------|------|------|
| **ラボ模擬環境** | 社内ラボ | ハードウェアセットアップ、VLAスキル基本動作確認 |
