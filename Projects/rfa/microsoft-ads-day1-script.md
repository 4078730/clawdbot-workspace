## Session 1: Project Overview（10:20-10:40 / 20分）

**目的:** RFAプロジェクトの全体像とビジネス課題をMicrosoftチームに共有し、「なぜこのプロジェクトが必要か」を理解してもらう

---

### S1-1. ビジネス課題（5分）

**キーメッセージ:** 物流・製造現場には「自動化の最後の砦」が残っている

**台本（日本語）:**

物流・製造の現場では、自動化が進んでいない領域がまだ大きく残っています。

特に3つの課題があります。

1つ目は**非定型物品の多様性**です。例えば彩都パーツセンターでは12万品番を扱っていて、形状も包装もバラバラです。段ボールの中にプチプチで包まれたパーツ、袋入りの小型部品、シール台紙など、商品ごとに把持方法が異なります。これを全部事前にプログラムするのは現実的ではありません。

2つ目は**作業の暗黙知**です。熟練作業者は「袋から1個だけ取り出す」「壊れやすいものは上に置く」といった判断を無意識にやっています。これはルールに書き切れません。

3つ目は**労働力不足**です。これは日本全体の構造的な問題で、現場の人手が年々減っています。

つまり、「ルールベースでは自動化できない」けど「人手には頼れない」という領域が拡大し続けています。ここがRFAのターゲットです。

---

### S1-2. RFAのビジョン（5分）

**キーメッセージ:** 言語指示→自律実行。「完全自律」ではなく「段階的自律化」

**台本（日本語）:**

RFA — Robotics Foundation Agent — は、人間の言語指示を理解して、ロボットが非定型作業を自律実行する知能システムです。

技術的には4つのレイヤーで構成されています。

- **Task Planning Hub（LLM Agent）** — 言語理解、タスク計画、失敗時のリカバリー判断
- **Symbol Hub** — 言語世界と物理世界をつなぐ変換レイヤー。VLMによる状態認識
- **Sensorimotor Hub（VLA）** — カメラ画像から直接ロボットの行動を生成する基盤モデル
- **RoboSync + Data Platform** — 物理ロボットの制御基盤とデータ収集・管理

重要なのは、我々は「完全自律」を最初から目指していないということです。段階的に自律度を上げていく戦略を取っています。最初はルールベースが主力で、VLAはそのルールベースでは対応できない部分を補完する。失敗したら人が介在する。この「不完全だけど価値がある」構成で、早期に現場導入しながらデータを蓄積していきます。

---

### S1-3. 差別化と競争戦略（5分）

**キーメッセージ:** Deploy-Time Scaling — 現場データが参入障壁

**台本（日本語）:**

BigTechのアプローチと我々のアプローチは根本的に異なります。

BigTechは大規模なテレオペ施設を作って、数百〜数千人のオペレーターを雇って、力技でデータを集めます。これをTraining-Time Scalingと呼んでいます。初期投資は数百億円規模です。

我々のアプローチはDeploy-Time Scalingです。RoboSyncを通じて現場に導入した瞬間からデータ収集が始まる構造を作ります。顧客への省人化という価値提供とデータ収集が同時に起きるので、データ収集の追加コストはほぼゼロです。

そしてこのデータ、特に失敗データやリカバリーデータは、テレオペ施設では得られない現場固有の学習資産になります。時間が経つほどデータの差は広がり、後発が追いつけなくなる。

このサイクルを構築・運用できること自体が、我々の競争優位です。

---

### S1-4. FY26の目標と4つの柱（5分）

**キーメッセージ:** 研究フェーズ→現場PoC/事業検証フェーズへの転換年

**台本（日本語）:**

FY26は研究フェーズから現場実証フェーズへの転換年です。

2つのマイルストーンを設定しています。

**α版（9月）** — VLAピックスキルがRoboSync上で動作し、ビン内パーツのピック→作業台配置ができる状態。ピック成功率90%を目標としています。

**β版（3月）** — 出庫作業サイクル全体（バーコードスキャン→ピック→個装→台車配置）をAgentic制御で動作させ、結果を定量的に説明できる状態。

これを実現するために、4つの柱でリソースを集中させています。

| 柱 | 内容 |
|----|------|
| ① RoboSync × VLA | VLAをRoboSyncのスキルブロックとして統合 |
| ② DataOps | 現場ログ→学習データ変換→再デプロイのパイプライン |
| ③ Agent Operation | 失敗検知・再計画・スキル切替の自律制御 |
| ④ Field PoC | 彩都パーツセンターでの実証 |

Microsoftとの協業は、特に②のDataOps基盤とResponsible AIの領域で大きなシナジーがあると考えています。

### S2-3. 現場実証のターゲット（15分）

**キーメッセージ:** 彩都パーツセンターの具体的な作業と、なぜVLAが必要か

**台本（日本語）:**

実証の主ターゲットは彩都パーツセンターの出庫作業です。

作業フローはこうです：
1. 納品書スキャン
2. オートストアからビンが来る
3. ビンの中からパーツを取り出す（ここがVLAの出番）
4. 個装して台車に配置

なぜルールベースでは無理か？

動画分析の結果、ビンの中は段ボール＋プチプチ＋多種多様なパーツが混在していて、「箱物だけのピック」というシンプルなケースはほぼ存在しませんでした。形状、包装、姿勢すべてが毎回違う。VLAによる学習ベースの操作制御が最初のステップから必要です。

α版では「ピック→作業台配置」に絞り、β版で出庫サイクル全体に拡張します。

### S2-5. 技術的チャレンジ（25分）

**キーメッセージ:** 具体的な技術課題と、Microsoftに期待する支援領域

**台本（日本語）:**

現在抱えている主要な技術的チャレンジを5つ挙げます。

**Challenge 1: 非定型物品の把持・操作**
- 12万品番の多様な形状・包装に対応する汎用ピックスキルの実現
- エンドエフェクタ戦略（吸着 / 平行グリッパ / 指）の選択
- 段ボール開封、緩衝材の除去など付随作業への対応

**Challenge 2: DataOpsパイプラインの自動化**
- テレオペデータ + 実行ログの統合
- RLDS/LeRobot形式への自動変換
- Fine-Tuning→評価→再デプロイの自動化
- → **Microsoft支援期待: Azure ML Pipelinesとの統合**

**Challenge 3: Agentic制御**
- 複数スキルの動的切替（ピック→個装→梱包）
- 失敗検知と自律リカバリー
- バーコードスキャン・モニター操作との連携
- → β版の核心的チャレンジ

**Challenge 4: シミュレーション連携**
- Real2Sim / Sim2Realのギャップ
- 合成データによるデータ拡張
- → **Microsoft支援期待: シミュレーション環境の構築**

**Challenge 5: Responsible AI**
- VLAの安全性保証（推論の不確実性下での動作保証）
- OSSモデルのライセンス管理
- 倫理的なデータ利活用
- → **Microsoft支援期待: RAIフレームワークの適用支援**
