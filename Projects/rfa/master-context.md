## 1. エグゼクティブサマリー & ドキュメントの目的

### 1.1 エグゼクティブサマリー

Robotics Foundation Agent（以下、RFA）は、

人間の言語指示を理解し、
物流・製造現場における 属人的かつ複雑で柔軟な推論が求められるタスク を
ロボットが自律的に実行できる知能システム

の実現を目的とする、Cognitive AI テーマの研究プロジェクトである。

本プロジェクトでは、

Task Planning Hub（LLM Agent）によるタスク理解・計画・スキル構成

Symbol Hub による言語と物理世界の記号接地・状態管理

Sensorimotor Hub（VLA Policy）による視覚・言語・行動の統合制御

ロボット制御PF（Robo Sync）と連携したデータライフサイクル基盤

を統合し、SCM領域（特に入出庫・梱包タスク）における属人作業の自働化と、将来的な事業化に向けたコア技術の確立を狙う。

FY25では、以下を主要な成果として位置づける。

RFM（ロボット基盤モデル）と LLM Agent の統合アーキテクチャの確立

SCMドメインタスク（ビンピッキング・梱包）での実機 PoC と CXC 展示による価値訴求

KeypointVLA / IndexVLA / DRIP / MTP / ViReSkill 等の要素技術の確立とトップ会議投稿

ロボット制御PFを活用した「データ収集 → 学習 → 実行」サイクルの実証と、今後のデータ戦略の足場づくり

FY26では、以下を目指す：

社内実証 2〜3現場：神戸工場（MSBD）、彩都倉庫

Agent + VLA + Robo Sync統合完了：3層構造の一気通貫動作

VLAスキルブロック化：Robo Syncから利用可能な形で提供

継続学習サイクルの実装：Microsoftとの協業によるMLOps基盤構築

FY26ロードマップ（Q1〜Q4）と α/β 定義は `Projects/rfa/fy26-roadmap.md` に集約。
- α版（9月 / Q2末）：社内実証投入、成功率70%目安
- β版（3月 / Q4末）：運用安定・継続学習稼働、成功率90%目安

中長期的には、FY27 までに SCM 領域での社会実装（パートナー現場での本番運用）を視野に入れ、ロボティクスAIにおけるパナソニック コネクトの技術的プレゼンスと事業機会の創出を目指す。

### 2.2 ビジョン・目的

ビジョン

人間の言語指示を理解し、
物流・製造現場における 属人的かつ複雑で柔軟な推論が求められるタスク を
ロボットが自律的に実行できる知能システムを実現する。

プロジェクト目的（技術面）

本プロジェクトは、LLM Agent・Robot Foundation Model（VLA）・Symbol Integration・ロボット制御PF（Robo Sync）を統合し、以下を実現することを目的とする。

抽象的な業務指示からのタスク計画

「この出荷指示書に従って梱包して」「この型番の組立ラインを立ち上げて」などの曖昧な指示から、制約条件を踏まえた説明可能なタスクプランを自動生成する。

非定型・高難度な物理タスクの汎用実行

物流における袋物・緩衝材・ランダム配置ピッキングや、
製造におけるコネクタ挿入・ケーブル配線・テープ貼付など、
従来のルールベース制御では対応が難しいタスクを、
視覚・言語・行動を統合したロボット基盤モデル(VLA)により汎用的に実行する。

現場データに基づく継続学習の仕組み

Robo Sync を通じて収集される画像・ロボット状態・タスクログを、
LLM Agent / VLA の学習に利用できる形で構造化し、
「データ収集 → 学習 / Fine-tuning → 評価 → 再デプロイ」を回せる基盤を整える。

プロジェクト目的（事業・組織面）

物流・製造ドメインにおけるEmbodiedAI/ロボティクスAIの中核技術の確立

倉庫の入出庫・梱包、工場の組立・検査といった
少量多品種・高頻度変更の現場に適用可能なコア技術を確立し、
将来のソリューション・プロダクトの技術的土台とする。

現場接続されたデモ・PoCによる事業機会創出

CXC / 展示会 / パートナー現場での実機デモを通じて、
顧客の具体的な課題と接続されたユースケースを掘り起こし、
共創・共同研究・PoC へつながる接点を拡げる。

ロボティクスAI人材・知見の集約拠点形成

RFM / LLM Agent / VLA / Robo Sync / データ基盤といった要素技術を横断的に束ね、
コネクトにおけるロボティクスAIの技術リーダーシップと人材プールを形成する。

### 2.3 プロジェクトの位置づけ・意義

#### 2.3.1 物流・製造領域への貢献

物流・製造現場（SCMの中核領域）では、

SKU の増加・変更頻度の高さ

箱物・袋物・柔軟物・ケーブルなど多様な形状・状態

品質・安全・段取りに基づく黙知（どこを持つか / どこに詰めるか / どの順番で組むか）

によって、自動化対象が依然として限定されている。

RFA は、この領域に対して以下を提供することを狙う。

言語指示に基づくタスク自働化

出荷指示・工程指示など、現場で既に運用されている情報を
LLM Agent が読み解き、ロボットが実行可能なタスクプランに変換する。

非定型物体・高難度作業の操作

物流：ばら積みピッキング、箱組み立て、箱詰め配置、緩衝材投入など

製造：コネクタ挿入、ケーブル配線、テープ貼り付け、ジグなし作業など
について、VLA によって現場の変動に強い操作スキルを提供する。

失敗前提の自律リカバリ

把持失敗や小さな位置ズレを前提とし、
再トライ・姿勢変更・代替手順への切り替えを自律的に行うことで、
「人が止めに行く時間」を無くして、安定した自律運用レベルに近づける。

#### 2.3.2 コネクト内での役割

RFA は、コネクト内のロボティクス関連要素を以下のように接続する。

Robo Sync 上に載る「頭脳」

ロボティクス研究部が提供するロボット制御PF（Robo Sync）が
多様なロボット・ハンド・センサーに対する「確実に動く身体」を担い、

RFA が、その上で
言語理解・タスク計画・柔軟な手先スキルを提供する「状況に応じて考えて動く頭脳」を担う。

データ資産化の先導役

物流・製造・店舗など複数ドメインで蓄積される作業データを、
「ロボット基盤モデルの共通データ資産」として扱うための
収集・整形・活用フローを先行して実証する。

クロスチーム連携のハブ

PRDCSG（Skill Composition / Re-Planning）、PHD / DAICC（Symbol Integration）、
SDL FUK（ロボット制御PF PoC）、CXC / Idea Forge（デモ環境）などとの連携を通じて、
コネクト内のロボティクスAI技術群を束ねる役割を担う。

#### 2.3.3 社外に対するポジショニング

グローバルなロボット基盤モデル動向との関係

π0 / Octo / Open-VLA などの OSS ロボット基盤モデルや、
Helix / GR00T などのジェネラリスト VLA の潮流を前提とした上で、

RFA はそれらを単に追随するのではなく、
「物流・製造ドメインに特化したタスク設計とデータPF」を武器に差別化を図る。

「現場接続可能なロボティクスAI」としての立ち位置

モデル単体の性能ではなく、

ロボット制御PFとの統合アーキテクチャ

具体的ユースケース（梱包・組立）に紐づいたデモ・PoC

現場データを用いた Fine-tuning / 継続学習の運用フロー
によって、「実際に現場で動くかどうか」を軸にプレゼンスを築く。

### 2.4 解決する課題（Problem Statement）

RFA が直接ねらうのは、

物流・製造現場における
「属人的かつ、作業自体が複雑で柔軟な推論が求められるタスク」を
LLM Agent と VLA の二層 (System2, System1)で自律化すること

である。

#### 2.4.1 現場レベルの課題（物流・製造のペイン）

作業設計・判断の属人化

物流：箱サイズの選定、詰め順序、壊れ物の配置などを熟練作業者が経験則で決めており、仕様化が難しい。

製造：型番ごとの工程設計、部品の持ち方・挿し方、治具の使い方などが個人ノウハウに依存している。

非定型・例外ケースへの弱さ

物流：袋物・変形箱・ラベル不良・出荷内容変更など、「いつもと少し違う」ケースが頻発する。

製造：部品のばらつき、組立途中の干渉、治具の劣化などで、教示前提の動きがすぐ破綻する。

チューニング・立ち上げ工数の肥大化

SKU変更・工程変更のたびにロボット教示・パラメータ調整・デバッグが必要となり、
少量多品種の現場では自動化投資が採算に乗りにくい。

ナレッジの形式知化・再利用の不在

「この製品はここを持つと安定する」「この順番で組むとミスが減る」といった知見が、
データやモデルとして蓄積されず、現場や案件ごとに作り直されている。

#### 2.4.2 レイヤ別の技術課題（どこに何が効くか）

(1) LLM Agent レイヤでの課題

抽象的業務指示（出荷指示書・工程票・標準作業書）から、
制約を満たすタスク分解・段取り設計 を自動生成すること。

品切れ・設備トラブル・把持失敗などの例外時に、
代替手順・人へのエスカレーション・再計画 を自律的に設計すること。

生成した計画を、「手順書・タスクビヘイビアツリー」といった
人がレビュー・編集しやすい形で可視化 すること。

(2) VLA / RFM レイヤでの課題

物流における非定型物（袋物・緩衝材・ランダム配置商品）や、
製造における高精度作業（コネクタ挿入・ケーブル配線・テープ貼付）に対して、
その場の状態に応じた把持・姿勢・力加減を生成すること。

周辺物・狭隘空間・治具を考慮し、
衝突を避けながら目標状態に到達する軌道を生成すること。

把持失敗・位置ズレ・落下などの際に、
ローカルなやり直し・姿勢変更・別ルート選択 を自律的に行うこと。

(3) データ・プラットフォーム（Robo Sync 連携）レイヤでの課題

実機動作から得られる画像・ロボット状態・タスク結果を、
LLM Agent / VLA の学習に直接利用できる 構造化データセット として蓄積すること。

導入後の現場データを用いて、
モデルの Fine-tuning・性能評価・安全確認・再デプロイ を継続的に回す仕組みを作ること。

#### 2.4.3 RFA が狙う解決ポイント

LLM Agent

物流・製造の業務指示と現場制約から、
説明可能なタスクプランとリカバリ戦略を生成し、
「段取り・判断の属人化」を減らす。

VLA / RFM

非定型物体・高難度作業に対する汎用スキルを提供し、
「複雑で柔軟な推論が求められる手先作業」をロボットに代行させる。

Robo Sync 連携を含むデータ基盤

物流・製造双方の現場から得られるデータを統合し、
使えば使うほど現場適合度と自動化対象範囲が広がる
データ駆動の改善サイクル を実現する。

## 3. ターゲットドメインとユースケース

### 3.1 ターゲットドメインの選定方針

RFA が主対象とするのは、以下の 2 つのドメインである。

物流ドメイン（倉庫・センター内の入出庫・梱包）

製造ドメイン（組立・検査ライン）

これらのドメインを選定するポイントは以下の通り。

少量多品種・高頻度変更
SKU や型番の種類が多く、品種切替えやレイアウト変更が頻繁に発生するため、
従来型の「一度教示して固定」の自動化が採算に乗りにくい。

属人的な判断・段取りの存在

物流：箱サイズの選定、詰め順序、壊れ物の配置、緩衝材の入れ方

製造：型番ごとの組立順序、どこを持つか、どこまで押し込むか、どの治具をどう使うか
といった段取り・判断が熟練者の暗黙知として存在し、仕様化・標準化が難しい。

物理作業自体の複雑さ

物流：袋物・柔軟物・ランダム配置商品のピッキング、箱詰め配置、緩衝材投入

製造：コネクタ挿入、ケーブル配線、テープ貼付、ビス締結、微小部品の取り扱い
など、視覚・力加減・接触状態の制御が難しいタスクが多い。

RFA は、これらの 「属人的かつ複雑で柔軟な推論が求められるタスク」 に対して、
LLM Agent と VLA の二層構造でアプローチする。

### 3.2 リーディングユースケース（物流：梱包タスク）

#### 3.2.1 シナリオ概要

EC 向け倉庫などを想定し、以下のような梱包プロセスを対象とする。

納品書・WMS 情報に基づく商品リストの確認

適切な箱サイズの選定・箱組立

コンテナからの商品ピッキング（箱物・袋物・柔軟物の混在）

箱の中への配置（他商品との干渉・安定性を考慮）

緩衝材の投入・固定

テーピング・ラベリング

このうち、特に以下のステップを RFA の初期ターゲットとする。

商品ピッキング（非定型物ハンドリング）

箱の中への配置（空間配置＋安定性）

箱の組み立て

#### 3.2.2 物流現場における課題

SKU・形状の多様性により、「この商品はこう入れるといい」という知識が属人化している。

箱内の空間配置が経験則に依存しており、詰め方によっては破損や偏りが発生する。

出荷指示内容の変更・欠品などにより、現場での段取り変更が日常的に発生する。

#### 3.2.3 RFA による解決イメージ

LLM Agent レイヤ

出荷指示・商品情報（サイズ・重量・ fragility 属性など）を読み取り、
「どの箱を使うか」「どの順番で詰めるか」「どこに配置するか」といった段取りを自動生成する。

欠品・代替品・レイアウト変更などがあった場合に、計画を自律的に再構成する。

VLA / RFM レイヤ

コンテナ内の商品の姿勢・形状・ occlusion 状態を認識し、
その場で安定して掴める把持姿勢を決定する。

箱内の空きスペースと既に詰められた商品との関係を認識し、
衝突を避けつつ、安定するように配置動作を生成する。

### 3.3 リーディングユースケース（製造：組立タスク）

#### 3.3.1 シナリオ概要

Let's Note 等の組立工程を代表例として、以下のような作業を想定する。

部品トレイからの部品ピッキング

コネクタ・FPC 挿入

ビス挿入・ビス締結

ケーブル配線（ツメやガイドに沿わせる）

シール・両面テープ貼付

カバー嵌合・スナップフィット

このうち、初期ターゲットとして以下を重視する。

コネクタ・FPC 挿入（高精度位置合わせ）

ケーブル配線（柔軟物の経路制御）

テープ・シール貼付（面に沿った動き）

#### 3.3.2 製造現場における課題

型番ごとに部品構成やレイアウトが異なり、ライン構成・治具設計・手順書の更新コストが高い。

コネクタ挿入や FPC 挿入などは微小な位置ズレ・角度ズレに敏感で、
教示軌道がすぐ成立しなくなる。

ケーブルやフレキなど柔軟物の取り回しは、
現物合わせのノウハウに依存しており、自動化が遅れている。

#### 3.3.3 RFA による解決イメージ

LLM Agent レイヤ

図面・工程表・部品表を読み取り、
型番ごとの組立シーケンス（どの順番でどのスキルを使うか）を自動生成する。

工程変更・部品変更時に、必要なスキル構成の変更や治具要件を再設計する。

VLA / RFM レイヤ

コネクタ・穴・ピン位置などをカメラから認識し、
微小な誤差を吸収できる挿入軌道を生成する。

ケーブル配線では、ツメやガイドの形状をもとに、
ケーブルを沿わせるための経路と把持位置・力加減を決定する。

テープ貼付では、開始点・終点・貼付面の形状を認識し、
面に沿ったスムーズな動きと接触維持を実現する。

### 3.4 ラボ検証タスクとの対応関係 (WIP)

#### 3.4.1 現在の検証タスク

RFA では、以下のようなラボタスクを通じて、実ドメインタスクの要素を切り出して検証する。

物流寄り

ランダム配置ビンピッキング（箱物＋袋物）

単一商品の箱詰め配置

柔軟緩衝材（紙・エアクッション）の投入

製造寄り

単一コネクタ挿入（治具あり / 治具なし）

単純なケーブル配線タスク

テープ貼付（平面 / 曲面）

#### 3.4.2 ユースケースとのマッピング

ラボタスクは、それぞれ物流・製造ユースケースにおける
「高難度ボトルネック作業」を抽出したものとして位置づける。

物流ユースケース（梱包）

商品ピッキング ←→ ランダム配置ビンピッキング

箱詰め配置 ←→ 箱内配置タスク

緩衝材投入 ←→ 柔軟物の詰め込みタスク

製造ユースケース（組立）

コネクタ挿入 ←→ 単一コネクタ挿入タスク

ケーブル配線 ←→ 簡易配線タスク

テープ貼付 ←→ 単純貼付タスク

この対応関係を明示することで、

研究段階のタスクが、どの現場課題に対応しているか

どのタスクを優先的にクリアすれば、どのユースケースの実適用に近づくか

をプロジェクト内外で説明しやすくする。

### 3.5 ユースケース拡張の方針（簡略）

まずは 「梱包タスク」「組立タスク」 の中で、
最も価値が高く・切り出しやすく・再利用価値のある高難度作業を VLA ブロック化する。

その上で、

SKU / 型番の拡張

環境変動（レイアウト変更・協調作業）

複数スキルの組み合わせ（マルチステップ）
へと段階的にスコープを広げる。

ユースケース拡張のたびに、

LLM Agent 側：計画・段取りのパターン拡張

VLA 側：スキルセット・汎用性の拡張

データ側：対象ドメイン・現場の追加
をセットで進める方針とする。

### 6.2 Task Planning Hub

Task Planning Hub は、RFA における計画・判断の中枢であり、
自然言語で与えられた曖昧な指示を、実行可能なタスク構造へと落とし込む役割を担う。

本 Hub は、「何をすべきか」「どの順番で進めるか」「失敗したときにどう立て直すか」
といった意思決定そのものを扱うレイヤである。

#### 6.2.1 役割

タスク理解・分解

自然言語の指示（例：「この3種類の商品を、壊れ物に気をつけて梱包して」）を入力として、

必要なサブタスクの列
（例：箱選定 → ピッキング → 箱詰め → 緩衝材投入 → 封緘）

各サブタスクの入出力
（対象物、目標状態、制約条件、終了判定）

を構造化されたタスク表現として生成する。

これにより、曖昧な言語指示を、
ロボットが扱える明示的な「段取り」へと変換する。

長期タスクの計画と更新

複数ステップ・長時間にわたるタスクに対して、

現在どこまで進んでいるかの進捗管理

条件分岐（例：商品Aが欠品していた場合）

失敗時のリトライ・代替手段・スキップ判断

を含んだ動的な計画を維持する。

単発動作ではなく、
「一連の作業を最後までやり切る」ことを目的とする。

Skill Alignment（スキル構成とスキル呼び出し）

VLA や既存の Robo Sync ブロックを、すべて「スキル」として抽象化し、

どのタイミングで

どのスキルを

どの引数（対象、目標状態、制約）で

呼び出すかを決定する。

これにより、

ルールベースの既存資産

学習ベースの VLA スキル

を同一の計画フレームで混在利用できる。

Re-Planning / リカバリー

把持失敗、対象物の落下、センサー異常などが発生した場合に、

状況の要約（ログ・センサー・画像）

原因仮説の生成

再試行 / 別スキルへの切り替え / 人へのエスカレーション

を判断し、タスクを継続させる。

状況に応じて、

論理的な計画の誤り
→ LLM によるタスク再構成

局所的な制御の誤り
→ VLA へのパラメータ修正・再実行指示

といった切り分けを行う。

RFA では、

再計画が必要となるポイントの構造化

を目的として活用する。

MTP（Multi-step Task Planning）

複数ステップにまたがるタスクを、
階層的に整理・表現するための枠組み。

CXC デモで扱う長尺・複合タスクを、

人間にとって理解しやすく

Agent 実装にも載せやすい

形に落とし込むための基盤技術として位置付ける。


### 6.3 Symbol Hub

Symbol Hub は、RFA における 「言語世界」と「物理世界」を接続する中間層である。
Task Planning Hub が扱う抽象的・記号的なタスク表現と、
Sensorimotor Hub が扱う連続値の物理表現の間を橋渡しする役割を担う。

本 Hub の設計意図は、
「ロボットが、自分はいま何をしていて、世界がどのような状態にあるのかを理解できるようにする」
ことにある。

#### 6.3.1 役割

意味状態の管理

Symbol Hub では、

何を（オブジェクト）

どこを（位置・領域）

どういう状態か（配置・接触・完了／未完了）

といった情報を、**記号的な意味状態（Symbolic State）**として管理する。

これにより、

タスクの進捗（どこまで完了したか）

現実世界の状態（実際に何が起きているか）

を同一の枠組みで扱うことが可能になる。

言語 ↔ 物理の変換（記号接地 / Symbol Grounding）

Task Planning Hub が扱う言語的・概念的表現を、
Sensorimotor Hub が扱える物理表現へと変換する。

例：

「重い箱」「壊れ物」「左側の棚」
→ 座標、Keypoint、領域マスク

「箱詰めが完了しているか？」
→ 画像・配置状態に基づく判定

逆に、Sensorimotor Hub から得られる観測結果を、

「緩衝材がまだ不足している」

「対象物が正しい向きで置かれていない」

といった LLM が理解可能な状態記述として Task Planning Hub に返す。

状態認識・成功／失敗判定

VLM（Vision-Language Model）を用いて、

現在の作業状態の認識

スキルの成功／失敗判定

次に実行すべきスキル候補の提示

を行う。

これにより、
ルールベースでは困難だった
「状況を見て判断する」制御を可能にする。

タスク構造と物理状態の同期

Task Planning Hub が保持するタスク構造（計画上の進捗）と、
実際の物理世界の状態（配置・接触・未完了要素）を突き合わせる。

これにより、

「計画上は終わっているが、実際は未完了」

「想定外の状態に遷移している」

といったズレを検知しやすくなる。

非定型・不確実な物理世界の中で、意図された操作を成立させることにある。

## 8. データ戦略・プラットフォーム連携

### 8.1 データ戦略の基本方針

RFAは「モデルを作って終わり」ではなく、
Robo Sync をハブとしたデータ循環によって、現場で使いながら賢くしていくロボット知能システムを目指す。

そのために、データ戦略として以下を基本方針とする。

複数ソースのデータを前提としたロボット基盤モデル（RFM）

テレオペ、シミュレーション、人間動画など、性質の異なるデータを前提に設計する。

Robo Sync を「デプロイ＋ログ収集」の統合インターフェースとみなす

モデルは必ず Robo Sync 経由で現場に出し、同じ経路でログを回収する。

SCM／製造ドメインに特化したデータ資産を蓄積する

一般的な家庭タスクではなく、梱包・入出庫・組立など、コネクトが勝負する領域のデータに集中する。

図「Our Data Strategy via Robo Sync」で示している通り、
左側の多様なデータソースをロボット基盤モデルに集約し、Robo Syncを通じて現場へデプロイしつつ、
再びデータを回収して継続的に改善する循環構造を前提とする。

### 8.2 ロボットAIデータソースの階層構造

図のピラミッドが示す通り、RFAが扱うロボットAIデータは、価値とスケーラビリティのバランスで3層に整理できる。

#### 8.2.1 Human Teleoperation（テレオペレーション）

位置づけ

最も貴重だがコストの高いデータソース。

熟練者の判断・力加減・リカバリー行動をそのまま含むため、「現場の暗黙知」が最も濃縮されている。

RFAでの役割

RFMの初期スキル獲得と高精度な Fine-tuning 用データとして利用する。

特に、柔軟物ハンドリングや微妙な挿入・嵌合など、シミュレーションでは再現が難しいスキルに対して重点的に収集する。

戦略的位置づけ

量より質を重視し、「どのタスクをテレオペで取るか」を明確に決めて投下する。

FY25〜26は、CXCデモおよびRobo Sync案件PoCに直結するタスクを優先する。

#### 8.2.2 Synthetic Data（シミュレーション／合成データ）

位置づけ

大量にスケール可能だが、Sim-to-Realギャップを持つデータ。

物理法則やオブジェクト配置のバリエーションを増やすのに向いている。

RFAでの役割

テレオペで得たコアスキルをシミュレーションで拡張・Augmentし、
物体配置・照明・ノイズなどのバリエーションを広げる。

RLや自動探索による「失敗パターンの網羅」を補う。

戦略的位置づけ

CECのクラウド基盤上でシミュレーション環境を整備し、
RFMの事前学習やロバスト化のためのバルクデータ生成装置として活用する。

#### 8.2.3 Human Videos（エゴセントリック動画など）

位置づけ

現場・家庭などで蓄積されている大量の人間動画。

直接ロボットコマンドは含まないが、「どう動くべきか」のヒントが豊富に含まれる。

RFAでの役割

Egocentric動画から物体・手のキーポイント・接触状態を抽出し、
RFMの事前学習や意味的表現（Symbol Integration）の土台として利用する。

戦略的位置づけ

すぐにロボット制御に直結するわけではないが、
長期的には「多様な人の動きから汎用スキル表現を学習する」ための重要なソースとして位置づける。

PRDCAとの共同研究、PHDと連携した全社での現場からのデータ収集活動を先行的に進める。

### 8.3 Robo Sync を中核としたデータライフサイクル

Robo Syncは、単なるロボット制御PFではなく、
「デプロイとデータ収集を一体化した現場インターフェース」として扱う。

RFAにおけるデータライフサイクルは、以下のように整理できる。

Model Deploy（Robo Sync → 現場）

RFMで学習したポリシーを、Robo Syncのシナリオ／ブロックとして組み込み、
生産ラインや倉庫の実タスクに適用する。

Logging（現場 → Robo Sync）

実行中のカメラ画像、ロボット状態、タスクシナリオ、成功／失敗ラベルなどを
Robo Sync側で一元的にログとして取得する。

Data Export（Robo Sync → RFAデータPF）

取得したログを、CECのクラウド基盤上のデータレイク／RFAデータストアに転送し、
RFM・LLM Agent が扱いやすいフォーマットに変換する。

Train / Fine-tune（RFA側）

Teleop / Simulation / Human Video と現場ログを統合し、
RFMの再学習・Fine-tuning、およびDRIP/MTPなどのプランナーの改善に用いる。

Re-Deploy & Continuously Improve

改善されたモデルを再度Robo Sync経由でデプロイし、
成功率・サイクルタイム・人の介入頻度などの指標で効果を評価しながら
継続的改善サイクルを回す。

このループを、FY25はCXCデモ環境、FY26以降は実案件PoCへとスケールさせていく。

### 8.4 クラウド／データプラットフォーム（CEC）との連携

CEC（Cloud Engineering Center）は、RFAのデータ戦略において以下の役割を担う。

トレーニング基盤

RFMやLLM Agentの学習・Fine-tuningを行うクラウドGPU環境を提供する。

データレイク／MLOps

Robo Syncから集約された現場ログ、テレオペデータ、シミュレーションログなどを
一元的に保管し、バージョン管理・検索・再利用を可能にする。

CI/CDパイプライン

モデル更新〜検証〜デプロイまでの一連のフローを自動化し、
「小さく頻繁に改善する」運用を支える。

RFAとしては、CEC側に「Robo Sync / RFM専用のデータ・モデル管理レーン」を用意し、
将来的には製造・SCMの他プロジェクトとも共有可能な共通基盤に発展させる
