## 議題1: 体制案（1PJ・2ストリーム）

### 1.1 提案内容

レビューの「テーマ分割・責任明確化」指摘への回答として、**完全分割ではなく1PJ・2ストリーム体制**を提案。

```
RFAプロジェクト（PM: 黄瀬）
│
├─ 共通基盤（両ストリーム横断）
│   ├─ DataOps / MLOps
│   ├─ RoboSync基盤・MS連携
│   ├─ 予算・プロジェクト管理
│   └─ エンビジョニング・現場連携（風見さん）
│
├─ Stream A: Robo Sync AI Suite（α版リリース）
│   ├─ リーダー: 奥村
│   ├─ メンバー: 桑田、鎌田
│   ├─ ミッション: Q2末（9月）CA リリース
│   ├─ 権限: α版スコープ内の技術判断・現場調整を自律的に決定
│   └─ 拠点: 京橋
│
└─ Stream B: Robo Sync Seeds（VLA/Agent技術深化）
    ├─ リーダー: 黄瀬（小栗?）
    ├─ メンバー: 小栗、Zhu、黒川、村上、加賀屋
    ├─ ミッション: β版以降のVLA事業価値実証・双腕複雑タスク
    └─ 拠点: 横浜
```

### 1.2 完全分割しない理由

| 課題 | 完全分割した場合のリスク |
|------|------------------------|
| 共通技術の重複 | DataOps/MLOps/RoboSync基盤が二重化 → コスト・工数増 |
| α版にStream B技術が必要 | VLAモデル・制御ポリシーなしにα版は成立しない |
| 管理コスト増大 | 予算・レビュー・成果物管理がすべて二重化 |
| コミュニケーションコスト | 同じ技術領域で分断されたチームの連携コストが増える |

**→ 分離するのは「管理単位」ではなく「責任・権限のオーナー」。** 共通基盤を共有しながら、α版のデリバリー責任が奥村さんに明確に移る。

### 1.3 体制で解決できること

| 課題（レビューFB） | 2ストリーム体制での解決策 |
|------------------|------------------------|
| 「何がいつ出るのか分からない」 | Stream A/Bそれぞれのアウトプットが明確化される |
| 「α版スピードが出ない」 | 奥村さんに権限が移り、京橋での意思決定がリアルタイム化 |
| 「1PMで全体を見るには広すぎる」 | 黄瀬はStream B+共通基盤に集中できる |
| 「奥村さんの育成」 | Stream AリーダーとしてPM経験を積む |

### 1.4 今日の合意ポイント

- [ ] 1PJ・2ストリーム体制で進めることの合意
- [ ] 奥村さんのStream Aリーダー受諾確認
- [ ] Stream A（奥村）の権限範囲の明確化：「α版スコープ内の技術判断・現場調整は奥村が自律決定」
- [ ] 「完全分割すべき」という判断が出た場合の次ステップ確認

## 議題2: α版MVP定義（叩き台）

### 2.1 α版（CA: Controlled Availability）— 9月末

> **ゴール:** RoboSync上でVLA制御ポリシーが動作し、形状・包装状態が複雑なパーツの **ピック → 作業台配置** を実現する。

#### スコープ

| 項目 | 内容 |
|------|------|
| **IN（やること）** | ビン内パーツのピック → 作業台に配置 |
| **OUT（やらないこと）** | 個装、梱包、台車移動、納品書処理、Agentic制御 |
| **対象パーツ** | 形状・包装状態が複雑な代表3〜5カテゴリ（現場選定） |
| **人の役割** | 個装・梱包・納品書処理は人が担当（人×ロボット協働） |
| **拠点** | 彩都パーツセンター（固定ステーション） |

#### 扱う複雑さ

| カテゴリ | 具体例 |
|---------|--------|
| 形状の多様性 | 天板（平面）、不定形パーツ、小型部品 |
| 包装状態の多様性 | プチプチ包装、袋入り、仕切り付きビン、段ボール内 |
| つかみにくさ | 立てかかっている、把持点が定まらない |
| 埋まっている | 段ボール内に埋まっている |

#### ハードウェア構成

| 項目 | 仕様 |
|------|------|
| ロボットアーム | UR5（単腕） |
| エンドエフェクタ | 吸着 / 平行グリッパ（タスクに応じて選択） |
| カメラ | Intel RealSense D405 |
| 移動機構 | なし（固定ステーション） |

### 2.2 異常系対応（大坪さん指摘：異常系含むCAが必要）

| 異常ケース | 対応方針 |
|----------|---------|
| ピック失敗（把持できない） | 1回リトライ → 失敗なら安全停止・人にエスカレーション |
| 落下・位置ずれ | 安全停止・異常ログ記録・人によるリカバリー |
| ビン空（パーツなし） | 検知して人に通知 |
| 異常なパーツ形状（対象外） | 対象外判定して人にエスカレーション |
| システム停止 | 安全停止・状態保存・リスタート手順の確立 |

→ **リカバリー手順書をα版と同時に整備する。**

### 2.3 成功基準・KPI（大坪さん指摘：KPI設計）

| KPI | 定義 | α版目標 | 計測頻度 |
|-----|------|---------|---------|
| ピック成功率（1回目） | 1回目での把持成功率 | **90%** | 毎実行 |
| リトライ込み成功率 | リトライ含む最終成功率 | **95%** | 毎実行 |
| タクトタイム | 1ピックの所要時間（秒） | 計測・記録のみ（β版で目標設定） | 毎実行 |
| 人介在頻度 | 人のリカバリーが必要な回数/100ピック | 計測・記録のみ | 週次集計 |
| ログ蓄積 | 実行ログが正常に記録される状態 | 稼働していること | 日次 |
| 安全性 | 物理的インシデント件数 | **0件** | 都度 |

**評価の回数設計（大坪さん指摘：1日の判断ではない評価が必要）**

| 時期 | 評価内容 |
|------|---------|
| 7月 | 第1回フォーマル評価（初期動作確認・KPI計測） |
| 8月 | 第2回フォーマル評価（修正後の精度確認） |
| 9月 | CA判定前の最終評価（KPI達成確認・異常系テスト） |

→ **少なくとも3回の定量評価を経てCA判定とする。**

### 2.4 β版（Q4末/3月）との境界

| 項目 | α版（CA/9月） | β版（3月） |
|------|-------------|-----------|
| 腕 | 単腕 | 双腕 |
| 対象工程 | ピック→配置のみ | 出庫サイクル全体（個装含む） |
| 制御 | VLA補完（ルールベース併用） | VLA主力 + Agentic制御 |
| 人との協働 | 人が残工程担当 | 自律サイクル運用（テレオペ介入あり） |
| KPI目標 | 成功率90%・タクトタイム計測 | タスク成功率TBD・継続運用可能 |
