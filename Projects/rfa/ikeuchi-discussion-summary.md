## 3. VLAの学習単位：GMR（Grasp-Manipulation-Release）

### 3.1 GMRとは

**GMR = Grasp – Manipulation – Release**（池内先生論文 arXiv:2008.01513, 2020）

ロボット操作タスクを3つの基本フェーズに分解した概念：

| フェーズ | 内容 | 例 |
|---------|------|-----|
| **Grasp（把持）** | 対象物を安定して掴む | コップの側面を掴む |
| **Manipulation（操作）** | 掴んだまま動かす | コップを移動、フタを回す |
| **Release（解放）** | 安全に手を離す | 棚の上に置いて離す |

**なぜGMRが重要か**：
- 日常・物流・製造作業の大半は「掴んで → 何かして → 離す」に分解できる
- **中間レベルの抽象表現**として、LLM/VLAとの相性が良い
- VLA Policyは **GMR単位** で分割するのが妥当

### 3.2 GMRの具体例（梱包タスク）

| GMR | 内容 | 手の使い方 |
|-----|------|----------|
| 商品ピッキング | コンテナから商品を取り出す | 片手 |
| 箱詰め配置（片手） | 商品を箱に入れる | 片手 |
| 箱詰め配置（両手） | 大きい商品を箱に入れる | 両手 |
| 箱閉め | フラップを折り畳む | 両手 |
| テープ貼り | 封緘テープを貼る | 両手 |

### 3.3 VLA × LLM/VLM の役割分担

```
┌─────────────────────────────────────────────┐
│ LLM / VLM                                   │
```

## 4. Robo Syncとの統合戦略

### 4.1 2つの統合ルート

| ルート | 内容 | 優先度 |
|--------|------|--------|
| **ルート①：Agent経由** | LLM AgentがMCP経由でRobo Syncに接続し、タスクシナリオを自動生成 | 将来（完全自律化） |
| **ルート②：VLAスキルブロック** | VLAをRobo Syncの「スキルブロック」として提供。GUIで配置・組み合わせ | **FY26最優先** |

### 4.2 なぜルート②が先か

- 既存のRobo Syncワークフローに自然に統合できる
- 現場エンジニアがGUIで操作可能 → 導入障壁が低い
- Agent統合は後から載せられる（疎結合設計）

### 4.3 VLAスキルブロックのイメージ

```
[Robo Sync GUI]
  │
  ├─ [従来ブロック] 位置決め移動
  │
  ├─ [VLAブロック] 商品ピッキング ← NEW
  │     └─ 入力: カメラ画像 + 言語指示
  │     └─ 出力: 軌道
  │
  ├─ [VLAブロック] 箱詰め配置 ← NEW
  │
  └─ [従来ブロック] 待機
```

## 5. スケール戦略：継続学習サイクル

### 5.1 RFAの差別化ポイント

> **「使いながら賢くなる」継続学習サイクルが差別化の核心**

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  現場で実行 → VLMが失敗検知 → リカバリー試行 → HILで人間教示  │
│       │                                            │     │
│       │         ┌───────────────┐                 │     │
│       └─────────│  データ蓄積   │←────────────────┘     │
│                 └───────────────┘                       │
│                        │                                │
│                        ▼                                │
│                 ┌───────────────┐                       │
│                 │  学習/Fine-tune │                      │
│                 └───────────────┘                       │
│                        │                                │
│                        ▼                                │
│                 ┌───────────────┐                       │
│                 │  再デプロイ    │ ────────────────────→ │
│                 └───────────────┘                       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 5.2 現実的なアプローチ

**池内先生との合意**：

| ポイント | 内容 |
|---------|------|
| **90%自動化 + 10%人間介入** | 最初から100%を目指さない |
| **Human-in-the-Loop** | 人間の介入データを学習に活用 |
| **失敗検知 → 人間 → データ収集** | 運用の中でデータが蓄積される設計 |

### 5.3 データ収集の実績（iREXデモより）

| 項目 | 数値 |
|------|------|
| 1スキルあたりのエピソード数 | 約200 |
| データ収集時間 | 約90分/スキル |
| Pi0ファインチューニング | 約1日/スキル |

→ **現実的な工数でVLAスキルを追加できる**

## 6. 事業的示唆

### 6.1 VLAが価値を出す領域

| 特性 | VLAの優位性 |
|------|------------|
| **少量多品種** | SKU変更のたびに再教示不要 |
| **動的状況変化** | 毎回違う状態に対応可能 |
| **属人的判断** | 暗黙知をデータから学習 |

→ **一品種大量生産の固定タスクは従来技術が有利。VLAは「毎回状況が変わるタスク」で真価を発揮**

### 6.2 最初のVLAブロック候補

**物流（梱包）**：
1. **商品ピッキング（STG11）** — 不定形物のピッキング
2. **箱詰め配置（STG13）** — 空間配置＋安定性

**製造（神戸工場PoC候補）**：
1. **キッティング** — 部品を指定位置に配置（P2P系）
2. **コネクタ挿入** — 精密挿入（Vector系、RLが必要かも）
3. **段ボール内箱組立** — 複雑だがニーズ高い

## 8. 池内先生からの重要な指摘（抜粋）

### スキルの汎化問題

> 「物体ごとにスキルを分けると扱う物が増えた時に問題。『Xを箱に入れる』という汎用スキル化が必要」

### エラーリカバリーの設計

> 「本来、アクションというのは何らかのアクションがあって、なおかつそのアクションが終わりましたという判定まで含めたものがアクション」
> 「すべてのアクションにおいて、エラーリカバリーが必要」

### VLAの過学習

> 「ファインチューニングで特定物体・動作に特化してしまう。プレトレインの汎化能力が失われる」

## 付録：接触状態分類（ICF / PTG / STG）

### ICF（接触状態）コード

| コード | 状態 |
|--------|------|
| NC | 非接触 |
| PC | 面接触 |
| PR/TR/OT | 線・点・その他の接触 |
| RV/NR | 回転拘束あり/なし |
| FT | 完全拘束に近い固定 |

### PTG（Physical Task Group）

純粋な物理制約に従うタスク：
- PTG11: Detach/Picking
- PTG12: Bringing
- PTG13: Attach/Placing
- PTG31/32/33: LinearOpen/Adjust/Close
- PTG51/52/53: RotateOpen/Adjust/Close

### STG（Semantic Task Group）

意味的制約を含むタスク：
- STG11/12/13: CarefullyDetach/Bring/Attach
- STG2: PlanarTranslate（テーブル拭きなど）
- STG3: LinearTranslate（皮をむく等）
- STG4: SphericalMove（お椀の内側を洗う等）
- STG5: Rotate（注ぐ等）
- STG6: Fix（まな板上で野菜を固定など）

→ **梱包タスクの多くはSTGに属する**（物理的には単純だが意味的に複雑）
