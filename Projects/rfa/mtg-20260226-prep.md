#### FB #1・#2・#3・#4 — 宮嶋さん「事業化スピード・顧客アウトカム」

**FB内容（全文）:**
> FY27で事業化判断とはどういう意味か？FY26中の事業化を目指す旗を降ろすということか？FY26もこれまでと同じやり方で技術研究をやっているように感じる。顧客に刺さるユースケースが見えない中、何を持ってα版、β版と言っているのか？PoTレベルではないのか？昨年秋から事業化の話が出ながら、何か取り組み方を変えたのか？今のやり方では賞味期限切れになのではないか。全般的に活動スピードが遅い感。顧客を中心に据えたアウトカムで目標を設定していない。単に世に何かを出すアウトプットで、技術開発を積み上げているだけになっていないか。

**分析:**
- 計画が「技術コンポーネントの開発」として構成されており、「いつ・誰に・何の価値を提供するか」が可視化されていない
- α版・β版の呼び方が、顧客価値ではなく技術成熟度の指標として使われている
- FY27に事業化判断を先送りしているように見えている
- 「昨年秋から事業化の話があるのに変化がない」という継続的な不満が蓄積している

**返答案:**
ご指摘の通り、現計画は技術コンポーネントの整理に偏っており、顧客価値の実証として設計し直す必要がある。

1. **α版の再定義:** 「技術デモ」から「彩都パーツセンターで実際のオペレーションの一部をロボットが代替し、省人化効果を定量計測できる状態」に変える。成功基準を技術指標だけでなく省人化効果（1時間あたり処理件数・人手削減率）で設定する。

2. **事業化判断を前倒し・明確化:** FY26 Q2（9月）時点でGo/No-Go判定を実施。「彩都PoCで省人化効果が定量的に示され、RoboSync顧客への提案ストーリーが成立するか」をGo条件とする。No-Goなら旗を降ろす覚悟を持つ。FY27に先送りしない。

3. **スピード改善:** 現場実証をQ3開始からQ1前倒しへ。技術が完成してから現場ではなく、現場実証と技術開発を並行で回す。

**→ 今日のMTGで確認すること:**
- Go/No-Go判定基準についてのレビュワーの期待レベルを大坪さんと合わせる
- 「旗を降ろす」の具体的な意味（プロジェクト終了？縮小？ピボット？）の定義

#### FB #5 — 松山さん「テーマ分割再検討」

**FB内容:**
> 取り組みが多岐に分かれて、アウトプットのコントロールが難しいと思います。テーマを分割して再検討を実施してはいかがですか？

**分析:**
- 「技術テーマが多い＝スピードが出ない・成果が分散する」という観点からの指摘
- 本日大坪さんへ1PJ・2ストリーム案を送付済み（返答待ち）
- 松山さんは「完全分割」を想定している可能性がある

**返答案:**
「テーマを分割して責任を明確にする」という方向性はその通りであり、**1PJ・2ストリーム体制**として提案済み。完全分割としない理由は以下の通り：

| 完全分割した場合のリスク | 2ストリームで解決 |
|------------------------|----------------|
| DataOps/MLOps/RoboSync基盤が二重化 | 共通基盤として一元管理 |
| α版にStream B（VLA）の技術が不可欠 | 両ストリームが共通技術を共有 |
| 予算・レビュー・管理が二重化 | 1PJとして一元管理 |
| 分断されたチームのコミュニケーションコスト増 | 同じPJ内でのストリーム分担 |

分離するのは「管理単位」ではなく「責任と権限のオーナー」。Stream A（奥村）がα版デリバリーに関する技術判断・現場調整を自律的に決定することで、スピードと責任の明確化を両立する。

**→ 今日のMTGで確認すること:**
- 松山さんが「完全分割」にこだわる場合、その根拠を確認し再議論
- 1PJ・2ストリームで合意できるかの確認

#### FB #6 — 松山さん「α版の定義が曖昧・スピードが出ない要因」

**FB内容:**
> α版の定義が曖昧であり、スピードが上がらない要因は何ですか？（スケジュールは再検討必要だと思います）

**分析:**
- α版の定義が技術コンポーネントのリストになっており、「何が動けばα版か」が明確でない
- スピードが出ない根本原因として、①スコープが広すぎる、②現場と技術開発が直列（技術完成を待ってから現場）、③責任者が分散している、が考えられる

**返答案:**
α版を以下に明確定義する（FB #7の大坪さん指摘と共通の対応）。

**α版（CA: Controlled Availability）9月末:**
- **What:** VLA制御ポリシーによる「ビン内パーツのピック → 作業台への配置」
- **スコープ外（MVPから除外）:** 個装、梱包、移動、Agentic制御
- **成功基準:** ピック成功率90%（対象パーツ）・異常系で安全停止・ログ蓄積稼働
- **スケジュール:** Q1から現場と並行スタート（Q3開始から前倒し）

スピードが出なかった主因：技術完成度を追いすぎた（ラボで完璧にしてから現場に持ち込む発想）。FY26は「動くものを現場に持ち込みながら改善する」スタイルに切り替える。

**→ 今日のMTGで確認すること:**
- α版定義の叩き台を松山さんにレビューしてもらい合意を得る
- スケジュール再検討の具体的な方針合意

#### FB #7 — 大坪さん「α版・β版定義・MVP・異常系・CA 9/E」

**FB内容:**
> α版・β版の定義が全体で必要。タスクをMVPに絞った上で異常系を含めてCAでは9/Eが必要。

**分析:**
- MVP（最小限の価値提供）に絞り込んだ上で、異常系も含めた完成形として9月末に出すことが求められている
- 「とりあえず動くものを出す」ではなく「スコープを絞り込んだ上で品質を確保して出す」が期待値

**返答案（α版MVP定義）:**

| 項目 | 内容 |
|------|------|
| **IN（やること）** | ビン内パーツのピック → 作業台に配置 |
| **OUT（やらないこと）** | 個装・梱包・移動・Agentic制御 |
| **対象パーツ** | 代表3〜5カテゴリ（現場選定）形状・包装多様なもの |
| **人の役割** | 個装・納品書処理は人が担当（協働） |

**異常系対応（CAに必要）:**

| 異常ケース | 対応 |
|----------|------|
| ピック失敗 | 1回リトライ → 安全停止 → 人にエスカレーション |
| 落下・位置ずれ | 安全停止 → 異常ログ記録 → 人によるリカバリー |
| ビン空 | 検知して人に通知 |
| 対象外パーツ | 対象外判定 → 人にエスカレーション |
| システム停止 | 安全停止 → 状態保存 → リスタート手順の確立 |

**→ 今日のMTGで確認すること:**
- α版MVP定義の叩き台を大坪さんにレビューしてもらい合意を得る
- α版・β版定義を独立ドキュメントとして3/10提出に含めることの合意

#### FB #10 — 松山さん「彩都倉庫の複雑さ・現場実証を急ぐ理由」

**FB内容:**
> 彩都倉庫はオペレーションが複雑ですが、現場実証を急ぐ理由はありますか？

**分析:**
- 「複雑な場所から始めなくても、もっと単純な場所から始めれば良いのでは？」という懸念
- 最初に難易度の高い現場に入ることのリスクを問われている

**返答案:**
彩都を選ぶ3つの理由：

1. **基盤が既にある:** RoboSyncが彩都で稼働しており、VLA統合のための実行基盤が整っている。他現場にゼロから入るより立ち上がりが速い。
2. **VLAの価値が最も発揮される環境:** 12万品番・多様な形状・包装状態がVLAの汎用性を試す理想的な環境。ルールベースが通用しないからこそVLAの意義が示せる。
3. **α版はスコープを絞る:** 複雑な全工程ではなく「ピック→作業台配置」のみに絞り込む（MVPアプローチ）。複雑なオペレーション全体を一度にやろうとするわけではない。

**→ 今日のMTGで確認すること:**
- この説明で松山さんの懸念が解消されるか確認
