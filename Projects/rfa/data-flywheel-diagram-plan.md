## フライホイールの核心メッセージ

**「Robo Syncが既に事業になっている → VLA追加で参入障壁が生まれる → スケールする」**

1. ルールベース（Robo Sync）+ VLAリカバリーで**現場導入障壁を下げる**
2. 失敗・カバー外の範囲で**人間オペレーションのデータが自然に集まる**
3. 現場データ蓄積 → **あらゆるタスクに拡張可能**
4. **事業スケールとデータスケールが両立**（使うほど強くなる）

---

## 図に含める要素

### 1. 現場レイヤー
- Robo Sync（主力・ルールベース）
- VLA（リカバリー役）
- 人介在（最終リカバリー）
- → 「不完全でも価値提供」がスタート地点

### 2. データ収集
- 成功データ（Robo Sync実行ログ）
- 失敗データ ★最高価値
- リカバリーデータ（VLA/人介在）
- → 収集コストほぼゼロ（運用の副産物）

### 3. 学習・改善サイクル
- Fine-tuning
- OOD領域の縮小
- モデル改善 → 再デプロイ

### 4. スケール効果
- 現場が増える → データ増える → モデル強くなる
- VLAリカバリー率向上 → 人介在頻度低下
- Robo Sync主力 → VLA主力へ段階的移行

### 5. 差別化ポイント（BigTech対比）
| | RFA (Deploy-Time Scaling) | BigTech (Training-Time Scaling) |
|--|---------------------------|--------------------------------|
| データ収集 | 現場運用の副産物 | 専用施設で収集 |
| コスト | ほぼゼロ | 数百億円規模 |
| データ種別 | 現場固有OOD、失敗含む | 管理環境、成功中心 |
| 価値提供 | 不完全でも即座に | モデル完成後 |
