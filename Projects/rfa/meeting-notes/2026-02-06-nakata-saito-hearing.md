# 彩都パーツセンター オペレーション詳細ヒアリング 議事録

**日時:** 2026年2月6日  
**参加者:** 黄瀬、中田真一（デジタルツイン部）、奥村亮、若杉勇作  
**目的:** 彩都パーツセンターのオートストア出庫作業の詳細理解、VLA/ロボットアーム導入可能性の検討  
**参考資料:** 彩都PC（as-is）業務プロセス分析結果 ver1.2（2023年5月25日、CPSアーキテクチャー研究部）

---

## 5行サマリー

1. **オートストア出庫はオーダーピッキング方式**：伝票1枚につき顧客オーダー1つが紐づいており、トータルピッキングではない（PMS起因）
2. **ロボット導入にはプロパティ登録が鍵**：入庫時にロボット把持可能フラグを立て、専用ビンに格納→専用ポートに流す制御が可能
3. **事前商品化（事前個装）が荷姿変動の原因**：時間に余裕があるうちに個包装を済ませておく運用があり、マスターデータと実際の荷姿が異なる
4. **WMSへの移行で改善の可能性**：現在のPMS（パーツマネジメントシステム）からBlue Yonder WMSへ移行中、トータルピッキング対応も可能に
5. **若杉さんのシミュレーションと連携可能**：倉庫全体のスループットシミュレーションとVLAの物理シミュレーションを連携させる構想

## 詳細サマリー

### 1. 会議概要

黄瀬が彩都パーツセンター見学後のフォローアップとして、中田真一さん（SCMソリューション部、オートストア導入経験者）にオペレーションの詳細をヒアリング。CPSアーキテクチャー研究部の業務プロセス分析結果（2023年）と照らし合わせながら、VLA適用に向けた技術的課題と対応策を整理した。

### 2. 彩都パーツセンターの全体像

#### 2.1 倉庫構成（4種類の保管エリア）

| エリア | 特徴 | 対象商品 | 備考 |
|--------|------|----------|------|
| **高稼働棚** | オートストア横の棚 | MSBD高稼働品（Aランク） | 需要大商品を配置、作業者移動あり |
| **一般棚** | 中物保管エリア | 中サイズ商品 | エリアごとに担当が分かれる |
| **オートストア** | AGVによる自動搬送 | 小物＆高稼働でない商品 | 作業者移動なし、ビン取り合いで遅延リスク |
| **ネステナー** | フォークリフトで運搬 | 大物（LCDパネル、梱包材、プロジェクターランプ等） | 入口右手の赤い棚 |

**配置基準:**
- **大きさ**: 小物→AS、中物→棚、大物→ネステナー
- **稼働頻度**: A,B,Cランク管理。A=高稼働、D=6カ月以上滞留
- **特殊品**: 全カテゴリーのバッテリーは指定棚で集中管理
- **年間不移動品**: 外部倉庫へ都度移動

#### 2.2 オートストア設備詳細

| 項目 | 仕様 |
|------|------|
| ビン数 | **2万ビン** |
| AGV台数 | **14台** |
| ビン段数 | **13段** |
| 充電 | 定期的に充電ステーションで充電 |

**運用特性:**
- 対象ビンより上にあるものは取り出して仮置き
- SAP（基幹システム）から事前にデータを送り、下段のビンを取り出すために上段に上げておく
- **頻度が低いものが必然的に下段に移動**する仕組み
- 入庫時に商品を入れすぎるとエラー（現在は充填率に気を付けて運用）

#### 2.3 導入済みソリューション

| ソリューション | 機能 |
|---------------|------|
| **積載量可視化** | カート・コンテナ・棚の充填率自動計測（上部カメラで計測） |
| **V-SLAM** | 棚移動誘導及び実績把握（QRコード読取）、ピッキング商品判定 |
| **ダッシュボード** | 進捗管理、生産性分析、動線分析、収支分析、在庫管理の統合可視化 |
| **AI画像処理** | 標準化作業プロセスに基づくリアルタイム作業分解・分析 |

### 3. 入庫プロセス

#### 3.1 入荷受入（1次検品）

**作業フロー:**
```
①入荷予定確認（週2回：水・金）
  ↓ 入荷数により作業者割り当て調整
②トラック荷卸し・検品
  ↓
③入出庫伝票番号より入荷伝票作成
  ↓
④2次検品へ
```

**入荷経路:**
- MEBD:AVS/オーディオ佐賀 → 彩都
- MEBD:佐賀/部品 → 大阪南港倉庫経由 → 彩都
- MEBD:門真/プロジェクター・フラットパネル → 大阪南港倉庫経由 → 彩都
- MSBD:神戸 → 彩都

**注意点:**
- 海外仕入れ分は現品票なしで入荷（後日門真材料管理より郵送）

#### 3.2 2次検品・棚入庫（光洋：請負）

**棚指定ルール:**
- 新規品番登録時に棚指定を実施
- 基本: 小物→AS、中物→棚、大物→ネステナー
- 特殊: MSBD高稼働品はASの横の高稼働棚に保管

#### 3.3 AS入庫

**特徴:**
- **1ビン複数商品保管可能**（ビン内配置も登録可能）
- **上積み保管あり**
- 同じ商品でも入荷タイミングにより別ビン格納
- 滞留6カ月以上の場合、ビン積載率向上のため1ビン複数商品格納を推進

**ビンの間仕切り:**
- ビンに溝があり、スリットを入れて棚分割が可能
- 分割ごとにビンIDが付与され、WMSと紐づく
- 細かすぎる分割はロボットアームのアクセス精度が課題

### 4. 事前商品化（事前個装）

**概要:**
時間に余裕があるうちに、個包装を済ませておく運用。作業者が暇な時に事前個装を実施する。

**対象:**
- システム側で管理されている対象パーツ
- PAV（プロAV）含む

**ロボット導入への影響:**
- 事前個装されると、プチプチ包装などで荷姿が変わる
- マスターデータは「標準荷姿」のサイズ → 実際と異なる
- 中身が見えない状態でバーコード読み取りで照合している
- **ロボット把持の難易度が上がる要因**

### 5. 出庫プロセス

#### 5.1 伝票発行（伝発）

**タイミング:**
- 売上伝票はAM8時伝発、9時には売上伝票棚に仕分け済み
- 商品カテゴリごとに伝発時間が指定されている（複数タイミングで発行）

**伝票の仕分け:**
- **エリア別**: 倉庫エリアごとに担当が分かれており、売上伝票はエリアごとに仕分け
- **時間帯別**: 遠方地（北海道、沖縄、九州）はトラック切り離しが早いため先に処理
- **行き先別**: 京橋行き、関東行きなど
- **保管場所別**: オートストア / 一般棚 / ネステナー

**伝票構成:**
- 売上伝票と個装ラベルがセットで保管
- 商品コードは個装ラベルに記載

#### 5.2 棚出庫

**作業フロー:**
```
①伝票棚から担当エリア分の売上伝票をピック
  ↓
②タブレットにログイン（作業者識別）
  ↓
③伝票のバーコードをスキャン → PMSに作業開始を通知
  ↓
④棚番号を確認し、該当棚へ移動
  ↓
⑤商品をピック、商品バーコードをスキャン（照合）
  ↓ ※違う商品を読み取るとエラー
⑥個装作業（袋入れ、輪ゴムくくり等）+ 個装ラベル貼付
  ↓ ※棚からピック時の数量入力は不要（PMS上で自動計算）
⑦次の伝票へ（繰り返し）
```

#### 5.3 AS出庫

**作業フロー:**
```
①売上伝票棚よりAS分を一括ピック
  ↓
②ポートでタブレットにログイン
  ↓
③伝票のバーコードをスキャン
  ↓
④AGVがビンをポートに搬送
  ↓
⑤商品をピック、商品バーコードをスキャン
  ↓ ※複数ビンからのピックあり
  ↓ ※上積み保管のため該当商品上に別商品存在の可能性
⑥個装作業 + 個装ラベル貼付
  ↓
⑦完了ボタン → 次のビンが流れてくる
  ↓
⑧全伝票処理完了 → 梱包エリアへ
```

**AS出庫の課題:**
- 商品ピック数により**複数ビンからのピック**あり
- **上積み保管**のため該当商品上に別商品が存在する可能性
- 他ポートと商品重複の場合**待ち時間発生**あり
  → 対策: 高稼働品は棚保管運用を導入

#### 5.4 オーダーピッキング方式

| 方式 | 特徴 | 彩都での採用 |
|------|------|-------------|
| **オーダーピッキング** | 顧客ごとにピッキング、動線が長い | ✓ 現在採用 |
| **トータルピッキング** | エリアごとに商品を集約、後で仕分け | △ WMS移行後に可能性 |

**PMSを使っている理由:**
- 彩都は「工場物流」の位置づけ（パソコン工場の出荷前倉庫が分離したもの）
- WMSではなくPMS（パーツマネジメントシステム）を使用
- Blue Yonder WMSへの移行プロジェクトが進行中（ラピュタASRS導入に合わせて部分導入の可能性）

### 6. 仕分け・個装・梱包

#### 6.1 仕分け

- 仕分け品数により作業者割当を都度実施
- 売上伝票内の商品内容及び出荷先を確認

#### 6.2 個装

- ピッキング時に流通加工（個装）を実施
- 商品と売上伝票を輪ゴムでくくってセット
- 特大品（75インチ以上のパネル）は専用作業場で個装
- 45-60インチは個装状態で入庫

#### 6.3 出荷梱包

**現状:**
- 別エリアの作業台で実施
- 箱サイズは作業者が選定（属人的）
- 梱包材（発泡スチロール、緩衝材）の使用もマニュアルベース
- 伝票と商品をバーコードスキャンして情報一致を確認
- 伝票を入れて封函、出荷

**課題:**
- 現在は属人的な判断が多い
- Amazon等は箱サイズ・テープ長さを自動化済み

### 7. ロボット導入に向けた技術的課題と対応策

#### 7.1 主要課題

| 課題 | 内容 | 影響度 |
|------|------|--------|
| **事前商品化による荷姿変動** | 個包装されるとマスターデータと実際の荷姿が異なる | 高 |
| **バーコード位置のランダム性** | 商品のバーコード位置が統一されていない | 高 |
| **仕切り内の複数SKU** | 1つの仕切りに複数SKU（最大3つ程度）が入る場合あり | 中 |
| **上積み保管** | 該当商品の上に別商品が存在する可能性 | 中 |
| **ピース/ボール/ケースの単位** | ネジ等は袋から1個取り出す場合あり | 中 |

#### 7.2 プロパティ登録による制御

**仕組み:**
1. 入庫時に「ロボット把持可能」フラグを立てる
2. WMS/PMSに商品のプロパティが登録される
3. 出庫時、ロボット専用ビンが専用ポートに流れるよう制御

**中田さんコメント:**
> 「入荷の方でプロパティ登録をしてあげることによって、ピッキング指示をかけた時に、その特定のロボットアーム用ポートにしか、その掴める商品が流れてこないという仕組みができる」

#### 7.3 導入ロードマップ案

**Phase 1: 簡単なものから**
- 個装なし、マスターデータ通りの荷姿の商品を対象
- ABCランク分けでロボット把持可能な商品を選別
- 専用ポートに流す制御を実装

**Phase 2: 複雑な荷姿への対応**
- 事前商品化されたものも把持できるよう拡張
- 置き方の最適化（充填率 vs ロボット把持しやすさのトレードオフ）

**レイアウト案:**
- ロボット専用ポートを設置
- ピッキング → 梱包まで一貫してロボットで処理
- コンベアで商品+伝票をセットで次工程へ

### 8. シミュレーション連携の可能性

**若杉さんの取り組み:**
- 出庫を想定した物理シミュレーション
- 彩都倉庫全体のシミュレーションモデルも開発中
- ロボット導入時の効果検証（スループット等）

**連携の可能性:**
- VLAの物理シミュレーション + 倉庫全体のマクロシミュレーション
- 彩都はデータ取得しやすく、連携に適している

## 参考：技術用語

| 用語 | 説明 |
|------|------|
| **PMS** | Parts Management System。パーツマネジメントシステム。工場物流向け |
| **WMS** | Warehouse Management System。倉庫管理システム。Blue Yonderへ移行予定 |
| **オートストア** | AutoStore。AGVがビンを自動搬送する自動倉庫システム |
| **ラピュタASRS** | Rapyuta社のAutomatic Storage and Retrieval System |
| **ネステナー** | 重量物保管用の棚。フォークリフトで運搬 |
| **GTP** | Goods to Person。商品が人のところに来るオペレーション |
| **プロパティ登録** | 入庫時に商品の属性（ロボット把持可能等）を登録すること |
| **SKU** | Stock Keeping Unit。在庫管理単位 |
| **個装** | 個別包装。袋入れ、輪ゴムくくり等 |
| **事前商品化** | 時間に余裕があるうちに個包装を済ませておく運用 |
| **伝発** | 伝票発行の略 |
| **光洋** | 請負会社名。2次検品・棚入庫を担当 |
| **MEBD/MSBD** | パナソニック社内の事業部コード |
