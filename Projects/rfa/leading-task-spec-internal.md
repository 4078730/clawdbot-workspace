### 2.2 α版の制御フロー

α版では **Sensorimotor Hub（VLA Policy）+ Robo Sync** のみ。Task Planning Hub、Agentic制御は不使用。

```
入力                         処理                          出力
─────                       ────                          ────
カメラ画像（RGB-D）  ──→  VLA Policy  ──→  6DoFアクション
  + ロボット関節角度         （推論）         （Δx,Δy,Δz,Δrx,Δry,Δrz）
  + 言語指示                               + グリッパ開閉指令
    「このパーツをピックして
     作業台に置いて」
        │
        ▼
    Robo Sync
    ・安全チェック（速度・力・範囲制限）
    ・UR5eへの指令送信
    ・実行ログ記録
```

#### VLAモデル（確定）

**α版採用モデル: π0 / π0.5 (Physical Intelligence)**

| モデル | 特徴 | α版での位置づけ |
|--------|------|---------|
| **π0** | Physical Intelligence社。Flow Matching + VLA。汎化性能が高い | **採用確定**。Fine-tuningベースライン |
| **π0.5** | π0の後継。マルチタスク・スケーラビリティ向上版 | **採用確定**。π0と並行評価 |

> **⚠️ 残り確認事項:**
> - π0/π0.5のAPI/モデルウェイトアクセス方法（要Physical Intelligence確認）
> - 推論頻度（目標10Hz以上）
> - **GPU: RTX 5070 Ti（VRAM 16GB）でπ0が推論可能か検証**（モデルサイズ次第ではA100検討）
> - Fine-tuning用データ形式（LeRobot形式との互換性確認）

#### VLA Policy の入出力仕様（α版）

| 項目 | 仕様 |
|------|------|
| **入力: 画像** | RGB 画像（Hand-Eye カメラ）、解像度 TBD（モデル依存、256×256〜640×480） |
| **入力: 深度** | Depth画像（RealSense D405から取得）。VLAモデルが深度を使うかはモデル依存 |
| **入力: ロボット状態** | 関節角度 6次元（q1〜q6）+ グリッパ状態 1次元 = 7次元 |
| **入力: 言語指示** | 自然言語テキスト（例: "Pick up the part wrapped in bubble wrap and place it on the green table"） |
| **出力: アクション** | TCP位置変位 3次元（Δx, Δy, Δz）+ TCP姿勢変位 3次元（Δrx, Δry, Δrz）+ グリッパ指令 1次元 = 7次元 |
| **出力: チャンク** | 将来N步のアクション列（Action Chunking）。チャンクサイズはモデル依存（典型: 10〜50ステップ） |
| **制御周期** | TBD（目標: 10Hz以上） |

### 2.3 ルールベースとVLAの切替条件

**設計原則:** VLAはルールベースの「置き換え」ではなく「段階的補完」。

```
                パーツ認識
                    │
            ┌───────┴───────┐
            │               │
     ルールベースで       ルールベースでは
     対応可能か？         対応不可
            │               │
            ▼               ▼
     ┌────────────┐   ┌────────────┐
     │ルールベース │   │ VLA Policy │
     │ピック       │   │ ピック      │
     │（既存）     │   │（新規）     │
     └────────────┘   └────────────┘
```

| 条件 | 制御方式 | 判定方法 |
|------|---------|---------|
| 箱物・整列済み・吸着面が明確 | ルールベース | 画像認識で物体カテゴリ判定 |
| 不定形・包装状態が複雑・把持点不明確 | VLA Policy | 上記以外（α版初期は全VLAも可） |
| VLA推論が異常値を返した場合 | **停止→人介在** | アクション値の範囲チェック |
| VLAが一定時間内にタスク完了しない場合 | **タイムアウト→人介在** | タイマー（閾値TBD） |

> **α版初期の方針:**
> 最初はVLAのみで動かし、性能データを蓄積する。
> ルールベースとの切替は、VLAの得意/不得意が明確になった段階で導入。

### 2.4 β版の制御フロー

β版では **Task Planning Hub（LLM Agent）** が加わり、8工程のオーケストレーションを行う。

```
LLM Agent（Task Planning Hub）
│
├─ [1] バーコードスキャン → スキャンスキル呼出
├─ [2] モニター操作      → UI操作スキル呼出
├─ [3] 段ボール開封       → 開封スキル呼出（双腕）
├─ [4] パーツピック       → VLA Policyスキル呼出
├─ [5] ビンに戻す        → 配置スキル呼出
├─ [6] 個装              → 個装スキル呼出（双腕）
├─ [7] 台車配置          → 配置スキル呼出
└─ [8] 次のオーダー      → サイクル管理

各スキルは Robo Sync 上のスキルブロックとして実装
スキル間の遷移は BT（ビヘイビアツリー）で定義
```

### 2.5 RoboSyncとの統合

#### スキルブロックの役割分担

| レイヤ | 責任 | 実装場所 |
|--------|------|---------|
| **タスク管理** | スキル選択・切替・再計画 | Task Planning Hub（β版） |
| **スキル実行** | VLA推論 → アクション生成 | VLAスキルブロック（RoboSync上） |
| **安全制御** | 速度制限・力制限・範囲制限・緊急停止 | RoboSync標準安全レイヤ |
| **ハード制御** | UR5eドライバ・EE制御・カメラ取得 | RoboSyncハード抽象化レイヤ |
| **ログ収集** | 画像・関節角・アクション・成功/失敗の記録 | RoboSync DataOpsレイヤ |

#### VLAスキルブロック インタフェース（α版）

```yaml
# VLA Skill Block Interface Definition (DRAFT)
skill_name: "vla_pick_and_place"
version: "0.1.0"

inputs:
  camera_image:
    type: "RGB"
    source: "realsense_d405_hand"
  depth_image:
    type: "Depth"
    source: "realsense_d405_hand"
  robot_state:
    type: "JointAngles"
    dimensions: 7  # q1-q6 + gripper
  language_instruction:
    type: "String"
    example: "Pick up the part and place it on the table"

outputs:
  action:
    type: "TCPDelta"
    dimensions: 7  # dx,dy,dz,drx,dry,drz,gripper
  confidence:
    type: "Float"
    range: [0.0, 1.0]

safety_constraints:
  max_tcp_velocity: 0.5  # m/s
  max_joint_velocity: 1.0  # rad/s
  max_force: 50  # N
  workspace_bounds:
    x: [-0.5, 0.5]  # m, relative to base
    y: [-0.5, 0.5]
    z: [0.0, 0.8]

failure_conditions:
  - type: "timeout"
    threshold_sec: 60
  - type: "action_out_of_range"
    description: "アクション値が安全範囲外"
  - type: "force_limit_exceeded"
    description: "力センサ閾値超過"
  - type: "confidence_low"
    threshold: 0.3

on_failure: "stop_and_notify"  # α版: 停止→人介在
```

### 2.6 リカバリー方法

| フェーズ | リカバリー方式 | 詳細 |
|---------|-------------|------|
| **α版** | **人が手動リカバリー** | ロボットが停止 → 人がパーツを正しい位置に配置 → ロボット再開 |
| **β版（前半）** | **テレオペ介入** | 遠隔からオペレーターが操作を引き継ぎ、タスクを完了 |
| **β版（後半）** | **自律リトライ + テレオペ** | VLAが再計画して2回目のトライ。2回失敗したらテレオペに移行 |

#### α版のリカバリーフロー

```
VLA実行中
    │
    ├─ 成功 → ログ記録 → 次のパーツへ
    │
    └─ 失敗検知（タイムアウト / 力超過 / 信頼度低）
         │
         ▼
    ロボット安全停止
         │
         ▼
    アラート通知（モニター表示 or 音声）
         │
         ▼
    人が現場に来てリカバリー
    ・パーツを正しい位置に配置
    ・ビン内を整理（必要時）
         │
         ▼
    「再開」ボタン押下
         │
         ▼
    ロボット再開 → 次のパーツへ
```

```
─────────────────────────────────────────────────────────────────────────────────────────────
```

#### 各ステップの時間見積もり（目標）

| ステップ | 動作 | 目標時間 |
|---------|------|---------|
| S0 | 初期位置移動 | 2秒 |
| S1 | ビン到着検知 | 1秒 |
| S2 | 認識 | 2秒 |
| S3 | 対象決定 | 1秒 |
| S4-S6 | アプローチ→把持→持ち上げ | 10秒 |
| S7 | 搬送 | 3秒 |
| S8 | 配置 | 3秒 |
| S9 | 配置確認 | 2秒 |
| S10 | 復帰 | 2秒 |
| **合計** | **1ピックサイクル** | **約26秒（目標）** |

> **参考:** 人間の作業者は1オーダー約1〜3分（複数パーツ含む）。
> α版の目標は「人と同等〜2倍の時間」で、速度より確実性を優先。

#### 困難なケース別の対応方針

| ケース | 状況 | VLAに期待する動作 |
|--------|------|------------------|
| **プチプチ包装** | パーツがプチプチに包まれてビン内にある | 包装ごと把持→持ち上げ。包装を剥がす操作はα版スコープ外 |
| **袋入り** | 小型部品が透明袋に入っている | 袋ごと把持。袋から出す操作はα版スコープ外 |
| **立てかかっている** | パーツがビン壁面に立てかかっている | 倒してから把持、または斜めからアプローチ |
| **段ボール内に埋もれている** | パーツが段ボールの奥にある | 上のパーツを退避→対象パーツにアクセス（困難な場合は人介在） |
| **複数パーツが密着** | パーツ同士が重なっている | 1個だけを分離して把持（分離困難な場合は人介在） |

### 3.2 β版：8工程の詳細プロセス

#### 工程[1] バーコードスキャン

```
入力:  俯瞰カメラ画像（納品書の位置検出）
処理:  ロボットがコードリーダーを把持 → 納品書上のバーコードに位置合わせ → スキャン実行
出力:  オーダー情報（品番・数量・個装指示）
技術:  ツール操作スキル（コードリーダーの把持と操作）、バーコード位置検出（画像認識）
課題:  バーコードリーダーの把持安定性、読み取り角度の調整
```

**伝票の実物仕様（現場確認済み）:**

| 項目 | 内容 |
|---|---|
| 伝票セット構成 | 3種類1セット：①売上計算票（経理用）②納品通知票・出荷明細票（仕入先元帳用）③入庫指図票・出庫票（商品元帳用） |
| バーコード種別 | 1次元バーコード（Code 128系）、各伝票右下に印刷 |
| スキャン対象 | 3枚のうち最上面の1枚のバーコードをスキャン（伝票番号・オーダー番号に対応） |
| スキャンで取得する情報 | 品番（例: ET-LAV300）、棚番号（例: B02A-C000）、数量、個装形態コード（例: NW1） |
| 伝票の置き方 | 作業台上にまとめて積まれた状態で到着。上面が読み取り対象 |

#### 工程[2] モニター操作（ビン呼び出し）

```
入力:  オーダー情報（工程[1]の出力）、モニター画面画像

```

#### 工程[3] ビン内段ボール開封（必要時）

```

出力:  段ボール開封完了、パーツが見える状態
技術:  双腕協調制御、力制御（段ボールを破壊しない程度の力）
課題:  段ボール形状の多様性（テープ有無、フラップの向き）、双腕VLAの難易度

```

#### 工程[4] パーツのピック（α版拡張）

```
入力:  ビン内画像、オーダー情報（品番・数量）
処理:  α版のピック動作を拡張
       ・指定品番のパーツを識別（12万品番の一部）
       ・指定数量をピック（「3個取り出す」→ 3回ピックサイクル）

出力:  指定パーツが作業台上に配置された状態
技術:  VLA Policy（α版ベース）+ 品番識別（VLM）+ カウント

```

```
入力:  ビン内・作業台の画像
処理:  取り出した後のビン内を整理。緩衝材や不要な梱包材をビンに戻す
出力:  ビンが元の状態に近い形で整理された
技術:  VLA Policy（整理・配置スキル）
課題:  「元の状態」の定義、壊れ物の取り扱い
```

#### 工程[6] 個装（袋入れ + 納品書同封 + 袋閉じ）

```
入力:  作業台上のパーツ、納品書（3枚セット）、個装用袋
処理:
  6a. 個装形態コード（伝票から取得）に基づいて適切なサイズの袋を選択
  6b. 片腕で袋を開き、もう片腕でパーツを入れる
  6c. 納品書（3枚セットのうち指定の1枚）を袋に同封
  6d. 袋を閉じる（輪ゴム or シーラー — 個装形態コードで指定）
出力:  個装済みパーツ（袋入り・納品書同封）
技術:  双腕協調制御、柔軟物操作（袋）、VLA Policy
課題:  袋を開く動作（柔軟物操作の高難度タスク）、輪ゴム留め、納品書の枚数管理
備考:  輪ゴム留めは特に高難度。β版で全工程を完遂できるかは検証次第
```

#### 工程[7] 台車/ワゴンに配置

```
入力:  個装済みパーツの画像、台車の空き位置
```

| 基準 | 条件 | 理由 |
|------|------|------|

## 6. 検証計画

### 6.1 α版の検証

#### 成功/失敗の定義

| 判定 | 定義 | 判定方法 |
|------|------|---------|
| **ピック成功** | 対象パーツをビンから持ち上げ、作業台の指定エリアに配置完了 | VLM判定 + 人目視確認 |
| **ピック失敗：把持ミス** | パーツに接触したが把持できなかった（滑り、吸着不足等） | 力/吸着圧センサ + カメラ |
| **ピック失敗：認識ミス** | 間違ったパーツをピックした、またはパーツを認識できなかった | カメラ画像の事後確認 |
| **ピック失敗：配置ミス** | ピックは成功したが、配置位置が不適切（台から落ちた等） | VLM判定 |
| **ピック失敗：タイムアウト** | 制限時間内にタスクを完了できなかった | タイマー |
| **ピック失敗：安全停止** | 安全制約違反で緊急停止した | RoboSync安全ログ |
| **破損** | パーツまたはビン/作業台を破損した | 人目視確認 |

#### 評価指標

| 指標 | 定義 | α版目標 | 計測方法 |
|------|------|---------|---------|
| **ピック成功率** | 成功回数 / 総試行回数 | **90%** | 自動ログ + 人確認 |
| **リトライ込み成功率** | リトライ含めた成功回数 / 総オーダー数 | **95%** | 自動ログ |
| **サイクルタイム** | 1ピックサイクルの所要時間（S0〜S10） | **30秒以内** | 自動計測 |
| **安全インシデント数** | 物理的な怪我・設備破損の件数 | **0件** | 報告ベース |
| **人介在頻度** | 人がリカバリー介入した回数 / 総試行回数 | 計測（目標は設定しない） | ログ記録 |
| **パーツカテゴリ別成功率** | カテゴリA〜D別の成功率 | 各カテゴリの傾向把握 | ログ分析 |

#### 評価環境

### 8.4 技術的不確実性の高い部分

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|---------|------|
| **VLA Fine-tuningで目標成功率に達しない** | 高 | 中 | 対象パーツを絞り込む。ルールベースとのハイブリッド比率を上げる |
| **現場環境（照明・振動等）でラボ性能が再現しない** | 高 | 中〜高 | Domain Randomization、現場データでのFine-tuning追加 |
| **双腕ロボットの調達が間に合わない** | 高 | 低〜中 | 早期発注（Q1）。間に合わない場合、ALOHA Stationaryをラボ検証で先行 |
| **個装タスク（袋開き・輪ゴム留め）のVLA制御が困難** | 中 | 高 | 個装をβ版の「達成度に応じて段階的に追加」する設計に変更 |
| **Agentic制御のBT設計が複雑化し安定しない** | 中 | 中 | 最小限のBT（2〜3スキル切替）から始めて段階的に拡張 |
| **テレオペデータの品質が低い** | 高 | 低 | データ収集ガイドライン策定、品質チェック基準の設定 |

## 付録A: 用語集

本ドキュメント内で使う主要用語。詳細は `glossary.md` を参照。

| 用語 | 説明 |
|------|------|
| **VLA** | Vision-Language-Action model。画像+言語+ロボット状態からアクションを生成 |
| **VLM** | Vision-Language Model。画像+言語で状態認識・判定を行う（アクション生成はしない） |
| **RoboSync** | ロボット制御PF。ハード抽象化・安全制御・シナリオ実行を担う |
| **スキルブロック** | RoboSync上でVLAスキルを実装・管理する単位 |
| **GMR** | Grasp-Manipulation-Release。把持→操作→解放のタスク構造化フレームワーク |
| **BT** | Behavior Tree。複数スキルの実行順序・条件分岐を定義する制御構造 |
| **EE** | エンドエフェクタ。ロボットアーム先端の把持装置（吸着パッド、グリッパ等） |
| **テレオペ** | テレオペレーション。人が遠隔でロボットを操作すること |
| **Action Chunking** | 将来N步のアクション列をまとめて予測する手法 |
| **Domain Randomization** | 学習データの環境条件（照明・色・位置等）をランダムに変化させる手法 |
| **CA** | Controlled Availability。限定的な利用可能状態（α版の位置づけ） |
