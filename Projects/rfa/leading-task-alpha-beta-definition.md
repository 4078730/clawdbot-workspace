## 1. リーディングタスクとは

### 1.1 概念

FY26において、RFAの技術価値と事業価値を**一本のタスクライン上で同時に実証する**ための中核タスク。

短期の成果（RoboSyncデプロイ実績）と中長期の成果（VLA事業価値の証明）を**別々のタスクに分散させず、同一ターゲットの難易度グラデーションとして設計する**ことで、すべての投資が累積的に価値を生む構造にする。

### 1.2 選定結果

**彩都パーツセンター出庫作業（オートストア出庫ピッキング）** をリーディングタスクとして選定。

### 1.3 選定理由（3軸評価）

| 評価軸 | 評価 | 根拠 |
|--------|------|------|
| **① RoboSync事業接続** | ◎ | RoboSync上にVLAスキルブロックとしてデプロイ可能。奥村さんがUR5テレオペを実現済みで、α版QMS定義を通す最初のユースケースとなる。一力さんチームとの協業体制も構築済み |
| **② VLA事業価値** | ◎ | 12万品番・多様な形状・包装状態のパーツが混在。動画分析の結果、吸着のみのルールベース制御では対応不可能と判明。**オペレーションが多様・複雑で標準化不可能 → VLAの汎用操作制御ポリシーが本質的に不可欠** |
| **③ エージェント自律性** | △→◎ | 難易度グラデーション設計により、α版（単一スキル: △）からβ版（複数スキル切替・再計画: ◎）まで段階的にエージェント自律性を引き上げ可能。RFA 3層アーキテクチャの存在意義を証明できる |

### 1.4 なぜ彩都ピッキングか — 他候補との比較

```
事業価値
  高 │ ③エアコン解体              ★①彩都ピッキング
     │                            ④段ボール組立
     │ ②-b ケーブル挿入
  中 │                            ⑤ピースピッキング
     │ ②-a コネクタ挿入
  低 │
     └──────────────────────────────────────→
      低              VLA難易度             高
```

**彩都ピッキングが唯一、短期軸（RoboSyncデプロイ）から中長期軸（VLA事業価値証明）まで、同一タスク内の難易度グラデーションでカバーできる。** 他候補はサイドラインとして位置づけ、メインラインの成果を横展開する構造とした。

## 3. α版タスク定義（CA: Controlled Availability — Q2末/9月）

### 3.1 ゴール

> RoboSync上でVLA制御ポリシーが動作し、形状・包装状態が複雑なパーツの **ピック → 作業台配置** を実現する。

### 3.2 スコープ

| 項目 | 内容 |
|------|------|
| **IN** | ビン内パーツのピック → 作業台に配置 |
| **OUT** | 個装、梱包、台車への移動、納品書処理、Agentic制御 |
| **対象パーツ** | 形状・包装状態が複雑なパーツ（RoboSyncチームと選定） |
| **人の役割** | 個装・梱包・納品書処理は人が担当（人×ロボット協働） |

### 3.3 α版が扱う複雑さ

| 課題 | 具体例 |
|------|--------|
| **形状の多様性** | 天板（平面パーツ：把持点が一意に定まらない）、不定形パーツ、小型部品 |
| **包装状態の多様性** | 段ボール内のプチプチ包装、袋入り、仕切り付きビン |
| **つかみにくさ** | 立てかかっている、掴む場所が決めにくい |
| **埋まっている** | 段ボールの中に下に埋まっている |

→ **VLAによる操作制御ポリシー**が、形状・状態に応じた把持・取り出し動作を生成

### 3.4 ハードウェア構成

| 項目 | 仕様 |
|------|------|
| **ロボットアーム** | UR5（単腕） |
| **エンドエフェクタ** | 吸着 / 平行グリッパ（タスクに応じて選択） |
| **指** | 使用しない（指の制御自体が研究対象） |
| **カメラ** | Intel RealSense D405 |
| **移動機構** | なし（固定ステーション） |

### 3.5 技術要件

| # | 要件 | 詳細 |
|---|------|------|
| T-1 | VLA制御ポリシー | 形状・包装状態が複雑なパーツに対して、操作軌道を生成・実行 |
| T-2 | RoboSync統合 | VLAスキルブロックとしてRoboSync上で実装・実行・管理 |
| T-3 | ログ収集 | 実行ログ（画像・ロボット状態・成功/失敗）の記録 |
| T-4 | 安全制御 | RoboSync標準安全制約に準拠 |
| T-5 | ルールベース連携 | ルールベースで対応可能なケースはルールベースで処理、VLAは困難なケースを補完 |

※ Agentic要素（複数スキル切替、自律的再計画等）はα版には含まない

### 3.6 成功基準

| 指標 | 目標 | 備考 |
|------|------|------|
| ピック成功率 | 90% | 対象パーツを絞った上での成功率 |
| リトライ込み成功率 | 95% | 1回失敗後のリトライを含む |
| ログ蓄積 | 稼働 | 実行ログが記録される状態 |
| 安全性 | インシデント0 | 物理的な怪我に繋がる事故はゼロ |

## 4. β版タスク定義（Q4末/3月）

### 4.1 ゴール

> 固定ステーション内の出庫作業サイクル全体をカバー。**Agenticな制御**で複数スキルを切り替えながらサイクルを回す。

### 4.2 スコープ

| 項目 | 内容 |
|------|------|
| **IN** | 固定ステーション内の出庫作業サイクル全体 |
| **OUT** | 出荷用の梱包作業、ステーション外への移動 |
| **個装** | ✅ 含む（袋入れ・納品書同封・袋閉じ） |
| **台車配置** | ✅ 含む（作業空間内の姿勢変化） |
| **Agentic制御** | ✅ 含む（複数スキル切替・サイクル管理に必須） |
| **対象パーツ** | α版で扱った範囲を多品種に拡張 |

## 5. α → β の難易度グラデーション

### 5.1 複雑さの3軸での変化

| 軸 | α版 | β版 | 何が複雑化するか |
|----|------|------|-----------------|
| **手** | 吸着/平行グリッパ | 同左 | 変化なし（指はGA以降） |
| **腕** | 単腕・VLA制御 | **双腕**・個装・段ボール開封 | 腕間協調、片方で押さえて片方で操作 |
| **移動** | なし（固定） | 作業空間内の姿勢変化 | 台車への配置でリーチ・姿勢が変化 |
| **Agentic** | なし | **サイクル全体のスキル切替・管理** | 8工程のオーケストレーション、異常検知と再計画 |

### 5.2 段階設計の全体像

```
  α版（Q2末/9月）                β版（Q4末/3月）
  ──────────────                ──────────────
  単一スキル                     複数スキル統合
  ピック→配置のみ                出庫サイクル全体
  単腕（UR5）                    双腕
  ルールベース＋VLA補完           VLA主力＋Agentic制御
  人×ロボット協働                自律サイクル運用

          難易度グラデーション
  ├─────────────┼─────────────┤
  低            中            高

  ①-a         ①-b          ①-c         ①-d
  多様形状     梱包材        回避+        フルAgentic
  パーツ       取り出し      リカバリー    （β版相当）
  ピック       多品種対応    状況適応      個装・切替
```

### 5.3 具体的に何が複雑化するか

| 次元 | α版 | β版 | 技術的ジャンプ |
|------|------|------|--------------|
| **対象物** | 選定されたパーツ | 多品種（12万品番の一部） | 汎化能力の実証 |
| **操作** | ピック＆プレース | ピック＋個装＋配置＋開封 | 操作種類の爆発的増加 |
| **判断** | 把持戦略のみ | 商品識別→手順選択→異常対応 | LLM Agentの介入が必須に |
| **制御構造** | 1GMRの実行 | 複数GMRの連鎖・切替 | BT（ビヘイビアツリー）によるオーケストレーション |
| **ハードウェア** | 単腕＋固定 | 双腕＋姿勢変化 | 双腕協調制御、ハードウェア選定 |
| **失敗対応** | 人が手動リカバリー | 自律リトライ＋テレオペ介入 | リカバリースキルの追加 |

## 7. 池内先生へのディスカッション論点

### 7.1 GMRフレームワークとの接続

α版のタスクは **1つのGMR（Grasp → Manipulation → Release）** として定義できる：

| GMRフェーズ | α版での対応 |
|------------|------------|
| **Grasp** | ビン内の多様なパーツを把持（形状・包装状態に応じた把持戦略） |
| **Manipulation** | ビンから持ち上げて作業台まで移動 |
| **Release** | 作業台上に安全に配置 |

β版では **複数のGMRが連鎖する構造** となる：

| 工程 | GMR構造 | 備考 |
|------|---------|------|
| 段ボール開封 | GMR（押さえ→開封→離す）| 双腕で別々のGMR |
| パーツピック | GMR（把持→移動→配置） | α版の拡張 |
| 袋入れ | GMR（袋を開く）+ GMR（パーツを入れる） | 双腕協調 |
| 袋閉じ | GMR（袋を掴む→閉じる→離す） | 柔軟物 |

**論点①:** α版の「ピック→配置」を1GMRとして定義する際、VLA Policyの粒度は「1 GMR = 1 Policy」の仮説と整合するか？特に、ビン内の複雑な状態（プチプチ包装、埋まっているパーツ）への対応は、1つのPolicyで吸収できるか、あるいはGrasp前の「前処理GMR」として分離すべきか。

### 7.2 ICF/PTG/STGとの接続

α版のピッキングタスクのPTG/STG分類：

| 工程 | 分類 | コード | 説明 |
|------|------|--------|------|
| ビンからのピック | PTG | **PTG11 (Detach/Picking)** | 物理的な分離・ピッキング |
| 作業台への配置 | PTG | **PTG13 (Attach/Placing)** | 配置 |
| 緩衝材からの取り出し | STG | **STG11 (CarefullyDetach)** | 慎重な分離（壊れ物を傷つけない） |

**β版で加わるSTG領域：**

| 工程 | 分類 | コード | 説明 |
|------|------|--------|------|
| 袋入れ | STG | STG13 (CarefullyAttach) | 慎重な配置（向きや位置の意味的制約） |
| 袋閉じ | STG | — | 柔軟物操作（STGの新しい領域） |
| 商品識別→手順選択 | — | Semantic Constraint | 常識的判断（物理制約では表現不可能） |

**論点②:** α版の段階でも、ビン内のパーツは「ただ掴む」のではなく、包装状態に応じた慎重さが求められる。これはPTG11（物理的なDetach）とSTG11（CarefullyDetach）の境界にある。VLA制御ポリシーは、この物理制約と意味的制約の両方を暗黙的に学習する能力を持つと期待されるが、それを**明示的にICFの遷移として構造化すべきか、VLAの暗黙的学習に委ねるべきか**。

### 7.3 IndexVLAの実証タスクとしての位置づけ

IndexVLAは「LLMとVLAの通訳」として、言語指示と視覚情報から**VLAが実行可能な中間表現（Index）** を生成する。

```
                        IndexVLA の役割
  ┌──────────┐     ┌──────────────┐     ┌──────────┐
  │  LLM     │────→│  IndexVLA    │────→│  VLA     │
  │  Agent   │     │  (通訳層)    │     │  Policy  │
  │          │     │              │     │          │
  │ 「この   │     │ Keypoint     │     │ 操作軌道 │
  │  パーツを│     │ Trajectory   │     │ を生成   │
  │  取って」 │     │ Vector       │     │          │
  └──────────┘     └──────────────┘     └──────────┘
```

**α版での検証ポイント：**
- ビン内の多様なパーツに対して、IndexVLAが適切な把持点・軌道のIndexを生成できるか
- 包装状態の違い（プチプチ、袋、段ボール内）をIndexとして区別できるか

**β版での検証ポイント：**
- 複数スキル間の切替時に、IndexVLAが「次に何をすべきか」のコンテキストを保持できるか
- 商品の種類に応じた個装手順の選択をIndex空間で表現できるか

**論点③:** 彩都ピッキングのα版は、IndexVLAの「最初の本格的な実証タスク」としてどう位置づけるべきか。特に、12万品番という多様性は、IndexVLAの**汎化能力を検証する理想的な環境**だが、最初の検証ではどの程度品番を絞るべきか。

### 7.4 RFA 3層アーキテクチャとの対応

```
  RFA 3層              α版での役割              β版での役割
  ─────────            ─────────              ──────────
  Task Planning        （なし）                 サイクル全体の
  Hub (LLM Agent)                              オーケストレーション

  Symbol Hub           （限定的）               IndexVLAによる
  (IndexVLA)           把持Index生成            スキル切替Index生成

  Sensorimotor Hub     VLA制御ポリシー          複数VLAスキルの
  (VLA Policy)         ピック→配置             個装・開封・配置
```

**論点④:** α版ではTask Planning Hubを使わず、β版で初めて導入する設計としている。池内先生の方針4（BTベースのプランニング）との整合性として、β版のAgentic制御をBT（ビヘイビアツリー）で設計する場合、[1]〜[8]の8工程はどのようなBT構造で表現するのが適切か。

### 7.5 ディスカッションで確認したいこと（まとめ）

| # | 論点 | 関連する池内先生の理論 |
|---|------|----------------------|
| 1 | α版「ピック→配置」のGMR粒度は適切か。前処理GMR分離の要否 | GMR、VLA Policy = 1 GMR仮説 |
| 2 | PTG11 vs STG11の境界をVLAの暗黙的学習に委ねるか、明示的に構造化するか | ICF/PTG/STG |
| 3 | IndexVLAの実証として、初期検証での品番絞り込み戦略 | IndexVLA、汎化 |
| 4 | β版のAgentic制御のBT設計方針 | 方針4（BT）、Task Planning Hub |
| 5 | α→βの拡張において、単腕→双腕のGMR構造の違い（双腕GMRの定義） | GMR、双腕協調 |
