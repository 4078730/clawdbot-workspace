## 0. 本ドキュメントの目的

**「自分たちが何を作って、どう検証するか」を全員が同じイメージで理解する。**

- 本ドキュメントは**プロジェクトメンバー全員（経営・PM・エンジニア）が同じ認識を持てる粒度**で、対象業務・フェーズ定義・ハードウェア・制御・プロセス・検証を具体化する
- 未確定事項は「TBD」として明示し、決定待ちの項目を可視化する
- 技術的な詳細（制御仕様・入出力仕様等）はエンジニアがすぐ動ける粒度で記載する

## 1. 対象業務

### 1.1 現場概要

| 項目 | 内容 |
|------|------|
| **拠点** | 彩都パーツセンター |
| **業務** | オートストア出庫作業 |
| **規模** | 12万品番、月間3万点出荷 |

| **付加価値作業比率** | 30%（業界トップ水準） |

### 1.2 出庫作業フロー（動画分析 2026-02-16）

```
[1] 準備・登録
    納品書の束を作業台に持ってくる → モニター操作で作業者登録

[2] スキャン・チェック                   ← β版の対象範囲
    バーコードリーダーで納品書を1枚ずつスキャン（〜10枚/束）
    ペンでチェック記入

[3] ビン到着・ピック                     ← α版の対象範囲
    オートストアからビンが右側のポートに到着
    ビンは仕切りで4区画に分かれており、各区画に種類ごとのパーツが格納されている
    納品書に対応するパーツをピックし、作業台に配置

[4] 個装（物による）                     ← β版の対象範囲
    パーツと納品書を適切なサイズの袋に同封
    袋を閉じ、左側のワゴンに配置

[5] 次のオーダーへ
    1オーダー: 約1〜3分
```

![出庫作業フロー — Picking Workflow](assets/workflow-saito.png)

### 1.3 α版が扱う複雑さ

α版では「オペレーションの標準化が不可能」な部分はスコープ外とし、
**商品の形状・包装状態の複雑さに対するVLA制御の有効性**を実証する。

#### α版が扱う課題

| 課題 | 具体例 |
|------|--------|
| **形状の多様性** | 天板（平面パーツ：端の把持点が一意に定まらない）、不定形パーツ、小型部品 |
| **包装状態の多様性** | 段ボール内のプチプチ包装、袋入り、仕切り付きビン |
| **取り出しにくさ** | 立てかかってつかみにくい、一度で掴めない、下に埋まっている |

> → VLAによる操作制御ポリシーで、形状・状態に応じた把持・取り出し動作を生成する

#### α版のスコープ外（β版以降）

- 個装（袋入れ・納品書同封）
- 梱包（段ボール開封含む）
- 移動（台車配置）
- 複数スキル切替のAgenticな制御

## 2. フェーズ定義

### 2.1 全体構造 — 複雑さの3軸で段階的に拡張

**設計原則:** VLAはルールベースの「置き換え」ではなく「段階的補完」。
ルールベースで対応可能なものはルールベースで処理し、VLAが困難なタスクを補完。
複雑さを3軸に分け、時間的に分離して取り組む。

| 軸 | α版（CA） | β版 | GA（参考） |
|----|----------|-----|----------|
| **手** | 吸着/平行グリッパ（手動交換） | 同左 + 自動チェンジャー検討 | 指の制御（研究成果統合、GA オプション） |
| **腕** | 単腕・VLA制御 | 双腕・Agentic制御・個装/段ボール開封 | 双腕高度化（GA オプション） |
| **移動** | なし（固定ステーション） | TBD（台車配置まで） | 移動ロボット（GA オプション） |

### 2.2 α版（CA: Controlled Availability）— Q2末/9月

**ゴール:** RoboSync上でVLA制御ポリシーが動作し、形状・包装状態が複雑なパーツの ビン内ピック → 作業台配置を実現する

#### スコープ

| 項目 | 内容 |
|------|------|
| **IN** | ビン内パーツのピック → 作業台に配置 |
| **OUT** | 個装、梱包、台車への移動、納品書処理、Agentic制御 |
| **対象パーツ** | 形状・包装状態が複雑なパーツ（具体ケースはRoboSyncチームと選定） |
| **人の役割** | 個装・梱包・納品書処理は人が担当。リカバリー作業も人が実施（人×ロボット協働） |

> **人の介在について:** ロボットのピック失敗・異常時には人が手動でリカバリーを行う。
> β版以降ではテレオペ介入や操作ログ記録によるリカバリーデータ蓄積を目指すが、
> α版では開発工数を考慮し手動リカバリーとする。

#### 成功基準（仮）

*期初時点の目安。現場・実際のタスク・技術レベルに合わせて調整。*

| 指標 | 目標 | 備考 |
|------|------|------|
| ピック成功率 | **90%** | 対象パーツを絞った上での成功率 |
| リトライ込み成功率 | **95%** | 1回失敗後のリトライを含む |
| ログ蓄積 | 稼働 | 実行ログが記録される状態 |
| 安全性 | **インシデント0** | 物理的な怪我に繋がる事故は絶対にゼロ |

#### 技術要件

| # | 要件 | 詳細 |
|---|------|------|
| T-1 | VLA制御ポリシー | 形状・包装状態が複雑なパーツに対して操作軌道を生成・実行 |
| T-2 | RoboSync統合 | VLAスキルブロックとしてRoboSync上で実装・実行・管理 |
| T-3 | ログ収集 | 実行ログ（画像・ロボット状態・成功/失敗）の記録 |
| T-4 | 安全制御 | RoboSync標準安全制約に準拠。物理的な怪我につながる危険箇所は絶対に対処 |
| T-5 | ルールベース連携 | ルールベースで対応可能なケースはルールベースで処理。VLAは困難なケースを補完 |

### 2.3 β版 — Q4末/3月

**ゴール:** 固定ステーション内の出庫作業サイクル全体（工程[1]〜[8]）をAgentic制御で回す。「ちゃんと動くこと」が最優先。

#### スコープ

| 項目 | 内容 |
|------|------|
| **IN** | 固定ステーション内の出庫作業サイクル全体（工程[1]〜[8]） |
| **OUT** | 出荷用の梱包作業、ステーション外への移動 |
| **Agentic制御** | 含む（複数スキル切替・サイクル管理に必須） |
| **対象パーツ** | α版で扱った範囲を多品種に拡張 |

#### 技術要件

| # | 要件 | 詳細 |
|---|------|------|
| T-6 | Agentic制御 | 複数スキルの切替・異常検知・再計画。サイクル全体のオーケストレーション |
| T-7 | 段ボール開封 | ビン内段ボールの封を開ける制御（双腕・力制御） |
| T-8 | 個装スキル群 | 袋入れ・納品書同封・袋閉じ・輪ゴム留め（双腕） |
| T-9 | ビン内操作 | 緩衝材からの取り出し、残りのビンへの戻し |
| T-10 | システム連携 | バーコードスキャン、モニター操作の自動化 |
| T-11 | リカバリー | テレオペ介入・UMI操作ログ記録の導入検討 |
| T-12 | 現場固有安全 | 異常時自動停止、人介在データ記録フロー |

*現時点では未確定。商用展開を見据えた方向性として記載。*

## 4. 制御方法

### 4.1 ソフトウェアアーキテクチャ全体像

```
┌──────────────────────────────────────────────────────────────────┐
│                        RFA 制御スタック                          │
│                                                                  │
│  ┌─────────────────┐                                            │
│  │ Task Planning    │  β版で導入。α版では不使用                  │
│  │ Hub (LLM Agent)  │  工程[1]〜[8]のオーケストレーション        │
│  └────────┬────────┘                                            │
│           │ スキル選択・パラメータ指示                            │
│           ▼                                                      │
│  ┌─────────────────┐     ┌──────────────────┐                   │
│  │ Sensorimotor    │     │ ルールベース制御  │                   │
│  │ Hub (VLA Policy)│     │ （既存RoboSync）  │                   │
│  └────────┬────────┘     └────────┬─────────┘                   │
│           │                       │                              │
│           └───────┬───────────────┘                              │
│                   │ 切替: VLA or ルールベース                     │
│                   ▼                                              │

│  └─────────────────────────────────────────┘                     │

│                   ▼                                              │

└──────────────────────────────────────────────────────────────────┘
```

![RFA Control Stack — α/β Architecture](assets/control-stack-architecture.png)

### 4.2 α版の制御フロー

α版では **Sensorimotor Hub（VLA Policy）+ Robo Sync** のみ。Task Planning Hub、Agentic制御は不使用。

```
入力                         処理                          出力
─────                       ────                          ────
カメラ画像（RGB）    ──→  VLA Policy  ──→  関節角度制御
  640×480 / 30fps            （推論）         （q1〜q6）
  + ロボット関節角度                        + グリッパ開閉指令
  + パーツ情報（PMS）
    パーツ名・品番・
    ビン内ピック位置
        │
        ▼
    Robo Sync
    ・安全チェック（速度・力・範囲制限）
    ・UR5eへの指令送信
    ・実行ログ記録
```

#### VLA Policy の入出力仕様（α版）

| 項目 | 仕様 |
|------|------|
| **入力: 画像** | RGB 画像（Hand-Eye カメラ）/ **640×480 / 30fps** / 深度は使用しない |
| **入力: ロボット状態** | 関節角度 6次元（q1〜q6）+ グリッパ状態 1次元 = 7次元 |
| **入力: パーツ情報** | PMSから取得するパーツ仕様（パーツ名・品番・ビン内ピック位置 ※仕切りがある場合）。言語指示ではなくパーツ仕様書として読み込む |
| **出力: アクション** | **関節角度制御** — 各関節の目標角度 6次元（q1〜q6）+ グリッパ指令 1次元 = 7次元 |
| **制御周期** | **10Hz**（確定） |

### 4.3 β版の制御フロー

β版では **Task Planning Hub（LLM Agent）** が加わり、8工程のオーケストレーションを行う。

```
LLM Agent（Task Planning Hub）
│
├─ [1] バーコードスキャン → スキャンスキル呼出
├─ [2] モニター操作      → UI操作スキル呼出
├─ [3] 段ボール開封       → 開封スキル呼出（双腕）
├─ [4] パーツピック       → ピックスキル呼出
├─ [5] ビンに戻す        → 配置スキル呼出
├─ [6] 個装              → 個装スキル呼出（双腕）
├─ [7] 台車配置          → 配置スキル呼出
└─ [8] 次のオーダー      → サイクル管理

各スキルは Robo Sync 上のスキルブロックとして実装
スキル間の遷移は BT（ビヘイビアツリー）で定義
```

### 4.4 RoboSyncとの統合

| レイヤ | 責任 | 実装場所 |
|--------|------|---------|
| **タスク管理** | スキル選択・切替・再計画 | Task Planning Hub（β版） |
| **スキル実行** | VLA推論 → アクション生成 | VLAスキルブロック（RoboSync上） |
| **安全制御** | 速度制限・力制限・範囲制限・緊急停止 | RoboSync標準安全レイヤ |
| **ハード制御** | UR5eドライバ・EE制御・カメラ取得 | RoboSyncハード抽象化レイヤ |
| **ログ収集** | 画像・関節角・アクション・成功/失敗の記録 | RoboSync DataOpsレイヤ |

## 5. 対象物体の定義

### 5.1 パーツカテゴリ分類

> **⚠️ 暫定分類（確定版は風見さん提供の動画をもとに更新予定）**
> 以下はネット調査・伝票情報・既存動画から推定した暫定カテゴリ。α版対象パーツの絞り込みに使用する。

**確認済み品番例:** ET-LAV300（プロジェクター用ランプ）/ 個装形態 NW1

| カテゴリ | 品目例（パナコネ補修部品） | 形状特徴 | 包装例 | VLA難易度 | α版対象 |
|---------|---------|---------|--------|----------|---------|
| **A: ランプ・光学部品** | プロジェクターランプ(ET-LAV系)、レンズユニット | 細長い or 円柱形。ガラス素材含む | 専用袋 or 個装箱 | 中 | ✅（要慎重把持） |

| **C: 箱物・ユニット** | 電源ユニット、基板ユニット、リモコン | 直方体・箱形 | 段ボール小箱 or 袋 | 低 | ✅ |
| **D: ケーブル・フレキ** | 各種ケーブル、FFC、ハーネス | 柔軟・不定形 | 袋入り | 高 | ✅（袋ごと把持） |
| **E: 小型部品** | ネジ、ボルト、コネクタ、スイッチ | 小さい（<50mm）| 小袋入り | 中 | ✅（袋ごと把持） |
| **F: ゴム・シール類** | パッキン、O-リング、クッション | 軟質・変形しやすい | 袋入り | 高 | △（形状次第） |
| **G: 重量物** | モーター、トランス、コンプレッサー | 5kg超 | 段ボール+プチプチ | — | ❌（ペイロード超過） |
| **H: 超大型** | 室外機パネル、筐体部品 | ビンに収まらない | — | — | ❌（スコープ外） |

> **次アクション:** 風見さんより現場動画を入手し、実際の出庫品番・形状・包装を確認してカテゴリを確定する

### 5.2 α版の対象パーツ選定基準

| 基準 | 条件 | 理由 |
|------|------|------|
| **重量** | 5kg以下 | UR5eペイロード制限 |
| **サイズ** | 425ビン内寸（603×403×404mm）に収まる | AutoStore 425ビン確定値 |
| **把持可能性** | 吸着 or 平行グリッパで把持可能な形状 | α版EE制約 |
| **VLA価値** | ルールベースでは把持困難 | VLAの価値実証が目的 |
| **安全性** | 鋭利でない、壊れにくい | α版の安全要件 |

> **具体的なパーツリスト:** RoboSyncチーム・奥村さんと協議して選定（FY25 Q4）。
