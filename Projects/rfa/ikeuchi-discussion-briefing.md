## 2. 得られた知見①：VLAの学習単位（GMR）

### 2.1 従来の問題

**動詞ベースの分類の限界**:
- 「掴む」「置く」では曖昧すぎる
- 同じ「掴む」でも物体・状況によって全く違う動作が必要
- スキルの境界が不明確 → 再利用できない

**【図①挿入箇所】**: 動詞ベース分類の問題を示す図
- 「Pick」という1つの動詞に対して、実際には多様な動作パターンが存在することを示す
- 例：上から掴む / 横から掴む / 柔軟物を掴む / ケーブル付きを掴む

### 2.2 池内先生のアプローチ：接触状態遷移（ICF）

**核心**: スキル = **接触状態の遷移**

ロボット操作を「物体と環境の接触状態がどう変化するか」で定義する。

| 接触状態 | 意味 | 例 |
|---------|------|-----|
| NC（非接触） | 触れていない | 空中移動中 |
| PC（面接触） | 面で接触 | テーブル上に置いた状態 |
| FT（完全固定） | 動かない | ビス締結後 |

**スキルの定義**:
> スキル = ある接触状態から別の接触状態への遷移

例：
- Pick = NC → 把持状態
- Place = 把持状態 → PC（面接触）
- 挿入 = PC → FT（完全固定）

### 2.3 GMR（Grasp-Manipulation-Release）

池内先生の論文（arXiv:2008.01513, 2020）で提唱された概念。**GMR単位**でVLA Policyを分割することで合意。

**GMRとは**: ロボット操作タスクを3つの基本フェーズに分解した概念

| フェーズ | 内容 | 例 |
|---------|------|-----|
| **Grasp（把持）** | 対象物を安定して掴む | コップの側面を掴む |
| **Manipulation（操作）** | 掴んだまま動かす | コップを移動、フタを回す |
| **Release（解放）** | 安全に手を離す | 棚の上に置いて離す |

**なぜGMRが重要か**:
- 日常・物流・製造作業の大半は「掴んで → 何かして → 離す」に分解できる
- **中間レベルの抽象表現**として、LLM/VLAとの相性が良い

```
低レベル:  joint / torque / waypoint
中レベル:  GMR（意味的操作単位）← ここでVLAを分割
高レベル:  「箱を棚に収納する」などの言語指示
```

**【図②挿入箇所】**: GMRの概念図
- 1つのGMRの中に複数の接触状態遷移が含まれる
- GMR内は連続制御（VLA）、GMR間はLLM/VLMで切替
- 例：「箱詰め」GMR = 接近 → 把持 → 移動 → 配置 → リリース

### 2.4 梱包タスクでのGMR分割例

| GMR | 内容 | 特徴 |
|-----|------|------|
| 商品ピッキング | コンテナから商品を取り出す | 片手、不定形対応 |
| 箱詰め（片手） | 小物を箱に入れる | 片手 |
| 箱詰め（両手） | 大物を箱に入れる | 両手協調 |
| 箱閉め | フラップを折り畳む | 両手、シーケンス |
| テープ貼り | 封緘テープを貼る | 軌道追従 |

**ポイント**: 
- 「片手で入れる」と「両手で入れる」は別GMR（物理特性が異なる）
- 「シェーバーを入れる」と「チャージャーを入れる」は同じGMR（物理特性が同じ）

→ **物体ごとではなく、物理特性でスキルを分類** → 再利用性が大幅に向上

## 3. 得られた知見②：再利用可能なスキル体系

### 3.1 池内先生の重要な指摘

> 「物体ごとにスキルを分けると扱う物が増えた時に問題。『Xを箱に入れる』という汎用スキル化が必要」

### 3.2 再利用のための2つの軸

**【図③挿入箇所】**: タスク分類の二軸フレームワーク

| 軸 | 説明 | 例 |
|----|------|-----|
| **タスクグループ**（業務フロー） | 業務単位の分解 | 梱包フロー、組立フロー |
| **GMRカテゴリ**（物理パターン） | 物理的操作の分類 | ピッキング系、配置系、貼付系 |

この二軸で整理することで：
- **横展開が容易に**: 「梱包のピッキング」で学んだスキルを「製造のキッティング」に転用
- **必要なスキルの見積もりが可能に**: 新タスクに必要なGMRを事前に特定

### 3.3 iREXデモからの実証データ

| 項目 | 数値 |
|------|------|
| 1スキル（GMR）あたりのデータ | 約200エピソード |
| データ収集時間 | 約90分/スキル |
| 学習時間（Pi0ファインチューニング） | 約1日/スキル |

→ **新GMRを追加するコストが明確化** → 事業計画に織り込み可能

## 4. 得られた知見③：継続学習サイクル

### 4.1 池内先生の指摘：エラーリカバリーの重要性

> 「本来、アクションというのは何らかのアクションがあって、なおかつそのアクションが終わりましたという判定まで含めたものがアクション」
> 「すべてのアクションにおいて、エラーリカバリーが必要」

### 4.2 現実的なアプローチ：完璧を目指さない

**池内先生の指摘**:
> 「100%って絶対難しいから。90%やっといて、10%があったらなんか呼ぶっていうのが一番いい」

**合意事項**:
- 最初から100%自動化を目指さない
- **エラー時は人間を呼ぶ運用が現実的**
- 人間介入のデータを学習に活用（Human-in-the-Loop）

### 4.3 継続学習サイクルの設計

**【図④挿入箇所】**: 継続学習サイクルの全体像

```
┌─────────────────────────────────────────────────────────────┐
│  現場で実行                                                  │
│      │                                                      │
│      ▼                                                      │
│  VLMが失敗検知 ──────→ 自律リカバリー試行                    │
│      │                       │                              │
│      │ (リカバリー不可)       │ (成功)                       │
│      ▼                       ▼                              │
│  人間が介入 ─────────────→ データ蓄積 ←─────────────────────│
│                              │                              │
│                              ▼                              │
│                         学習/Fine-tune                       │
│                              │                              │
│                              ▼                              │
│                          再デプロイ ──→ 現場で実行...        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**ポイント**:
- 失敗も価値あるデータになる
- 人間介入が多いスキルから優先的に改善
- 「使いながら賢くなる」が差別化の核心

## 5. 事業的意義

### 5.1 スキル再利用による展開コスト削減

**従来**（スキル再利用なし）:
- 新タスクごとに一からデータ収集・学習
- タスク数に比例してコスト増大

**GMRベース**（スキル再利用あり）:
- 既存GMRの組み合わせで多くのタスクをカバー
- 新GMRが必要な場合のみ追加学習
- **タスク数の増加に対してコストが漸近的に収束**

**【図⑤挿入箇所】**: 展開コストの比較グラフ
- X軸: タスク数
- Y軸: 累積コスト
- 2本の曲線: 「従来（線形増加）」vs「GMRベース（漸近収束）」

### 5.2 理論的基盤による説明可能性

**池内先生のフレームワーク（PTG/STG/ICF）** を採用することで：

| 効果 | 内容 |
|------|------|
| **顧客への説明** | 「なぜこのスキル分割なのか」を物理的根拠で説明可能 |
| **技術者の共通言語** | チーム内・パートナーとの議論が効率化 |
| **デバッグの容易化** | 問題が「どのGMRの、どの接触遷移で起きたか」を特定しやすい |

### 5.3 研究と事業の両立

**現状の位置づけ**:
- IndexVLA論文をRA-L（トップジャーナル）に投稿予定
- 池内先生の理論的バックボーンにより学術的価値も担保

**意義**:
- 「単に動くデモを作る」ではなく、**理論的基盤を持った技術**として確立
- 将来的な技術の拡張・発展の土台になる
- 採用・ブランディングにも寄与

## 6. VLAが価値を出す領域

### 6.1 少量多品種・動的変化が主戦場

**池内先生との議論からの合意**:

| 領域 | VLAの適性 | 理由 |
|------|----------|------|
| 少量多品種 | ◎ | SKU変更のたびに再教示不要 |
| 動的状況変化 | ◎ | 毎回違う状態に対応可能 |
| 属人的判断 | ◎ | 暗黙知をデータから学習 |
| 一品種大量・固定 | △ | 従来技術でも対応可能 |

### 6.2 梱包タスクが最適な理由

池内先生のフレームワークで分析した結果：

> **梱包は「物理的には単純」だが「意味的には複雑」**

- 多くの工程が**STG（意味タスクグループ）**に属する
  - 例：「水をこぼさない」「壊れ物を守る」などの意味的制約
- 純粋な物理タスク（PTG）は少数
- → **VLAが最も価値を出せる典型タスク**
