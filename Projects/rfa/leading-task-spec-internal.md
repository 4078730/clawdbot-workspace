### 2.2 α版の制御フロー

α版では **Sensorimotor Hub（VLA Policy）+ Robo Sync** のみ。Task Planning Hub、Agentic制御は不使用。

```
入力                         処理                          出力
─────                       ────                          ────
カメラ画像（RGB）    ──→  VLA Policy  ──→  関節角度制御
  640×480 / 30fps            （推論）         （q1〜q6）
  + ロボット関節角度                        + グリッパ開閉指令
  + パーツ情報（PMS）
    パーツ名・品番・
    ビン内ピック位置
        │
        ▼
    Robo Sync
    ・安全チェック（速度・力・範囲制限）
    ・UR5eへの指令送信
    ・実行ログ記録
```

#### VLAモデル（確定）

**α版採用モデル: π0 / π0.5 (Physical Intelligence)**

| モデル | 特徴 | α版での位置づけ |
|--------|------|---------|
| **π0** | Physical Intelligence社。Flow Matching + VLA。汎化性能が高い | **採用確定**。Fine-tuningベースライン |
| **π0.5** | π0の後継。マルチタスク・スケーラビリティ向上版 | **採用確定**。π0と並行評価 |


#### VLA Policy の入出力仕様（α版）

| 項目 | 仕様 |
|------|------|
| **入力: 画像** | RGB 画像（Hand-Eye カメラ）/ **640×480 / 30fps** / 深度は使用しない |
| **入力: ロボット状態** | 関節角度 6次元（q1〜q6）+ グリッパ状態 1次元 = 7次元 |
| **入力: パーツ情報** | PMSから取得するパーツ仕様（パーツ名・品番・ビン内ピック位置 ※仕切りがある場合）。言語指示ではなくパーツ仕様書として読み込む |
| **出力: アクション** | **関節角度制御** — 各関節の目標角度 6次元（q1〜q6）+ グリッパ指令 1次元 = 7次元 |
| **制御周期** | **10Hz**（確定） |

### 2.3 ルールベースとVLAの切替条件

**設計原則:** VLAはルールベースの「置き換え」ではなく「段階的補完」。

```
                パーツ認識
                    │
            ┌───────┴───────┐
            │               │
     ルールベースで       ルールベースでは
     対応可能か？         対応不可
            │               │
            ▼               ▼
     ┌────────────┐   ┌────────────┐
     │ルールベース │   │ VLA Policy │
     │ピック       │   │ ピック      │
     │（既存）     │   │（新規）     │
     └────────────┘   └────────────┘
```

| 条件 | 制御方式 | 判定方法 |
|------|---------|---------|
| 箱物・整列済み・吸着面が明確 | ルールベース | 画像認識で物体カテゴリ判定 |
| 不定形・包装状態が複雑・把持点不明確 | VLA Policy | 上記以外（α版初期は全VLAも可） |
| VLA推論が異常値を返した場合 | **停止→人介在** | アクション値の範囲チェック |
| VLAが一定時間内にタスク完了しない場合 | **タイムアウト→人介在** | タイマー（閾値TBD） |

> **α版初期の方針:**
> 最初はVLAのみで動かし、性能データを蓄積する。
> ルールベースとの切替は、VLAの得意/不得意が明確になった段階で導入。

### 2.4 β版の制御フロー

β版では **Task Planning Hub（LLM Agent）** が加わり、8工程のオーケストレーションを行う。

```
LLM Agent（Task Planning Hub）
│
├─ [1] バーコードスキャン → スキャンスキル呼出
├─ [2] モニター操作      → UI操作スキル呼出
├─ [3] 段ボール開封       → 開封スキル呼出（双腕）
├─ [4] パーツピック       → ピックスキル呼出
├─ [5] ビンに戻す        → 配置スキル呼出
├─ [6] 個装              → 個装スキル呼出（双腕）
├─ [7] 台車配置          → 配置スキル呼出
└─ [8] 次のオーダー      → サイクル管理

各スキルは Robo Sync 上のスキルブロックとして実装
スキル間の遷移は BT（ビヘイビアツリー）で定義
```

### 2.5 RoboSyncとの統合

#### スキルブロックの役割分担

| レイヤ | 責任 | 実装場所 |
|--------|------|---------|
| **タスク管理** | スキル選択・切替・再計画 | Task Planning Hub（β版） |
| **スキル実行** | VLA推論 → アクション生成 | VLAスキルブロック（RoboSync上） |
| **安全制御** | 速度制限・力制限・範囲制限・緊急停止 | RoboSync標準安全レイヤ |
| **ハード制御** | UR5eドライバ・EE制御・カメラ取得 | RoboSyncハード抽象化レイヤ |
| **ログ収集** | 画像・関節角・アクション・成功/失敗の記録 | RoboSync DataOpsレイヤ |

## 3. タスクプロセス詳細

### 3.1 α版：ビン到着 → ピック → 配置（ステップバイステップ）

```
ステップ   動作                          入力（センサー）         出力（アクション）           備考
─────────────────────────────────────────────────────────────────────────────────────────────
S0        初期位置に移動                 ロボット関節角度         ホームポジションへ移動       ルールベース
          │
S1        ビン到着を検知                 俯瞰カメラ画像          ビン到着フラグ               画像認識 or
          │                                                                                 オートストア連携
S2        ビン内を撮影・認識             Hand-Eye RGB-D          パーツ候補リスト              VLM or 物体検出
          │                             俯瞰カメラ RGB-D        ・位置、姿勢推定
          │                                                     ・包装状態分類
          │
S3        ピック対象を決定               S2の認識結果            対象パーツID                  α版: 1個ずつ処理
          │                             パーツ情報（PMS）         把持戦略（吸着 or グリッパ）
          │
S4        アプローチ                     Hand-Eye RGB-D          Joint制御（アプローチ）         VLA Policy
          │                             ロボット状態             ・ビン上方からパーツ上方へ
          │
S5        把持動作                       Hand-Eye RGB-D          Joint制御（把持）               VLA Policy
          │                             ロボット状態             ・パーツに接近
          │                                                     ・吸着ON or グリッパ閉
          │
S6        持ち上げ                       Hand-Eye RGB-D          Joint制御（リフト）             VLA Policy
          │                             ロボット状態             ・パーツをビンから持ち上げ
          │                             （力/吸着圧センサ）       ・把持成功を確認
          │
S7        搬送                          ロボット状態             Joint制御（搬送）               VLA or ルールベース
          │                                                     ・ビン上方→作業台上方へ
          │
S8        配置                          Hand-Eye RGB-D          Joint制御（配置）               VLA Policy
          │                             ロボット状態             ・作業台上に降ろす
          │                                                     ・吸着OFF or グリッパ開
          │
S9        配置確認                       俯瞰 or Hand-Eye        成功/失敗判定                 VLM判定
          │                                                     ・パーツが作業台上にあるか
          │                                                     ・向き・位置は適切か
          │
S10       初期位置に復帰                 ロボット状態             ホームポジションへ移動        ルールベース
```

#### 各ステップの時間見積もり（目標）

| ステップ | 動作 | 目標時間 |
|---------|------|---------|
| S0 | 初期位置移動 | 2秒 |
| S1 | ビン到着検知 | 1秒 |
| S2 | 認識 | 2秒 |
| S3 | 対象決定 | 1秒 |
| S4-S6 | アプローチ→把持→持ち上げ | 10秒 |
| S7 | 搬送 | 3秒 |
| S8 | 配置 | 3秒 |
| S9 | 配置確認 | 2秒 |
| S10 | 復帰 | 2秒 |
| **合計** | **1ピックサイクル** | **約26秒（目標）** |

> **参考:** 人間の作業者は1オーダー約1〜3分（複数パーツ含む）。
> α版の目標は「人と同等〜2倍の時間」で、速度より確実性を優先。

#### 困難なケース別の対応方針

| ケース | 状況 | VLAに期待する動作 |
|--------|------|------------------|
| **プチプチ包装** | パーツがプチプチに包まれてビン内にある | 包装ごと把持→持ち上げ。包装を剥がす操作はα版スコープ外 |
| **袋入り** | 小型部品が透明袋に入っている | 袋ごと把持。袋から出す操作はα版スコープ外 |
| **立てかかっている** | パーツがビン壁面に立てかかっている | 倒してから把持、または斜めからアプローチ |
| **段ボール内に埋もれている** | パーツが段ボールの奥にある | 上のパーツを退避→対象パーツにアクセス（困難な場合は人介在） |
| **複数パーツが密着** | パーツ同士が重なっている | 1個だけを分離して把持（分離困難な場合は人介在） |

### 3.2 β版：8工程の詳細プロセス

#### 工程[1] バーコードスキャン

```
入力:  俯瞰カメラ画像（納品書の位置検出）
処理:  ロボットがコードリーダーを把持 → 納品書上のバーコードに位置合わせ → スキャン実行
出力:  オーダー情報（品番・数量・個装指示）
技術:  ツール操作スキル（コードリーダーの把持と操作）、バーコード位置検出（画像認識）
課題:  バーコードリーダーの把持安定性、読み取り角度の調整
```

**伝票の実物仕様（現場確認済み）:**

| 項目 | 内容 |
|---|---|
| 伝票セット構成 | 3種類1セット：①売上計算票（経理用）②納品通知票・出荷明細票（仕入先元帳用）③入庫指図票・出庫票（商品元帳用） |
| バーコード種別 | 1次元バーコード（Code 128系）、各伝票右下に印刷 |
| スキャン対象 | 3枚のうち最上面の1枚のバーコードをスキャン（伝票番号・オーダー番号に対応） |
| スキャンで取得する情報 | 品番（例: ET-LAV300）、棚番号（例: B02A-C000）、数量、個装形態コード（例: NW1） |
| 伝票の置き方 | 作業台上にまとめて積まれた状態で到着。上面が読み取り対象 |

#### 工程[2] モニター操作（ビン呼び出し）

```
入力:  オーダー情報（工程[1]の出力）、モニター画面画像
処理:  ロボット or ソフトウェア連携でオートストアにビン呼び出し指令を送信
出力:  ビン呼び出し完了通知
技術:  API連携（オートストアシステムとの直接通信）or タッチパネル操作（VLA）
課題:  オートストアとのシステムIF定義、タッチ操作の精度
備考:  API連携が可能ならロボットの物理操作は不要。要現場確認
```

#### 工程[3] ビン内段ボール開封（必要時）

```
入力:  ビン内の画像（段ボール有無の判定）
処理:  片腕で段ボールを押さえ、もう片腕でフラップを開ける
出力:  段ボール開封完了、パーツが見える状態
技術:  双腕協調制御、力制御（段ボールを破壊しない程度の力）
課題:  段ボール形状の多様性（テープ有無、フラップの向き）、双腕VLAの難易度
前提:  双腕ロボットが必要
```

#### 工程[4] パーツのピック（α版拡張）

```
入力:  ビン内画像、オーダー情報（品番・数量）
処理:  α版のピック動作を拡張
       ・指定品番のパーツを識別（12万品番の一部）
       ・指定数量をピック（「3個取り出す」→ 3回ピックサイクル）
       ・緩衝材からの取り出し（プチプチを剥がす等）
出力:  指定パーツが作業台上に配置された状態
技術:  VLA Policy（α版ベース）+ 品番識別（VLM）+ カウント
課題:  品番識別の精度、数量管理、緩衝材除去の制御
```

#### 工程[5] 残りをビンに戻す

```
入力:  ビン内・作業台の画像
処理:  取り出した後のビン内を整理。緩衝材や不要な梱包材をビンに戻す
出力:  ビンが元の状態に近い形で整理された
技術:  VLA Policy（整理・配置スキル）
課題:  「元の状態」の定義、壊れ物の取り扱い
```

#### 工程[6] 個装（袋入れ + 納品書同封 + 袋閉じ）

```
入力:  作業台上のパーツ、納品書（3枚セット）、個装用袋
処理:
  6a. 個装形態コード（伝票から取得）に基づいて適切なサイズの袋を選択
  6b. 片腕で袋を開き、もう片腕でパーツを入れる
  6c. 納品書（3枚セットのうち指定の1枚）を袋に同封
  6d. 袋を閉じる（輪ゴム or シーラー — 個装形態コードで指定）
出力:  個装済みパーツ（袋入り・納品書同封）
技術:  双腕協調制御、柔軟物操作（袋）、VLA Policy
課題:  袋を開く動作（柔軟物操作の高難度タスク）、輪ゴム留め、納品書の枚数管理
備考:  輪ゴム留めは特に高難度。β版で全工程を完遂できるかは検証次第
```

**個装形態コードの実物仕様（現場確認済み）:**

| 項目 | 内容 |
|---|---|
| コード例 | NW1（伝票上の「個装形態」欄に印字）|
| 意味 | 品番ごとに袋のサイズ・包装方法・同封書類が異なる（コード体系は要詳細確認） |
| 同封する納品書 | 3枚セットのうち指定の1枚（どれかは要確認） |
| 袋サイズのバリエーション | 作業台上に各種サイズの小袋が常設されており、コードに応じて選択 |
| ⚠️ 要確認 | 個装形態コードの全種類と対応する袋サイズ・包装手順の一覧（中田さん/一力さんに確認）|

#### 工程[7] 台車/ワゴンに配置

```
入力:  個装済みパーツの画像、台車の空き位置
処理:  個装済みパーツを台車/ワゴンの適切な位置に配置
出力:  パーツが台車上に配置された
技術:  配置スキル（VLA or ルールベース）、台車位置認識

| 項目 | 仕様 | 出典 |

> - ビン内寸 603×403mm、深さ 404mm → リーチ・手首角度の制約条件として反映すること
> - 仕切り有り・なしの両ケースをVLAが識別・対応できることを検証項目に追加
> - UR5e最大リーチ850mm に対してビン底まで届くか要検証（台座・ビン位置間の距離次第）

### 5.3 照明条件

| 項目 | 仕様 | 備考 |
|------|------|------|
| **照明タイプ** | 室内蛍光灯 / LED照明 | 倉庫内の一般照明。要現場確認 |
| **明るさ** | 推定 300〜500 lux | 一般的な倉庫照明レベル |
| **均一性** | やや不均一（動画で影が確認される） | 追加照明が必要かは要検討 |

> **⚠️ 検討事項:**
> - RealSense D405はIR投光型だが、外光の影響を受ける可能性あり
> - 追加のLED照明（リングライト等）をロボットアーム先端に設置するか検討

### 5.4 背景・外乱要因

| 外乱 | 影響 | 対策 |
|------|------|------|
| **人の往来** | ロボット作業範囲への侵入リスク | 安全柵 or ライトカーテン設置。RoboSync安全制約で減速/停止 |
| **他のオートストアポート** | 隣接ポートの作業者・ロボットとの干渉 | プロパティ登録で専用ポートを確保（一力さんチーム確認済み） |
| **ビンの揺れ** | オートストアからビン到着時の振動 | ビン到着後、安定するまで待機（1〜2秒） |
| **パーツの配置変動** | 輸送中にビン内でパーツが動く | 毎回カメラで認識し直す（把持位置を固定しない） |
| **温度変化** | 季節による温度変動 | ロボット精度には影響軽微。カメラキャリブレーションの定期実施 |
