## 2. 理論的背景：池内先生の研究

### 2.1 Learning-from-Observation（LfO）

池内先生が30年以上研究してきた枠組み。人間の操作ビデオを入力として、ロボットに操作シーケンスを学習させる。

**従来のLfD（Learning from Demonstration）との違い**：
- LfD: 状態＋行動データが必要
- LfO: 観察のみから学習 → 現実適用性が高い

### 2.2 タスクモデルの構造

LfO枠組みの中心は「タスクモデル」：

**構成要素**：
- **スロット**: ACTOR, OBJECT, 開始/終了姿勢, 進行方向など
- **デーモン関数**: 実演データからスロットを埋める処理

**役割分担**：
- LLM: 言語指示＋環境情報からどのタスクモデルを選ぶか、どのスロットをどう埋めるか
- VLA: 埋められたタスクモデルを具体的な軌道として実行

### 2.3 IJRR論文の位置づけ

池内先生のIJRR論文の一文サマリ：

> 「家庭内作業を成立させている"常識"を、物理制約と同一の数学的枠組みで扱える Semantic Constraint として定義し、人間作業を網羅するタスクモデル体系（PTG / STG / MTG）を導出した論文」

**RFAへの効果**：
- スキルを「人間の動詞」「デモ動画単位」ではなく、「接触状態の遷移」という普遍単位で定義できる
- LLMが扱う「意味的意図」とロボットが扱う「物理拘束」を同一言語で接続可能

## 4. 重要な合意事項（4方針）

2025年11月12日のMTGで、以下の4方針が合意された。

### 方針1: スキルセットはトップダウンで定義

自律的ボトムアップ獲得は現状のVLA技術では時期尚早。少量デモで実用レベルにするには**人間がスキルセットを定義する方が現実的**。「学会的理想」よりも早期市場投入MVPを優先。

### 方針2: 価値が最大なのは「少量多品種・動的状況変化」

一品種多量生産の固定タスクは従来技術が有利。LLM/VLAは**毎回状況が変わるタスク**でこそ真価を発揮。物流／柔軟な製造ラインが主戦場。

### 方針3: スキルは「オブジェクト中心の物理現象」として定義

動詞（掴む・置く）ではなく、**接触状態の変化（ICF）**で表現 → PTG/STG。VLAで解決すべきは「不定形対象物に対して目的の接触状態を確立するスキル」。

### 方針4: LLMのプランニングはビヘイビアツリー（BT）

可視性・デバッグ・条件分岐・業界標準性の点で有利。将来的にRobo Syncのタスクシナリオも BT的表現へ進化。

## 5. GMR（Grasp-Manipulation-Release）

### 5.1 GMRの定義

池内先生の論文（arXiv:2008.01513, 2020）で提唱された概念。

**GMR = Grasp – Manipulation – Release**

| フェーズ | 内容 | 例 |
|---------|------|-----|
| **Grasp（把持）** | 対象物を安定して掴む | コップの側面を掴む、引き出しの取手を掴む |
| **Manipulation（操作）** | 掴んだまま対象物を動かす | コップを移動、ドアを引く/押す、フタを回す |
| **Release（解放）** | 安全に手を離す | 棚の上に置いて離す |

### 5.2 GMRが重要な理由

**家庭・物流・現場作業の多くがGMR構造**：
- 日常的な作業の大半は「掴んで → 何かして → 離す」に分解できる
- 物を取って移動する
- 箱を掴んで向きを変えて置く
- 引き出しを開けて手を離す

**タスク抽象化として強力**：
GMRは「中間レベルの抽象表現」として機能する。

| レベル | 内容 |
|--------|------|
| 低レベル | joint / torque / waypoint |
| **中レベル** | **GMR（意味的操作単位）** |
| 高レベル | 「箱を棚に収納する」などの言語指示 |

→ LfO、LfD、VLA/LLM-Agent×ロボットと非常に相性が良い。

### 5.3 VLA Policy の単位

**現時点の合意仮説**：
- VLA Policy = 1 GMR
- GMR内部は連続制御で閉じる
- GMR間は LLM / VLM で切替

### 5.4 オペランド設計

GMRのパラメータ構造：
- (target, initial_position, final_position)
- タスクによって可変
- IndexVLA はこの未定オペランドを埋める役割

### 5.5 梱包タスクでのGMR分割例

| GMR | 内容 | 手の使い方 |
|-----|------|----------|
| 商品ピッキング | コンテナから商品を取り出す | 片手 |
| 箱詰め（片手） | 小物を箱に入れる | 片手 |
| 箱詰め（両手） | 大物を箱に入れる | 両手 |
| 箱閉め | フラップを折り畳む | 両手 |
| テープ貼り | 封緘テープを貼る | 両手 |

**重要なポイント**：
- 「シェーバーを入れる」と「チャージャーを入れる」は**同じGMR**（物理特性が同じ）
- 「片手で入れる」と「両手で入れる」は**別のGMR**（物理特性が異なる）

## 6. 接触状態とタスク分類（ICF/PTG/STG）

### 6.1 ICF（Interaction / Contact Form）

物体と環境の接触状態を定義する概念。

| コード | 状態 | 説明 |
|--------|------|------|
| NC | 非接触 | 触れていない |
| PC | 面接触 | 面で接触 |
| PR | 線接触 | 線で接触 |
| TR | 点接触 | 点で接触 |
| OT | その他の接触 | - |
| RV | 回転拘束あり | 回転が制限される |
| NR | 回転拘束なし | 回転は自由 |
| FT | 完全固定 | 完全拘束に近い固定 |

**「接触状態の遷移」が1つの"スキル"発生単位**として扱われる。

### 6.2 PTG（Physical Task Group）

純粋な物理制約に従うタスク分類。工業組立で有効。

| コード | 名称 | 説明 |
|--------|------|------|
| PTG11 | Detach/Picking | 分離・ピッキング |
| PTG12 | Bringing | 運搬 |
| PTG13 | Attach/Placing | 取付・配置 |
| PTG31 | LinearOpen | 線形開放 |
| PTG32 | LinearAdjust | 線形調整 |
| PTG33 | LinearClose | 線形閉鎖 |
| PTG51 | RotateOpen | 回転開放 |
| PTG52 | RotateAdjust | 回転調整 |
| PTG53 | RotateClose | 回転閉鎖 |

### 6.3 STG（Semantic Task Group）

人間の意図・常識に基づく仮想的制約を含んだタスク分類。梱包・家庭作業・柔軟物操作で支配的。

| コード | 名称 | 説明 | 例 |
|--------|------|------|-----|
| STG11 | CarefullyDetach | 慎重な分離 | 壊れ物を取り出す |
| STG12 | CarefullyBring | 慎重な運搬 | 水をこぼさない |
| STG13 | CarefullyAttach | 慎重な配置 | 壊れ物を置く |
| STG2 | PlanarTranslate | 平面移動（接触維持） | テーブル拭き |
| STG3 | LinearTranslate | 線形移動（仮想チューブ拘束） | 皮をむく |
| STG4 | SphericalMove | 球面移動（仮想球面拘束） | お椀の内側を洗う |
| STG5 | Rotate | 回転（仮想ヒンジ拘束） | 注ぐ |
| STG6 | Fix | 固定 | まな板上で野菜を固定 |

### 6.4 Semantic Constraint（見えない制約）

物理的には可能だが、目的・常識のために"動かないべき自由度"。

| 制約タイプ | 説明 | 例 |
|-----------|------|-----|
| Semantic Pin | 点を動かさない | 水をこぼさない |
| Semantic Wall | 面を離さない | 雑巾を離さない |
| Semantic Tube | 軌道を維持 | 皮むき |

### 6.5 MTG（Multi-step Task Group）

PTG / STG の組み合わせ。梱包、箱組立、貼付など複合タスク。

## 7. IndexVLAの理論的背景

### 7.1 記号接地問題

**課題**：
LLMが「電池を掴んで箱に入れて」と言えても、VLAには「どの電池」「どこを」「どの軌道で」掴めば良いかが曖昧。

### 7.2 Peirceの記号論

IndexVLAはPeirceの記号論に基づく：

| 記号タイプ | 説明 | 例 |
|-----------|------|-----|
| **Icon** | 画像（視覚的類似） | 写真、図 |
| **Symbol** | 言語（意味） | 「電池」という単語 |
| **Index** | 特定の対象・位置・方向を指し示す | 矢印、座標 |

### 7.3 Indexカテゴリ

IndexVLAでは、以下を使って「どの物体のどの部分を、どの方向にどう動かすか」を構造化されたIndex情報としてVLAに渡す：

| カテゴリ | 説明 | 用途例 |
|---------|------|--------|
| Point-to-Point (P2P) | 始点・終点の2点指定 | Pick-and-Place系 |
| Trajectory | 軌道そのものが意味を持つ | 拭く・線をなぞる・接着剤塗布 |
| Vector | p_object / p_target / v_action の3点構造 | 引き出しを引く・プラグ挿入 |

### 7.4 Affordance / Signifier / Index の関係

| 概念 | 提唱者 | 説明 |
|------|--------|------|
| **Affordance** | ギブソン | 物体が潜在的に持つ行為の可能性 |
| **Signifier** | ノーマン | どのaffordanceを選ぶべきかを伝える手がかり |

**池内先生との整理**：
- 世界は多様なaffordanceを持つ物体で満ちている
- 言語指示（「あの電池を箱に入れて」）は「どの物体」「どのaffordance列」を選ぶべきかを大まかに示すシグニファイア
- IndexVLAが付与するkeypointやtrajectoryは、曖昧さをさらに潰した**人工的シグニファイア**
- VLAは、言語・コンテキスト・Indexを統合し、対象物のaffordanceの中から、目的に合う一つの行動を同定して実行する

### 7.5 IndexVLA = LLMとVLAの"通訳"

2025年11月5日のMTGで得られた言語化。

IndexVLAは、LLMの抽象的な指示と、VLAの具体的な制御の間を橋渡しする「通訳」の役割を果たす。

### 8.2 得られた示唆

> **梱包は「物理的には単純」だが「意味的には複雑」**

- 多くの工程が**STG**に属する
- 純粋なPTGは少数
- 双腕協調が頻出
- **VLAが最も価値を出せる典型タスク**

### 8.3 段ボール箱組み立ての分析（2025年11月27日）

**主な論点**：
- 現状の工程分解の粒度が粗い → 右手／左手ごとの時系列分解が必要
- 再利用可能なロボットスキルは「箱端のピック」「折り目に沿うベンディング」「ホールド」「ローテーション」など数個の基本スキル＋パラメータに落とすのが良い

**人間の身体性とロボットの違い**：
- 人間は利き手／非利き手＋身体（腹・机の縁）でタスクを行う
- ロボットでは「器用ハンド／ホールドハンド」の役割分担か、完全対称か、設計判断が必要

**箱の展開図とロボット可動域の相性**：
- 「人にはやりやすいがロボットには難しい箱」が存在する
- 逆にロボットフレンドリーな箱設計という発想もあり

## 9. iREXデモの知見

### 9.1 デモ概要

**タスク構成**: シェーバー梱包の4段階タスク
1. シェーバーを箱に入れる
2. チャージャーを箱に入れる
3. マニュアルを箱に入れる
4. 箱を閉める

**データ条件**：
| 項目 | 数値 |
|------|------|
| 各スキル | 約200エピソード |
| データ収集 | 約90分/スキル |
| Pi0 ファインチューニング | 約1日/スキル |

**技術構成**：
- 4つのVLA Policyを独立サーバとして常時起動
- VLM（Gemini-Robotics-ER）による状態監視・スキル選択
- スキル切替時に初期姿勢リセット（Distribution Shift対策）

### 9.2 スキル構成の詳細

| スキル | 内容 | 手の使い方 | リカバリー |
|--------|------|----------|-----------|
| Skill 1 | シェーバーを箱に入れる | 片手（左） | はみ出し修正（右手） |
| Skill 2 | チャージャーを箱に入れる | 片手（左） | ケーブルはみ出し対応（一部） |
| Skill 3 | マニュアルを箱に入れる | 両手 | - |
| Skill 4 | 箱を閉める | 両手 | 閉まり判定必要 |

### 9.3 技術的成果

- **スキル分割＋切替が実際に動くことを証明**
- **Distribution Shift対策**：スキル切替時の初期姿勢リセットが有効
- **VLM切替方式**：スキルごとに個別特化Policyを育て、VLMでオーケストレーションする構造は実装可能
- **データ量のオーダ感**：1スキル ≒ 200エピソード / 90分 / 学習1日で実用レベルの挙動

### 9.4 見えた課題

- スキル粒度の設計が未確立
- リカバリーは模倣だけでは不可
- 継続学習のDataOpsが必須

### 9.5 池内先生からの主要な指摘

| 論点 | 内容 |
|------|------|
| **スキルの汎化問題** | 物体ごとにスキルを分けると扱う物が増えた時に問題。「Xを箱に入れる」という汎用スキル化が必要 |
| **VLAの過学習** | ファインチューニングで特定物体・動作に特化してしまう。プレトレインの汎化能力が失われる |
| **エラーリカバリーの設計** | すべてのアクションに成功/失敗判定とリカバリーを含めるべき。これがスキルの本来の姿 |
| **スキル分割の基準** | 片手で入れる（シェーバー、チャージャー）vs 両手で入れる（マニュアル）は別スキルでよい |
| **物理特性による分類** | シェーバー＋ケーブル（変形物付き剛体）は統合可能、マニュアル（柔軟物）は別 |

### 9.6 エラーリカバリーについての重要な議論

**池内先生の指摘**：
> 「本来、アクションというのは何らかのアクションがあって、なおかつそのアクションが終わりましたという判定まで含めたものがアクションというんであって」
> 「単純なピックやプレースに関しても、本来置こうとした時に倒れてしまうなんてことあると思う」
> 「すべてのアクションにおいて、そういうふうなエラーリカバリーが必要」

**合意事項**：
- VLMでエラー検知 → 人間を呼ぶ → データ収集という運用が現実的
- 最初は90%自動化 + 10%人間介入でよい
- ターゲット（物流倉庫）では箱物が多く、エラーリカバリーが必要なのは緩衝材投入など限定的

## 11. VLA × RL融合の議論

### 11.1 課題認識

- 模倣学習のみではOOD（Out-of-Distribution）を克服できない
- リカバリー能力は経験から獲得すべき

### 11.2 方針

- Human-in-the-Loop
- RL + Imitation
- 継続学習を前提とした設計

### 11.3 VLA × RL の役割分担

```
┌─────────────────────────────────────────────────────────┐
│ VLM / LLM                                               │
│ - 失敗状態の検知                                         │
│ - 「今どのスキルを呼ぶべきか？」の上位レベル意思決定       │
│ - プランニング・リプランニング                           │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ VLA（模倣ベース）                                        │
│ - データから「それらしく動く」スキルを短時間で獲得        │
│ - 大まかな軌道の再現は得意                               │
│ - 過学習による特化 vs 汎化のトレードオフあり             │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ RL / Pi*0.6 系                                          │
│ - VLAが苦手な最後の微調整（ラストワンマイル）            │
│ - 力加減や速度を伴うダイナミクス                         │
│ - Human-in-the-loop による介入データ活用                 │
└─────────────────────────────────────────────────────────┘
```

### 11.4 共有論文の示唆

- VLAとRLの融合は有望
- スイッチング方式 / 融合方式は今後検討
- スキルが定義されてからVLAにつなぐ

## 12. 継続学習とデータ戦略

### 12.1 データソースの階層構造

| 層 | データソース | 特徴 | RFAでの役割 |
|----|-------------|------|------------|
| 上位 | Human Teleoperation | 最も価値が高いが収集コストも高い | 初期スキル獲得、高精度Fine-tuning |
| 中位 | Synthetic Data | スケール可能、Sim-to-Realギャップあり | バリエーション拡張、ロバスト化 |
| 下位 | Human Videos | 大量に存在、ロボットコマンド含まず | 事前学習、意味表現の土台 |

### 12.2 データライフサイクル（Robo Sync経由）

1. **Model Deploy**: 学習済みポリシーをRobo Sync経由でデプロイ
2. **Logging**: 実行中のデータをRobo Syncでログ収集
3. **Data Export**: CECクラウド基盤へ転送、RLDS形式に変換
4. **Train / Fine-tune**: 現場ログを用いてモデル更新
5. **Re-Deploy**: 改善モデルを再デプロイ

### 12.3 継続学習サイクル

```
現場で実行 → VLMが失敗検知 → リカバリー試行 → HILで人間教示
    │                                              │
    │              ┌───────────────┐              │
    └──────────────│  データ蓄積   │←─────────────┘
                   └───────────────┘
                          │
                          ▼
                   ┌───────────────┐
                   │  学習/Fine-tune │
                   └───────────────┘
                          │
                          ▼
                   ┌───────────────┐
                   │   再デプロイ   │ → 現場で実行...
                   └───────────────┘
```

### 12.4 「90% + 10%」のアプローチ

**池内先生の発言**：
> 「100%って絶対難しいから。90%やっといて、10%があったらなんか呼ぶっていうのが一番いい」

完璧を目指さず、エラー時は人間を呼ぶ運用が現実的。

### 12.5 潜在空間の可視化（2026年1月21日の議論）

**議論内容**：
- Pi0やVLAの潜在空間ベクトルをPCAやt-SNEなどで可視化
- 時系列的な連続性やスキルの区切りがどのように表現されるかを調べる
- 複数モデルの確信度に基づくスキル切り替えの考え方

**技術的課題**：
- Pi0やVLAのバックボーンは従来の時系列依存型モデルと異なる
- 画像と言語に対して即応的な潜在状態を生成する構造
- 時系列的な依存性がどの程度獲得できているかは検証課題

### 12.6 データ再利用性（2026年1月21日の議論）

**池内先生の提案**：
- 模倣学習で取得したデータをいかに再利用するかが重要
- VLAのトークン抽出や言語情報の付与
- ビデオ解析によるキャプショニング

## 15. 用語集

| 用語 | 説明 |
|------|------|
| **VLA** | Vision-Language-Action Model。画像・言語・行動を統合したロボット制御モデル |
| **VLM** | Vision-Language Model。画像と言語を統合的に扱うAIモデル |
| **GMR** | Grasp-Manipulation-Release。把持→操作→解放の3フェーズで操作を分解 |
| **ICF** | Interaction/Contact Form。接触状態の分類 |
| **PTG** | Physical Task Group。物理制約のみのタスク分類 |
| **STG** | Semantic Task Group。意味的制約を含むタスク分類 |
| **MTG** | Multi-step Task Group。PTG/STGの組み合わせ |
| **LfO** | Learning from Observation。観察からの学習 |
| **LfD** | Learning from Demonstration。デモンストレーションからの学習 |
| **IndexVLA** | Index記号を用いたVLA制御手法。LLMとVLAの"通訳" |
| **π0 / Pi0** | Physical Intelligence社が開発したOSS VLA |
| **Robo Sync** | パナソニック コネクトが開発中のロボット制御プラットフォーム |
| **Distribution Shift** | 学習データと実行時データの分布の違い |
| **Human-in-the-Loop (HIL)** | 人間の介入を学習ループに組み込む手法 |
| **OOD** | Out-of-Distribution。学習データの分布外 |
| **BT** | Behavior Tree。ビヘイビアツリー |
| **MCP** | Model Context Protocol |
