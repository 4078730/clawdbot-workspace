## 1. 全体像：α版とβ版の位置づけ

### 1.1 一言サマリー

| 版 | リリース | 一言で言うと |
|----|---------|-------------|
| **α版** | Q2末（9月） | 単腕UR5で、ビン内の複雑なパーツを**ピック→作業台に置く**。VLA制御ポリシーがRoboSync上で動く最小構成 |
| **β版** | Q4末（3月） | 双腕で、出庫作業サイクル全体（スキャン〜個装〜台車配置）を**Agenticに回す** |

### 1.2 スコープ比較

| 項目 | α版 | β版 |
|------|------|------|
| **対象工程** | ピック→作業台配置のみ | 出庫サイクル全体（8工程） |
| **アーム** | 単腕（UR5） | 双腕（機種TBD） |
| **EE** | 吸着/平行グリッパ（手動切替） | 吸着/平行グリッパ（自動切替を目指す） |
| **制御方式** | VLA制御ポリシー＋ルールベース補完 | VLA主力＋Agentic制御（LLM Agent介入） |
| **移動** | なし（固定ステーション） | 作業空間内の姿勢変化（台車配置） |
| **個装** | なし（人が担当） | あり（袋入れ・納品書同封・袋閉じ） |
| **リカバリー** | 人が手動 | 自律リトライ＋テレオペ介入 |
| **人の役割** | 個装・納品書・リカバリーすべて人 | モニタリング＋例外対応 |
| **対象パーツ数** | 絞り込み（5〜10種程度から開始） | α版から段階的に拡張 |

## 3. 制御方法・ソフトウェアアーキテクチャ

### 3.1 VLAモデル

#### α版で使用予定のモデル

| 項目 | 内容 |
|------|------|
| **ベースモデル候補** | π0（Physical Intelligence）/ OpenVLA / Octo |
| **カスタムモデル** | IndexVLA（自社開発。LLMとVLAの通訳層） |
| **入力** | RGB画像（D405）、Depth画像（D405）、言語指示テキスト、ロボット関節状態（6DOF） |
| **出力** | エンドエフェクタの目標位置・姿勢（6DOF: x, y, z, roll, pitch, yaw）+ グリッパ開閉 |
| **制御モード** | **位置制御**（URのservo制御経由） |
| **推論周波数** | 10〜30Hz（モデルサイズにより変動） |
| **チャンクサイズ** | TBD（ACT系なら100ステップ予測、π0系なら可変） |

#### VLAモデルの入出力フロー

```
  入力                          VLAモデル                    出力
┌────────────┐                                          ┌────────────┐
│ RGB画像     │─┐                                       │ Δx, Δy, Δz │
│ (640×480)   │ │  ┌──────────────────────────┐         │ Δroll      │
├────────────┤ ├─→│                          │──────→ │ Δpitch     │
│ Depth画像   │ │  │  VLA Policy Network      │         │ Δyaw       │
│ (640×480)   │─┘  │  (π0 or IndexVLA)        │         │ gripper    │
├────────────┤     │                          │         │ (open/close│
│ 言語指示    │────→│  "ビンから天板を取って    │         │  /suction) │
│ (テキスト)  │     │   作業台に置いて"        │         └────────────┘
├────────────┤     │                          │              │
│ 関節状態    │────→│                          │              ▼
│ (6DOF)     │     └──────────────────────────┘         RoboSync経由で
└────────────┘                                          URにコマンド送信
```

#### β版で追加される制御

| 項目 | 内容 |
|------|------|
| **マルチスキルVLA** | 工程ごとに異なるVLAポリシーを学習。スキル切替はTask Planning Hub（LLM Agent）が判断 |
| **力制御** | 段ボール開封時に必要。UR5のForce/Torqueセンサ活用（β版で導入検討） |

### 3.3 ルールベースとVLAの切替条件

#### α版の切替ロジック

```
┌──────────────┐
│ タスク開始    │
└──────┬───────┘
       ▼
┌──────────────┐     ルールベース
│ ホーム→ビン  │ ──────────────→ 固定軌道で移動
│ 上方移動     │
└──────┬───────┘
       ▼
┌──────────────┐
│ ビン内撮影    │ ──────────────→ 固定位置で撮影
└──────┬───────┘
       ▼
┌──────────────────────────────────────┐
│ VLA制御ポリシーに切替                  │  VLA制御
│                                       │
│  VLAが以下を自律的に判断・実行:        │
│  ・パーツ認識（どれを取るか）          │
│  ・把持戦略（どこを、どう掴むか）      │
│  ・取り出し軌道（どうやって出すか）    │
│  ・配置動作（作業台のどこに置くか）    │
│                                       │
│  EE選択は事前に人が設定               │
│  （吸着 or グリッパ）                  │
└──────────────┬───────────────────────┘
               ▼
┌──────────────┐     VLM判定
│ 成功判定      │ ──────────────→ 配置完了を画像で確認
└──────┬───────┘
       ▼
┌──────────────┐     ルールベース
│ 退避→ホーム  │ ──────────────→ 固定軌道で退避
└──────────────┘
```

**切替の原則:**
- **空間移動（ビンから離れた移動）** → ルールベース（高速・確実）
- **対象物に近づいてからの操作** → VLA（状況適応が必要）
- **判定** → VLM（画像ベース成功/失敗判定）

### 3.4 リカバリー方法

#### α版のリカバリーフロー

```
  VLAピック実行
       │
       ├── 成功（VLM判定: OK）──→ 次のステップへ
       │
       └── 失敗
            │
            ├── パターン1: 把持失敗（滑り・脱落）
            │     → 自動リトライ（最大2回）
            │     → 3回失敗: 人に通知、手動リカバリー
            │
            ├── パターン2: 衝突検知（力超過）
            │     → 即時停止 → 人に通知
            │     → 人がビン内を整理 → 再開
            │
            ├── パターン3: 対象物認識失敗
            │     → 再撮影（アングル変更）
            │     → 認識不可: 人に通知、スキップ
            │
            └── パターン4: 配置失敗（ズレ）
                  → VLMで検知 → 自動リトライ
                  → 人が微調整
```

#### β版のリカバリー拡張

| リカバリー種別 | α版 | β版 |
|--------------|------|------|
| 把持失敗リトライ | 自動（最大2回） | 自動（最大3回）+ 把持戦略変更 |
| 衝突時 | 即時停止→人 | 即時停止→退避→再アプローチ |
| 認識失敗 | 人通知 | 別角度撮影→LLM判断→代替戦略 |
| テレオペ介入 | なし | **あり**（UMI等で遠隔操作） |
| 操作ログ記録 | ログのみ | リカバリーデータとして学習に活用 |

## 4. α版：タスクプロセス詳細

### 4.1 全体フロー

```
[Step 0] 前提条件確認
    ↓
[Step 1] ビン到着検知・撮影
    ↓
[Step 2] パーツ認識・把持戦略決定
    ↓
[Step 3] アプローチ・把持
    ↓
[Step 4] 持ち上げ・移動
    ↓
[Step 5] 作業台配置
    ↓
[Step 6] 成功判定
    ↓
[Step 7] 退避・次パーツへ
```

### 4.2 各ステップの詳細

#### Step 0: 前提条件確認

| 項目 | 内容 |
|------|------|
| **入力** | なし |
| **処理** | EEが正しく装着されているか確認（吸着 or グリッパ）。ロボットがホーム位置にいるか確認 |
| **出力** | Ready信号 |
| **実行主体** | ルールベース |
| **所要時間** | 2〜3秒 |

#### Step 1: ビン到着検知・撮影

| 項目 | 内容 |
|------|------|
| **入力** | オートストアからのビン到着信号（※連携方法はTBD）or 固定俯瞰カメラでのビン存在検知 |
| **処理** | ①ホーム→ビン上方の撮影位置へ移動（ルールベース） ②固定位置で俯瞰撮影（RGB+Depth） ③ハンドアイカメラで近接撮影（RGB+Depth） |
| **出力** | ビン内状態画像セット（俯瞰RGB, 俯瞰Depth, 近接RGB, 近接Depth） |
| **実行主体** | ルールベース移動 + カメラ撮影 |
| **所要時間** | 3〜5秒 |
| **備考** | 撮影位置はビン中央上方300mm程度（D405の最適距離範囲内） |

#### Step 2: パーツ認識・把持戦略決定

| 項目 | 内容 |
|------|------|
| **入力** | Step 1の画像セット + 言語指示（例: 「ビンから天板を取り出して作業台に置いて」） |
| **処理** | VLAモデルが画像+言語から把持戦略を決定。IndexVLAの場合、把持点・軌道のIndex（中間表現）を生成 |
| **出力** | VLAの内部状態更新（明示的な把持点座標ではなく、連続的なアクション生成の開始） |
| **実行主体** | VLAモデル |
| **所要時間** | 推論1ステップ: 50〜100ms |
| **備考** | VLAはstep-by-stepでアクションを生成するため、「認識」と「実行」は明確に分離されない。このStepは概念的な区切り |

#### Step 3: アプローチ・把持

| 項目 | 内容 |
|------|------|
| **入力** | リアルタイムのカメラ画像（ハンドアイ）+ ロボット関節状態 |
| **処理** | VLAが10〜30Hzでアクションを生成し続ける: ①ビン内のパーツに向かってアプローチ ②パーツ表面/エッジに接触 ③吸着ON or グリッパ閉じ |
| **出力** | EEの目標位置姿勢（Δx,Δy,Δz,Δroll,Δpitch,Δyaw）+ グリッパ/吸着コマンド |
| **実行主体** | VLA制御ポリシー |
| **所要時間** | 5〜15秒（パーツの状態に依存） |
| **安全制約** | 力リミット: 20N（ビン底面への押し付け防止）。速度制限: 100mm/s（ビン内操作時） |

#### Step 4: 持ち上げ・移動

| 項目 | 内容 |
|------|------|
| **入力** | 把持状態のセンサーフィードバック（吸着圧力 or グリッパ力覚） + カメラ画像 |
| **処理** | ①パーツを垂直に持ち上げ（ビンから抜き出す） ②ビン上方の安全高さまで上昇 ③作業台方向へ水平移動 |
| **出力** | アーム軌道コマンド |
| **実行主体** | VLA（持ち上げ）→ ルールベース or VLA（移動）※ 切替ポイントはTBD |
| **所要時間** | 3〜5秒 |
| **備考** | パーツが包装材に引っかかるケースが想定される。VLAが「引き出す」動作を学習する必要あり |

#### Step 5: 作業台配置

| 項目 | 内容 |
|------|------|
| **入力** | 作業台の状態画像（俯瞰カメラ）+ ロボット関節状態 |
| **処理** | ①作業台上の空き位置を認識 ②パーツを安全に降ろす ③吸着OFF or グリッパ開 |
| **出力** | 配置完了状態 |
| **実行主体** | VLA制御ポリシー |
| **所要時間** | 3〜5秒 |
| **備考** | 配置位置は「作業台上のおおよその位置」でOK（α版では厳密な配置精度は不要） |

#### Step 6: 成功判定

| 項目 | 内容 |
|------|------|
| **入力** | 俯瞰カメラのRGB画像（配置前後） |
| **処理** | VLMが「パーツが作業台上に正しく置かれているか」を判定 |
| **出力** | 成功 / 失敗 フラグ |
| **実行主体** | VLM（Vision-Language Model） |
| **所要時間** | 1〜2秒 |
| **失敗時** | リカバリーフロー（§3.4参照）へ遷移 |

#### Step 7: 退避・次パーツへ

| 項目 | 内容 |
|------|------|
| **入力** | 成功判定結果 |
| **処理** | ①退避位置へ移動 ②ログ記録（画像・関節軌道・成功/失敗・EE種別・パーツ種別） ③次のオーダーがあればStep 1へ |
| **出力** | ログデータ + 次タスクへの遷移 |
| **実行主体** | ルールベース |
| **所要時間** | 2〜3秒 |

### 4.3 α版1サイクルのタイムライン

```
  0s        5s        10s       15s       20s       25s       30s
  │─Step0──│─Step1───│──Step2+3─────────│─Step4─│─Step5──│─6,7─│
  │ 確認   │ 移動+   │  VLA制御ポリシー  │ 持上げ│ 配置   │判定 │
  │        │ 撮影    │  アプローチ+把持  │ +移動 │        │退避 │
```

**想定サイクルタイム: 25〜40秒/パーツ**（現場作業者は1オーダー1〜3分）

### 5.2 各工程の詳細

#### [1] バーコードスキャン

| 項目 | 内容 |
|------|------|
| **入力** | 作業台上の納品書の位置（カメラ認識） |
| **処理** | ①バーコードリーダーを把持（or 固定バーコードリーダーの前に納品書を持っていく） ②納品書のバーコードを読み取り ③オーダー情報を取得（品番・数量・個装方法） |
| **出力** | オーダー情報（品番、数量、個装指示） |
| **制御** | VLA or ルールベース（方式TBD） |
| **双腕の使い方** | 片方でリーダー把持、片方で納品書を押さえる（or 固定リーダーなら不要） |
| **技術課題** | バーコードリーダーの把持・位置合わせ精度。固定リーダー案なら課題が大幅に軽減 |

#### [2] モニター操作

| 項目 | 内容 |
|------|------|
| **入力** | オーダー情報（品番） |
| **処理** | モニター画面のタッチ操作でビンを呼び出す |
| **出力** | ビン呼び出しコマンド（オートストアAPI連携） |
| **制御** | ルールベース（API連携で物理操作を回避できる可能性） |
| **技術課題** | オートストアのAPI連携可否。APIなしならタッチ操作の自動化が必要 |

#### [3] 段ボール開封

| 項目 | 内容 |
|------|------|
| **入力** | ビン内の状態画像 |
| **処理** | ①ビン内に段ボールがあるかVLMで判定 ②段ボールのフラップを認識 ③片方の腕で段ボール本体を押さえ、もう片方でフラップを開く |
| **出力** | 段ボール開封完了状態 |
| **制御** | VLA（双腕協調） |
| **双腕の使い方** | 左: 段ボール固定（押さえ） / 右: フラップ開封 |
| **技術課題** | 力制御（段ボールを壊さずに開ける）。テープ付きの場合のカッター使用 |

#### [4] パーツピック（α版拡張）

| 項目 | 内容 |
|------|------|
| **入力** | ビン内状態画像 + オーダー情報（品番・数量） |
| **処理** | α版のピック動作を拡張: ①緩衝材（プチプチ等）を除ける or めくる ②指定品番のパーツを認識 ③指定数量をピック（1個ずつカウント） ④作業台上の仮置きエリアに配置 |
| **出力** | 指定数量のパーツが作業台上に配置された状態 |
| **制御** | VLA + VLM（数量カウント） |
| **技術課題** | 品番認識精度、数量カウントの正確性、緩衝材操作 |

#### [5] 残りをビンに戻す

| 項目 | 内容 |
|------|------|
| **入力** | ビン内の現状画像 + 取り出した緩衝材等 |
| **処理** | ①散らばった緩衝材をビンに戻す ②段ボールのフラップを閉じる（必要時） |
| **出力** | ビンが元の状態に近い形で整頓された状態 |
| **制御** | VLA |
| **技術課題** | 「元に戻す」動作の学習データ収集が困難 |

#### [6] 個装

| 項目 | 内容 |
|------|------|
| **入力** | 作業台上のパーツ + 個装指示（袋サイズ等） |
| **処理** | ①適切なサイズの袋を取り出す ②袋を開く（双腕で口を広げる） ③パーツを袋に入れる ④納品書を袋に同封 ⑤袋の口を閉じる（折り返し or テープ） |
| **出力** | 個装完了したパーツ |
| **制御** | VLA（双腕協調必須） |
| **双腕の使い方** | 左: 袋を押さえる/開く / 右: パーツ投入・納品書挿入・袋閉じ |
| **技術課題** | **β版最大の技術チャレンジ**。柔軟物操作（袋）、薄い紙（納品書）の操作 |

#### [7] 台車/ワゴンに配置

| 項目 | 内容 |
|------|------|
| **入力** | 個装済みパーツ + 台車の状態画像 |
| **処理** | ①台車の空き位置を認識 ②個装済みパーツを台車に配置 |
| **出力** | 台車上に配置完了 |
| **制御** | VLA |
| **技術課題** | リーチの確保（台車とロボットの距離）、配置の整列 |

#### [8] サイクル制御

| 項目 | 内容 |

| **処理** | ①現在のオーダーの完了をログ記録 ②次のオーダーがあれば[1]に戻る ③異常があればエスカレーション |
| **出力** | 次のサイクル開始 or 終了 |
| **制御** | Agentic（LLM Agent + BT） |
| **技術課題** | サイクル間の状態管理、異常からのリカバリー判断 |

## 6. 対象物体の定義

### 6.1 対象パーツのカテゴリ

動画分析（2026-02-16）に基づく彩都パーツセンターでの実際のパーツ分類:

| カテゴリ | 形状特徴 | α版対象 | β版対象 | 難易度 |
|---------|---------|---------|---------|--------|
| **天板・パネル系** | 薄い平面パーツ。端のみ把持可能 | ✅ | ✅ | 中 |
| **箱物** | 直方体。段ボール梱包されていることが多い | ✅ | ✅ | 低 |
| **不定形パーツ** | 非対称・曲面を持つ部品 | ✅ | ✅ | 高 |
| **小型部品** | ネジ・コネクタ等の小さなパーツ | △(一部) | ✅ | 高 |
| **袋入りパーツ** | ビニール袋に入った状態 | ✅ | ✅ | 中 |
| **ケーブル・配線** | 柔軟物。束になっている場合あり | ❌ | △ | 最高 |
| **液体・粉体含む** | 破損リスクのある容器類 | ❌ | ❌ | — |

### 6.2 包装状態のバリエーション

| 包装状態 | 説明 | α版対象 | 把持方法 |
|---------|------|---------|---------|
| **裸（無包装）** | ビンに直接入っている | ✅ | 形状に応じて吸着 or グリッパ |
| **プチプチ包装** | 気泡緩衝材で巻かれている | ✅ | 包装ごと掴む → 後で人が剥がす(α版) |
| **ビニール袋入り** | 透明/半透明の袋に入っている | ✅ | 袋ごと掴む |
| **段ボール内** | ビン内の小段ボールに入っている | △(開封不要なもの) | 段ボールごと取り出し(α版) |
| **仕切り付きビン** | ビン内に仕切り板がある | ✅ | 仕切りを避けてピック |
| **発泡スチロール固定** | 緩衝材に嵌め込まれている | △ | 引き抜き動作が必要 |

### 6.3 α版対象パーツの選定基準

α版で最初に扱うパーツは、以下の基準で**5〜10種類を選定**する:

| # | 基準 | 詳細 |
|---|------|------|
| 1 | **吸着/グリッパで把持可能** | 平面 or 安定した把持点がある |
| 2 | **重量5kg以下** | UR5のペイロード制約 |
| 3 | **ビンに単品 or 少数** | 山積みでないもの（最初は1〜3個/ビン） |
| 4 | **破損リスクが低い** | 落としても壊れにくいもの |
| 5 | **形状の多様性をカバー** | 選定5〜10種で平面・箱物・不定形・袋入りをカバー |
| 6 | **現場で頻出** | 出荷頻度が高く、実証の回数が稼げるもの |

> **⚠️ 未決定:** 具体的なパーツの品番リストはRoboSyncチーム（奥村さん）+ 彩都現場（中田さん・一力さん）と選定が必要。

### 6.4 除外対象（スコープ外）

| 除外対象 | 理由 |
|---------|------|
| **5kg超のパーツ** | UR5ペイロード超過 |
| **液体・粉体含む容器** | 破損時の安全リスク |
| **柔軟ケーブル・配線束** | α版では把持困難。β版でも限定的 |
| **ビン内に密集（10個以上）** | 認識・選別の難易度が現時点で高すぎる |
| **超小型部品（5mm未満）** | D405の深度精度の限界 |
| **ガラス製品** | 破損リスク |

## 8. 検証計画

### 8.1 α版の検証計画

#### 成功/失敗の定義

| 判定 | 定義 |
|------|------|
| **ピック成功** | 指定パーツがビンから持ち上がり、把持を維持した状態で作業台上方に到達 |
| **配置成功** | パーツが作業台上の指定エリア（50cm四方）内に安定して置かれた |
| **タスク成功** | ピック成功 AND 配置成功。パーツに損傷なし |
| **タスク失敗** | ピック失敗（把持不可/脱落）、配置失敗（範囲外/落下）、衝突停止、タイムアウト（60秒超） |

#### 評価指標

| 指標 | 目標値 | 測定方法 |
|------|--------|---------|
| **ピック成功率** | 90%（初回） | 成功回数 / 試行回数 × 100。対象パーツ絞り込み条件下 |
| **リトライ込み成功率** | 95% | リトライ（最大2回）を含めた最終成功率 |
| **サイクルタイム** | 40秒以内 | Step 0〜7の合計時間 |
| **安全性** | インシデント0 | 人への接触事故、パーツ破損事故がゼロ |
| **ログ蓄積率** | 100% | すべての試行でログが正常に記録される |
| **VLAモデル推論レイテンシ** | 100ms以下 | 1推論ステップの所要時間 |

#### 評価環境

| フェーズ | 環境 | 内容 | 時期 |
|---------|------|------|------|
| **ラボ検証** | CXC or 社内ラボ | 模擬ビン+模擬パーツで基本動作検証 | Q1〜Q2 |
| **現場検証** | 彩都パーツセンター | 実際のビン+実際のパーツで検証 | Q2末（9月） |

#### 必要データ量（学習用）

| データ種別 | 想定量 | 収集方法 |
|-----------|--------|---------|
| **テレオペデモ** | 100〜500エピソード/パーツ種別 | 奥村さんのテレオペ環境で収集 |
| **合計** | 500〜5,000エピソード（5〜10種×100〜500） | |
| **収集所要時間** | 1エピソード3分 × 5,000 = 250時間 | 2人で1ヶ月程度 |
| **追加データ** | 現場稼働中のログデータ（継続学習用） | RoboSync経由で自動記録 |

> **備考:** VLAモデルの種類によってデータ要件は大きく変わる。π0系は比較的少量データで動作する報告あり（50〜100エピソード）。事前にラボでパイロット学習を行い、必要データ量を見積もる。

### 8.2 β版の検証計画

#### 成功/失敗の定義

| 判定 | 定義 |
|------|------|
| **サイクル成功** | [1]〜[8]の全工程がロボット単独で完了（テレオペ介入なし） |
| **サイクル成功（介入あり）** | [1]〜[8]が完了するが、一部工程でテレオペ介入が発生 |
| **サイクル失敗** | いずれかの工程でリカバリー不能な失敗が発生し、人の直接介入が必要 |

#### 評価指標

| 指標 | 目標値 | 備考 |
|------|--------|------|
| **サイクル成功率** | TBD（50%以上を最低ライン） | 初めて全工程を通す段階なので高い目標は置かない |
| **工程別成功率** | 各工程80%以上 | ボトルネック工程の特定が目的 |
| **人介在頻度** | 計測・記録 | 初期値の確立が目的（削減目標はGA以降） |

| **連続運用** | 2時間以上 | システムダウンなしの連続稼働 |
| **安全性** | インシデント0 | 異常時自動停止が機能すること |

#### 評価環境

| フェーズ | 環境 | 内容 | 時期 |
|---------|------|------|------|
| **ラボ検証（個別工程）** | 社内ラボ | 個装・開封等、各工程を個別に検証 | Q2〜Q3 |
| **ラボ検証（統合）** | 社内ラボ | 全工程を通したサイクル検証 | Q3 |
